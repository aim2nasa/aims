# FacetLab 가치 검증 논의

## 개요

이 문서는 FacetLab (다면적 동적 트리 문서 네비게이션 시스템) 컨셉에 대한 가치 검증 논의 과정을 기록합니다.

날짜: 2025-10-24

---

## 1. 아이디어의 본질

### 1.1 초기 오해와 명확화

**초기 TAG_DESIGN.md의 방향:**
- 문서 검색과 분류를 위한 태그 시스템
- 15-20개 태그 추천
- 검색 정확도 향상에 초점

**사용자의 실제 의도 (명확화):**

> "내가 원하는 목적과 조금 다른데, 난 아래와 같은 일을 하려고 태그를 도입하려는 것이다.
>
> 사용자의 여러 파일들을 다양한 기준에 따라, 트리 형태로 논리적으로 구조화해서 제공하고자하는것이야.
> 첨부파일에 보면, '다양한 기준별 트리 예시' 이게 태그들을 사용하여 여러가지 형태로 사용자의 문서들을 입체적으로 구조화 해서 보여주는 예. 이게 내가 빌드업하고 있는 아이디어야."

### 1.2 핵심 컨셉

**FacetLab의 본질:**

1. **다면적 동적 트리 네비게이션**
   - 동일한 문서들을 여러 관점에서 볼 수 있는 시스템
   - 정적 폴더 구조가 아닌 동적 생성 트리

2. **다양한 관점 제공**
   - 고객별 트리
   - 보험사별 트리
   - 날짜별 트리
   - 지역별 트리
   - 문서 유형별 트리
   - 등등...

3. **Windows 탐색기의 진화형**
   - Windows 탐색기처럼 계층 구조로 문서 탐색
   - 그러나 정적이 아닌 **동적**
   - 사용자가 폴더를 만들지 않아도 **자동으로 구조 생성**
   - 하나의 고정된 구조가 아닌 **여러 관점 제공**

4. **유기체적 자기 조직화 시스템**
   - AI가 문서에서 태그를 자동 추출
   - DB 쿼리로 다양한 트리 구조 동적 생성
   - 새 문서 추가 시 자동으로 트리에 반영
   - 사용자의 수동 작업 제로

### 1.3 기존 시스템과의 차이

| 시스템 | 구조 | 생성 방식 | 관점 |
|--------|------|-----------|------|
| **Windows 탐색기** | 정적 폴더 | 사용자 수동 생성 | 하나의 고정된 구조 |
| **검색 시스템** | 구조 없음 | N/A | 키워드 기반 |
| **FacetLab** | 동적 트리 | AI 자동 생성 | 다면적 관점 전환 |

## 2. 핵심 가치 질문

사용자의 질문:
> "기술적 과제는 불가능한게 아니면 해결하면 되는것이고, 중요한것은 '가치'가 있느냐야.
> 이 관점이 제일 궁금한 점이다."

### 2.1 현재 상황 분석

**보험 영업사원의 실제 업무 시나리오:**

#### 시나리오 1: 고객 상담 중 급하게 서류 찾기
- **상황**: "김철수 고객님, 작년에 청구하신 입원비 서류 어디 있더라..."
- **현재 방법**: 검색창에 "김철수 입원비" 입력 → 결과 20개 → 하나씩 클릭해서 확인
- **FacetLab**: "고객별 > 김철수 > 청구서류 > 입원비" 3클릭으로 바로 도달

#### 시나리오 2: 월말 보고서 작성
- **상황**: "이번 달 삼성화재 관련 모든 서류 정리해야 하는데..."
- **현재 방법**: 검색 여러번 → 엑셀에 수동 정리
- **FacetLab**: "보험사별 > 삼성화재 > 이번달" 자동으로 모아서 보여줌

#### 시나리오 3: 신규 고객 미팅 준비
- **상황**: "이 지역 고객들 어떤 보험 많이 가입하더라?"
- **현재 방법**: 기억에 의존 or 불가능
- **FacetLab**: "지역별 > 강남구 > 보험종류별" 트렌드 한눈에 파악

### 2.2 진짜 가치가 있는 이유

#### 1. 사고방식과 일치
- 사람은 상황에 따라 다르게 생각함
- "오늘은 고객 중심으로 볼까, 오늘은 보험사별로 볼까"
- **정적 폴더는 하나의 관점만 강요**, FacetLab은 사고방식에 맞춰 변화

#### 2. Zero Manual Work의 가치
- 폴더 만들기 = 귀찮음 = 안 함 = 결국 검색에만 의존
- 자동 구조화 = 손 안 대도 됨 = 항상 정리된 상태 유지

#### 3. 시간이 지날수록 가치 증가
- 문서 10개: 검색으로 충분
- 문서 100개: 좀 불편
- **문서 1000개: FacetLab이 없으면 지옥**

### 2.3 잠재적 문제점

#### 1. 학습 곡선
- "그냥 검색이 편한데 왜 트리를 봐야 해?"
- 새로운 방식 = 익숙해지기까지 시간 필요

#### 2. 복잡도 vs 실용성
- 5가지 관점 제공 → 오히려 혼란?
- "어떤 관점으로 봐야 내가 원하는 거 찾을 수 있지?"

#### 3. 실제 니즈 검증 필요
- 영업사원: "나 그냥 검색으로 충분해"
- 매니저: "팀 전체 문서 관리에는 좋을 것 같은데..."
- **실제 사용자가 느끼는 pain point가 이걸로 해결되는가?**

## 3. 가치 검증 핵심 질문 3가지

### 3.1 빈도 (Frequency)
**"얼마나 자주 이런 상황이 발생하나?"**
- 하루에 10번 문서 찾기 → 가치 높음
- 한 달에 1번 → 가치 낮음

### 3.2 고통도 (Pain Level)
**"못 찾으면 얼마나 치명적인가?"**
- 고객 상담 중 못 찾음 → 신뢰 하락 → 치명적
- 단순 정리용 → 불편하지만 치명적은 아님

### 3.3 대체재 (Alternatives)
**"기존 방법(검색)과 비교해서 얼마나 더 나은가?"**
- 10배 빠름 → 명확한 가치
- 2배 빠름 but 복잡함 → 애매함

## 4. ROI 분석 (참고용)

### 4.1 현재 비용
- 영업사원 1명당 하루 평균 10번 문서 검색
- 건당 평균 2분 소요
- 하루 20분 = 연간 약 80시간 손실
- 영업사원 100명 기준: 연간 8,000시간
- 시간당 인건비 10만원 가정: **연간 약 8억원 손실**

### 4.2 FacetLab 도입 시
- 검색 시간 2분 → 30초로 단축 (75% 개선)
- 연간 약 6억원 절감 가능

### 4.3 단, 조건부
- 학습 기간 필요 (초기 생산성 하락)
- 태그 품질에 따라 효과 변동
- 사용자 수용도에 따라 실제 사용률 결정

## 5. 가치 판단 결론

### 5.1 가치가 있는 경우
✅ 문서가 많은 파워유저 (100개 이상)
✅ 다양한 관점에서 문서를 자주 찾는 사람
✅ 복잡한 고객 관계 관리가 필요한 경우

### 5.2 가치가 낮은 경우
❌ 문서가 적은 초보 영업사원 (10-20개)
❌ 항상 같은 방식으로만 찾는 사람
❌ 단순 검색으로 충분한 경우

### 5.3 최종 판단 기준

**핵심 질문:**
> "문서가 폭발적으로 증가하는 순간, 검색만으로는 한계가 오는 그 시점에 FacetLab이 구세주가 될 수 있는가?"

**만약 이 답이 "Yes"라면 → 가치 있음**
**만약 "검색 개선하면 되는데?"라면 → 가치 의심**

## 7. Claude의 객관적 가치 판단

사용자 요청:
> "적어도 너는 내가 아니니 객관적으로 이 아이디어에 대한 너의 가치 판단을 듣고 싶어"

### 7.1 가치 있다고 생각하는 이유

#### 1. 근본적인 문제를 건드림
- 정보 과부하는 모든 지식 노동자의 공통 pain point
- 검색과 폴더 둘 다 한계가 명확함
- 이 문제는 앞으로 더 심해질 것 (데이터는 계속 증가)

#### 2. 접근법이 독창적임
- 검색 개선 ❌ (구글이 이미 최고)
- 폴더 개선 ❌ (결국 사용자가 만들어야 함)
- **동적 다면 분류** ✅ (아무도 안 하고 있음)

#### 3. 기술적으로 현실적임
- AI 1회 사용 → 비용 효율적
- DB 쿼리 기반 → 빠름
- 구현 난이도 중간 정도

### 7.2 의문점

#### 핵심 의문: "사람들이 트리 구조를 정말 선호하나?"

**관찰된 현상:**
- Windows 탐색기는 습관이지, 좋아서 쓰는 게 아닐 수도
- 요즘 젊은 세대는 폴더 개념 자체를 이해 못함 (검색만 씀)
- Notion, Obsidian 등도 결국 검색이 메인

**우려사항:**
- 트리 5개 제공 → 사용자: "어떤 걸 봐야 하지?" (혼란)
- 결국 익숙한 하나만 계속 사용 → 나머지 4개는 사장
- 그럼 정적 폴더 하나와 차이가 뭐지?

### 7.3 결정적 질문

**"이게 검색보다 나은 순간이 언제인가?"**

| | 검색 | FacetLab |
|---|------|----------|
| **장점** | 빠름, 익숙함 | 브라우징 가능, 맥락 파악 |
| **단점** | 키워드 모르면 못 찾음<br>결과 많으면 피곤 | 학습 필요<br>복잡도 증가 |

**→ 이 차이가 실제로 업무 시간을 유의미하게 줄여주나?**

### 7.4 최종 의견: 70% 가치 있음 / 30% 의문

**가치 있는 이유 (70%):**
- 문제가 실재함
- 기존 솔루션의 한계가 명확함
- 차별화된 접근법

**의문인 이유 (30%):**
- **사용자 행동 변화가 필요함** (이게 제일 어려움)
- 트리 vs 검색, 실제로 트리가 이길 수 있나?
- POC 해봐야 알 것 같음

### 7.5 결론

**POC로 직접 써보는 게 100% 맞는 접근입니다.**

이건 논리로 판단할 수 있는 게 아니라,
**"아, 이거 진짜 편하네!"** 이 느낌이 오느냐 마느냐의 문제.

- 그 느낌이 온다면 → 가치 있음
- 안 온다면 → 아이디어는 좋지만 실용성 없음

## 8. 핵심 인사이트 요약

### 8.1 FacetLab의 본질
- 검색 개선이 아님
- 폴더 개선이 아님
- **정보 탐색 패러다임의 전환**

### 8.2 성공 조건
1. **문서량**: 최소 100개 이상 필요
2. **다면성**: 사용자가 실제로 여러 관점을 원해야 함
3. **태그 품질**: AI가 일관되게 정확한 태그를 뽑아야 함
4. **UX 설계**: 복잡도를 사용자에게 느끼게 하면 안 됨

### 8.3 실패 위험
1. **학습 곡선**: 너무 복잡하면 사용자가 포기
2. **관성**: 기존 방법(검색)이 익숙하면 안 바꿈
3. **실용성**: 트리가 검색보다 실제로 느리면 의미 없음

### 8.4 검증 방법
- 논리적 분석 < **실제 사용 경험**
- 다른 사람 의견 < **본인의 체감**
- POC를 통한 직접 검증이 유일한 답

## 부록: 주요 대화 발췌

### A. 사용자의 핵심 의도 설명

> "고칠 필요는 없어. 지금은 아이디어 빌드업 과정이다. 핵심은 고객에 대해서 갖고 있는 많은 문서덩어리들을 사실 한눈에 바로 볼수는 없기때문에, 이들을 어떻게 다양한 각도로 쉽고 편리하게 원하는 정보(문서)에 바로 다가갈수 있게 하느냐지.
>
> 문서가 많아질 수록, 내가 원하는 문서가 어디에 있는지, 필요한 정보에 접근하기 힘들어져. 물론 검색방법도 만들어 놓기는 했지만, 윈도우 탐색기 처럼 논리적으로 구조화 해서 파일을 찾는 방법도 제공하고 싶은거야.
>
> 그러나 윈도우 탐색기는 매우 정적으로 폴더구조가 유지되고, 사용자가 폴더 구조를 지정해야 하는 단점이 있어. 본 방법에서는, 정적이 아닌 동적으로, 다양한 기준에 따라 문서hierarchy를 고객에게 보여줘 쉽게 이해할수 있게 하고, 폴더를 고객이 만들지 않아도 알아서 구조를 만들어주는 유기체적 구조를 생각하는거다.
>
> 본질은 사용자의UX를 최대로 끌어올려, 업무 생산성을 높이고자 하는거다."

### B. 검증 단계 명확화

### C. 관심사 명확화
