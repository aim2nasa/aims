## 핵심 원칙

### 1. 설계 우선 (Design First)

코드를 한 줄이라도 쓰기 전에 **설계를 확정**한다.

| 단계 | 수행 내용 |
|------|-----------|
| 1. 요구사항 분석 | 무엇을 해결해야 하는가? 경계 조건은? |
| 2. 기존 코드 이해 | 현재 아키텍처, 패턴, 관례를 완벽히 파악 |
| 3. 설계 도출 | 최소 변경으로 최대 효과를 내는 설계 |
| 4. 영향 범위 분석 | 이 변경이 다른 모듈에 미치는 영향 |
| 5. 구현 | 설계대로 정확히 구현 |

### 2. 근본 해결 (Root Cause Only)

- 증상이 아닌 **원인**을 해결
- 임시 방편, band-aid fix 절대 금지
- 같은 문제가 재발하지 않는 구조적 해결

### 3. 최소 변경 원칙 (Minimal Change)

- 요청된 기능에 직접 필요한 부분만 수정
- 관련 없는 코드 리팩토링, 주석 추가 금지
- "이왕 하는 김에" 식의 불필요한 변경 금지

### 4. 재사용 가능한 설계 (Reusable Design)

- 컴포넌트는 재사용 가능하게 설계
- API는 RESTful 원칙 준수
- 에러 처리를 항상 포함
- 한글 주석 사용

### 5. 실행 환경 인식 (Runtime Awareness)

- 코드가 **실제로 어디서 실행되는지** 항상 인식
- 컴파일 타임 vs 런타임 vs 빌드 타임 구분
- 환경 차이 (개발/프로덕션, Python/Jython, 패키징/소스) 반드시 고려

### 6. 검증 후 완료 (Verify Before Done)

- 구현 완료 후 **반드시 검증**
- 빌드 성공, 구문 검증, 엣지 케이스 확인
- "될 것이다"가 아니라 "확인했다"

## 작업 프로세스

### Phase 1: 탐색 (Explore)

```
1. 관련 파일 모두 읽기
2. 기존 아키텍처/패턴 파악
3. 의존성 그래프 이해
4. 실행 환경 확인 (런타임, 인터프리터, OS)
```

### Phase 2: 설계 (Design)

```
1. 변경 범위 최소화
2. 기존 패턴과 일관된 설계
3. 엣지 케이스/에러 시나리오 고려
4. 영향받는 모든 파일 목록화
```

### Phase 3: 구현 (Implement)

```
1. 설계대로 정확히 구현
2. 한 파일씩 순차적으로 수정
3. 각 수정마다 기존 코드와의 정합성 확인
4. 하드코딩, 매직 넘버 금지
```

### Phase 4: 자체 코드리뷰 (Self Review)

```
1. 구문 검증 (Python: ast.parse, TypeScript: tsc --noEmit)
2. 빌드 검증 (npm run build 등)
3. 보안 점검: XSS, injection, 인증 누락 여부
4. 에러 처리 누락 점검
5. 영향받는 다른 파일과의 정합성 확인
6. 실행 환경별 동작 확인
```

작업 완료 후 다음 형식으로 보고:
