# /bug — 버그 리포트 + 수정 자동화 스킬

**버그 신고 → 이슈 등록 → 수정 → 검증** 한 번에.
**처리 과정 전반을 GitHub 이슈에 실시간 기록.**

## 트리거

- `/bug {설명}` (사용자 호출)

## 워크플로우

```
1. 사용자 설명 파싱 → 제목 + 본문 생성
2. gh issue create --label "bug" → 이슈 번호 획득
3. /compact-fix 자동 시작 (이슈 번호 연결)
   - Phase 0: 브랜치 생성 (fix/{이슈명})
   - Phase 1: 원인 분석 → 토의 (사용자 승인 대기)
   - Phase 2~5: 승인 후 자동 진행
4. 각 Phase 완료 시 GitHub 이슈에 코멘트 기록
```

## GitHub 이슈 기록 규칙 (CRITICAL)

**모든 처리 과정을 해당 GitHub 이슈에 코멘트로 남긴다.**

### 기록 시점 및 내용

| 시점 | 기록 내용 |
|------|-----------|
| Phase 0 완료 | 브랜치명, 관련 파일 목록 |
| Phase 1 완료 | 원인 분석 결과, 수정 방향 요약 |
| 사용자 승인 후 | 승인된 방향 명시 |
| Phase 2~4 완료 | 변경 파일 목록, 주요 수정 내용 요약 |
| Phase 5 완료 | 검증 결과 (테스트 통과 여부), 커밋 해시 |
| 최종 완료 | 이슈 close + 최종 요약 코멘트 |

### 코멘트 형식

```markdown
## Phase {N}: {Phase 이름}

{핵심 내용 — 3~5줄 이내}

- 변경: `파일경로` — 변경 설명
- 결과: PASS / FAIL / 대기
```

### 이슈 Close

- 모든 Phase 완료 + 테스트 PASS 시 자동 close
- `gh issue close {번호} --comment "완료 요약"`

## 이슈 생성 규칙

### 제목
- 사용자 설명에서 핵심 문제를 추출하여 간결하게
- 형식: `fix: {문제 설명}`
- 50자 이내

### 본문
```markdown
## 현상
{사용자가 설명한 증상}

## 환경
- 발견 시점: {현재 날짜}
- 보고자: 사용자

## 참고
- 스크린샷이 있으면 첨부
- 관련 파일/문서가 있으면 명시
```

### 라벨
- `bug` (기본)

## 스크린샷 자동 첨부

사용자가 `/bug` 호출 시 현재 Playwright 브라우저가 열려 있으면:
1. 현재 페이지 스크린샷 캡처
2. 이슈 본문에 참고 정보로 기록 (경로만 — GitHub에 직접 업로드는 하지 않음)

## compact-fix 연결

이슈 생성 후 자동으로 `/compact-fix` 프로세스를 시작한다.

1. Phase 0: `git checkout -b fix/{이슈-키워드}`
2. Phase 1: 원인 분석 시작 (코드 + DB 조사)
3. Phase 1 토의에서 **사용자 승인 대기** (유일한 개입 지점)
4. 승인 후 Phase 2~5 자동 진행
5. **각 Phase 완료마다 `gh issue comment`로 진행 상황 기록**

## 예시

```
사용자: /bug 0B로 표시되는 파일. 이미지인데 크기가 안 나옴
```

→ 자동 실행:
```
[1] GitHub 이슈 등록: "fix: 이미지 파일 크기 0B 표시 (#23)"
[2] 브랜치 생성: fix/image-filesize-zero
[3] 원인 분석 시작...
[4] → gh issue comment #23: "Phase 1 원인 분석 완료: ..."
[5] 토의 보고 → 사용자 승인 대기
[6] → gh issue comment #23: "Phase 2~4 수정 완료: ..."
[7] → gh issue comment #23: "Phase 5 검증 PASS"
[8] → gh issue close #23: "최종 완료 요약"
```

## 주의사항

- `/bug` 없이 일반 대화로 버그를 알려줘도 이 스킬을 자동 적용하지 않음 (명시적 호출만)
- 이슈 제목이 마음에 안 들면 사용자가 GitHub에서 직접 수정
- compact-fix 부적합 판단 시 (대규모 변경 등) 사용자에게 ACE 전환 제안
- **GitHub 이슈 기록 누락은 CRITICAL 위반** — 모든 Phase에서 반드시 코멘트 남길 것
- **이슈 기록 하네스**: 각 Phase 완료 시 `gh issue comment`가 다음 Phase 진입 조건. `pre_commit_review.py`가 커밋 시 이슈 코멘트 유무도 검증.
