# Compact Fix Process

**토의 → 구현 → 실증** — 본질에만 집중하는 경량 개발 프로세스.

2026-04-02 합의. ACE Process의 경량 버전으로, 버그 수정/소규모 개선에 최적화.

## 적용 범위

| 적합 | 부적합 (ACE 사용) |
|------|-------------------|
| 버그 수정 | 대규모 신기능 |
| 소규모 기능 개선 | 아키텍처 변경 |
| UI 동작 수정 | 다수 파일 영향 리팩토링 |
| 결과가 명확히 검증 가능한 작업 | 기준이 주관적인 작업 |

## Phase 1: 토의

**목적:** 코딩 전 방향 합의. 코딩부터 하는 것 절대 금지.

1. 각 이슈별 **원인 분석** (코드 + DB 조사)
2. 이슈별 **순차 토의** → 방향 합의
3. 사용자 승인 후에만 다음 Phase 진행
4. 토의 내용을 **보고서로 기록** (`docs/DISCUSSION_{날짜}_{주제}.md`)
   - 커밋에 포함하여 향후 참고 자료로 활용

### 토의 원칙
- 이슈 하나씩 합의 (묶어서 처리하지 않음)
- 사용자가 "OK" 할 때까지 코드 수정 금지
- 불확실한 부분은 솔직히 "확인이 필요합니다"

## Phase 2: 구현

**목적:** 합의된 방향대로 구현 + 코드 검증.

1. **Alex 위임** — 복수 이슈면 병렬 가능
2. **Gini 코드 리뷰** — 보안, 부작용, 로직 정확성
3. Gini FAIL 시 → 수정 후 재검수 (최대 2회)

### 구현 원칙
- 합의 범위만 수정 (관련 없는 코드 금지)
- 빌드 검증 (`npm run build`) 필수

## Phase 2.5: 배포 (백엔드 변경 시)

**백엔드 코드 변경이 포함된 경우, Phase 3 전에 반드시 배포한다.**

1. 변경 파일이 백엔드(.py, 서버 코드)를 포함하는지 확인
2. 포함하면 → 해당 서비스 배포 후 Phase 3 진행
3. **배포 없이 "코드 PASS"로 넘기는 것은 절대 금지**

### 배포 판단 기준
| 변경 위치 | 배포 필요 | 배포 대상 |
|-----------|-----------|-----------|
| `frontend/` only | 불필요 (dev 서버에서 즉시 반영) | - |
| `backend/api/document_pipeline/` | 필요 | `deploy_document_pipeline.sh` |
| `backend/api/aims_api/` | 필요 | `deploy_aims_api.sh` |
| `backend/embedding/` | 필요 | 크론이 자동 반영 (재시작 불필요) |
| 프론트 + 백엔드 혼합 | 필요 | 백엔드 먼저 배포 → 프론트 검증 |

## Phase 3: 실동작 검증

**목적:** 코드 리뷰로 발견 불가능한 버그를 실행으로 증명.

1. **모든 이슈를 실제 환경에서 테스트** — 예외 없음
2. **PASS/FAIL 명확히 판정** — 스크린샷, URL, 데이터로 증거 확보
   - 검증 4단계 (반드시 순서대로):
     1. **computed style 비교** — `getComputedStyle()`로 fontSize, fontWeight, color 등 추출. 두 화면의 값을 데이터로 비교
     2. **element 스크린샷** — 전체 페이지가 아닌 `element.screenshot()`으로 해당 요소만 고해상도 캡처
     3. **픽셀 diff** — 두 스크린샷을 `pixelmatch`/`sharp`로 비교, diff 이미지 생성
     4. **나란히 비교** — 동일 크기 crop으로 시각 증거 제공
   - **저해상도 핑계 금지** — viewport 1920x1080, element screenshot 사용. 읽히지 않는 스크린샷은 증거가 아님
   - **코드 분석만으로 CSS 문제를 판단하지 않음** — 반드시 브라우저에서 실제 적용된 값 확인
3. **FAIL 시 수정 루프:**
   - 원인 추적 (console.log, interceptor 등 활용)
   - 수정 적용
   - 재검증
   - 성공할 때까지 반복

### 검증 방법 원칙

**프론트엔드가 있는 기능은 반드시 프론트엔드(Playwright)로 검증한다.**
API 호출만으로 PASS를 부여하는 것은 절대 금지. 사용자가 실제로 하는 UI 흐름
(버튼 클릭 → 동작 → 결과 확인)을 그대로 재현하여 증명해야 한다.

| 변경 유형 | 검증 방법 |
|-----------|-----------|
| 프론트엔드 UI/동작 | **Playwright 브라우저 테스트 (필수)** |
| 백엔드 API (프론트 없는 순수 API) | API 직접 호출 (배포 후) |
| 백엔드 API (프론트에서 호출하는 API) | **Playwright로 프론트 통해 검증 (필수)** |
| DB 데이터 | MongoDB MCP 직접 조회 |
| 파이프라인 | 실제 파일 업로드 테스트 (배포 후) |

### 절대 규칙
- **"코드 PASS"는 PASS가 아니다** — 실행으로 증명되지 않은 것은 미검증
- **API 호출 PASS ≠ UI 동작 PASS** — 프론트가 있으면 프론트로 증명
- **코드 리뷰 PASS ≠ 동작 PASS** — 반드시 실행해서 증명
- 검증 실패 시 포기하지 않고 근본 원인까지 추적
- 증거(스크린샷, 로그, URL)를 남겨 사용자에게 보고

## 프로세스 흐름도

```
사용자 이슈 제기
    ↓
[Phase 1] 원인 조사 → 이슈별 토의 → 합의 → 보고서 작성
    ↓
[Phase 2] Alex 구현 (병렬) → Gini 코드 리뷰
    ↓
[Phase 2.5] 백엔드 변경 시 → 배포 (검증 전 필수)
    ↓
[Phase 3] 실동작 검증 — 모든 이슈 예외 없이 실행으로 증명
    ├─ PASS → Phase 4
    └─ FAIL → 원인 추적 → 수정 → 재검증 (루프)
    ↓
[Phase 4] Gini 검수 커밋
```

## 과거 위반 사례 (2026-04-02)

### 위반 1: "코드 PASS"
백엔드 originalName fallback 코드를 구현 후, 배포 없이 "코드 PASS"로 보고.
실제 사용자가 테스트하니 별칭 생성 실패 — 서버에 이전 코드가 동작 중이었음.
**교훈: 실행으로 증명되지 않은 것은 PASS가 아니다.**

### 위반 2: "API 호출 PASS"
백엔드 배포 후 API를 직접 호출하여 성공 응답을 확인하고 PASS로 보고하려 함.
그러나 사용자가 실제로 쓰는 것은 프론트엔드 UI이므로, Playwright로 UI 흐름
(별칭AI → 파일 선택 → 완료 → 결과 확인)을 재현해야 진짜 PASS.
**교훈: 프론트가 있는 기능은 프론트로 증명. API 호출은 PASS가 아니다.**
