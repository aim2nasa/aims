**AC first, Code, Evidence** — 기준 먼저, 코드, 증거.

## ACE 보고서

모든 ACE 작업은 `docs/ace-reports/{날짜}_{작업명}.md` 에 보고서를 남긴다.

**목적:** 향후 설계 수정, 버그 분석, 재작업 시 맥락을 빠르게 파악하기 위한 자료.

**원칙:**
- **git으로 파악 불가능한 정보만 기록한다** — 코드 diff, 파일 목록, 커밋 로그에서 알 수 있는 건 쓰지 않음
- 핵심 위주. 분량은 작업 규모에 비례 (제한 없음)
- 형식은 자유. 작업에 맞게 판단

**반드시 담아야 할 것:**
- 전체 흐름 — 뭘 왜 어떻게 했는지 흐름이 잡히는 수준
- 결정의 맥락 — 왜 이 방식을 택했는지, 고려한 대안과 버린 이유
- 함정/주의점 — 코드만 봐서는 알 수 없는 위험 요소, 건드리면 깨지는 것
- 당시 상황/제약 — 이 결정에 영향을 준 조건

**기록 시점:**
- ACE 0/6 시작 시 파일 생성
- 각 단계에서 핵심 결정/발견 사항 누적 기록 (Claude가 기록)
- 코드 커밋에 보고서 포함

## 작업 크기 분류 (Fast Track)

모든 작업을 S/M/L로 분류하고, 크기에 맞는 경로를 탄다.

| 크기 | 판단 기준 | 예시 |
|------|----------|------|
| **S** (Small) | 파일 1~2개, 단순 변경, 로직 변경 없음 | CSS 수정, 텍스트 변경, 단순 버그, 설정값 변경 |
| **M** (Medium) | 파일 3~5개, 기존 기능 수정, 로직 변경 있음 | 기능 수정, 일반 버그 수정, 컴포넌트 리팩토링 |
| **L** (Large) | 파일 6개+, 신규 기능, 아키텍처 변경 | 신규 기능, 대규모 리팩토링, 아키텍처 변경 |

### 경로별 단계

```
S 경로 (4단계, 5~10분):
  [ACE 1] AC 1줄 작성 (Claude 직접) → 사용자 승인
  [ACE 2] Alex 구현
  [ACE 3] 빌드 + Gini 코드리뷰
  [ACE 4] 완료 보고
  — Merlin 생략, Jude 생략, E2E 생략, 배포는 Gini PASS 후 1회

M 경로 (5단계, 15~25분):
  [ACE 1] AC 작성 (Claude 직접, Merlin 생략) → 사용자 승인
  [ACE 2] 테스트 생성 + Alex 구현
  [ACE 3] 검증 루프 (빌드 + 단위테스트 + Gini + E2E, 배포는 로컬 PASS 후 1회)
  [ACE 4] Jude 감사
  [ACE 5] 완료 보고
  — Merlin 생략, Mira 생략

L 경로 (6단계, 30분+):
  현행 풀 ACE 0~5/6 그대로 적용
```

### 에이전트 조건부 호출

| 경로 | 호출 에이전트 |
|------|-------------|
| S | Alex, Gini |
| M | Alex, Gini, Jude |
| L | Merlin, Alex, Gini, Mira, Jude |

### 병렬 실행 규칙

의존 관계가 없는 에이전트는 병렬 호출한다:

| 병렬 가능 | 설명 |
|-----------|------|
| Gini + 단위 테스트 | 코드 리뷰와 테스트 실행은 서로 독립 |
| E2E + Mira | 배포 후 E2E 테스트와 시각 검증은 동시 가능 |

| 순차 필수 | 이유 |
|-----------|------|
| Alex → Gini | 구현 완료 후 검수 |
| Gini PASS → 배포 | 검수 통과 후 배포 |
| 배포 → E2E | 배포 완료 후 E2E 테스트 |

### 컨텍스트 패키지

단계 전환 시 다음 에이전트에게 **컨텍스트 패키지**를 전달하여 코드베이스 재탐색을 방지한다:

```
컨텍스트 패키지 = {
  1. 변경 대상 파일 목록 + 절대 경로
  2. AC 목록 (행동 + 기대 결과 + 검증 방법)
  3. 이전 단계 산출물 요약 (핵심 결정, 주의사항)
  4. 관련 코드 스니펫 (이미 읽은 핵심 부분)
  5. 실패 이력 (있다면: 무엇을 시도했고 왜 실패했는지)
}
```

에이전트는 패키지에 포함된 파일부터 읽고, 필요한 경우에만 추가 탐색한다.

## GitHub 이슈 실시간 기록 (CRITICAL)

**GitHub 이슈 번호가 연관된 작업은, 모든 ACE 단계 진행 과정을 해당 이슈에 코멘트로 남긴다.**

| 시점 | 기록 내용 |
|------|-----------|
| 브랜치 생성 | 브랜치명 |
| ACE 0/6 완료 | Merlin 기획 요약 |
| ACE 1/6 완료 | AC 목록 |
| ACE 2/6 완료 | 생성된 테스트 수 |
| ACE 3/6 완료 | 변경 파일 목록, 주요 수정 내용 |
| ACE 4/6 완료 | 검증 결과 (PASS/FAIL), 루프 횟수 |
| ACE 5/6 완료 | Jude 감사 결과 |
| 머지 완료 | 이슈 close + 최종 요약, 커밋 해시 |

**코멘트 형식:**
```markdown
## ACE {N}/6: {단계 이름}

{핵심 내용 — 3~5줄 이내}

- 변경: `파일경로` — 변경 설명
- 결과: PASS / FAIL / 대기
```

**이슈 기록 누락은 CRITICAL 위반.**

### 이슈 기록 하네스 (구조적 강제)

1. **스킬 레벨**: 각 ACE 단계 완료 시 `gh issue comment`가 **다음 단계 진입 조건**. 코멘트 없이 다음 단계 진행 금지.
2. **커밋 레벨**: `pre_commit_review.py`가 커밋 메시지의 `#N`을 감지하여, 해당 이슈에 오늘 코멘트가 없으면 커밋 차단.

**ACE 단계 전환 시 필수 절차:**
```
ACE N/6 작업 완료
  → gh issue comment {번호} --body "ACE N/6 결과..."
  → 코멘트 성공 확인
  → ACE (N+1)/6 진입
```

## 트리거

- `/ace` (사용자 호출)
- 기능 개발, 버그 수정, 리팩토링 등 코드 변경이 수반되는 작업 요청 시 자동 적용

## 워크플로우

```
[사전] 브랜치 생성 — main에서 fix/ 또는 feat/ 브랜치 분기
[ACE 0/6] 기획 — Merlin 상세 기획서 산출 (What + Why)
[ACE 1/6] AC 작성 — Claude가 기획서 기반으로 AC 정의 (How to verify)
[ACE 2/6] 테스트 생성 — AC 기반 테스트 코드 자동 생성 (구현 전)
[ACE 3/6] 구현 — Alex 설계 + 구현
[ACE 4/6] 검증 루프 — 단위 테스트 → Gini 검수 → 배포 → E2E (최대 3회)
[ACE 5/6] Jude 감사 + 보고 — Jude 독립 재검증 → Claude가 결과 전달
[사후] 머지 + 정리 — main 머지 → push → 브랜치 삭제 (로컬+리모트)
```

### 브랜치 규칙

**모든 ACE 작업은 브랜치에서 진행한다. main에서 직접 작업 절대 금지.**

```bash
# 작업 시작
git checkout main && git pull && git checkout -b fix/{이슈명}  # 또는 feat/{기능명}

# 작업 완료 후 (Jude PASS + 사용자 확인 후)
git checkout main && git merge {브랜치명} && git push
git branch -d {브랜치명} && git push origin --delete {브랜치명}
```

## 상세

### [ACE 0/6] 기획

사용자의 요청을 **Merlin(기획자)**에게 전달하여 상세 기획서를 산출한다.

**Merlin에게 전달할 것:**
- 사용자의 요청 원문
- 관련 문서가 있으면 함께 전달

**Merlin이 산출하는 것:**
```
1. 목적 — 무엇을 왜 해야 하는지
2. 기준 정의 — 어떤 화면/구현이 정답인지 (코드 레벨)
3. 현재 상태 분석 — 전수 조사 결과 (문서 맹신 금지, 코드 직접 확인)
4. 변경 범위 — 수정 대상 파일:라인 + 현재→목표
5. 엣지 케이스 — SVG 구조 차이, 인라인 스타일, opacity 등
6. 범위 밖 — 건드리지 않는 것
7. 위험 요소 — 문서와 실제의 괴리, 검증 시 주의점
```

**ACE 0/6 + 1/6 통합 승인:**
```
Merlin 기획서 산출 → Claude가 기획서 기반으로 AC 작성
→ 기획서 + AC를 한 묶음으로 사용자에게 제시 (승인 1회)
→ 부분 반려 가능: "기획은 OK, AC#3 수정" 식
→ 승인 시 ACE 2/6으로 진행
```

```
[ACE 0/6] 기획 — Merlin 기획서 산출 완료. 변경 대상 5개 파일, 엣지 케이스 3건 식별
[ACE 1/6] AC 작성 — 기획서 기반 AC 3건 작성 완료. 승인 대기
```

### [ACE 1/6] AC 작성

Merlin 기획서를 기반으로 행동 기반 완료 기준(AC)을 작성한다.
**기획서와 AC를 한 묶음으로 사용자에게 제시하여 1회만 승인받는다.**

**형식:**
```markdown
## 완료 기준 (AC)

| # | 행동                  | 기대 결과              | 검증 방법      |
|---|----------------------|----------------------|--------------|
| 1 | "이름" 칼럼 클릭        | 가나다순 정렬           | Playwright   |
| 2 | 한 번 더 클릭          | 역순 정렬              | Playwright   |
| 3 | 빈값("-") 문서 위치     | asc 맨 뒤, desc 맨 앞  | API 검증      |

범위 밖: (건드리지 않는 것 명시)
→ 승인 / 수정 요청
```

**규칙:**
- 전문용어 최소화. "누르면 이렇게 된다" 식 행동 기반
- 검증 방법을 AC 단계에서 미리 명시
- 범위 밖을 명확히 적어 scope creep 방지
- **기획서 + AC를 한 묶음으로 제시. 사용자 승인 1회. 부분 반려 가능**
- **사용자 승인 후 다음 단계로 진행. 승인 없이 구현 시작 금지**

### [ACE 2/6] 테스트 생성

AC 각 항목을 테스트 코드로 자동 변환한다. **구현 전에 테스트를 먼저 만든다.**

두 트랙으로 구성:

```
[ACE 2/6] 테스트 생성
  ├─ (A) AC 테스트: 새 동작 검증
  └─ (B) Regression 테스트: 기존 동작 보호
```

#### (A) AC 테스트 — 새 동작 검증

작업 유형별 테스트 도구:
- 프론트엔드: Playwright E2E
- 백엔드 API: Jest/Pytest + curl
- 파이프라인: Pytest + DB 쿼리
- 배치/크론: DB 사전/사후 비교

#### (B) Regression 테스트 — 기존 동작 보호

변경으로 인해 깨질 수 있는 기존 동작을 보호하는 테스트.

| 작업 유형 | Regression 대상 | 예시 |
|-----------|----------------|------|
| 버그 수정 | 버그 재현 테스트 (수정 전 FAIL, 수정 후 PASS) | 빈 PDF 업로드 시 에러 재현 |
| 기능 추가 | 변경 영향권의 기존 동작 | 정렬 추가 시 기존 필터링 깨지지 않는지 |
| 리팩토링 | 리팩토링 대상의 현재 동작 스냅샷 | API 응답 구조 동일성 |

**테스트 레이어 선택**: 변경 레이어에 따라 자동 결정
- 백엔드/로직 변경 → vitest/pytest
- UI 동작 변경 → Playwright

**Regression 면제 대상** (테스트 불요):
- CSS 미세조정
- 텍스트/번역 수정
- 주석/문서 변경
- 설정값 변경

```
[ACE 2/6] 테스트 생성 — AC 3건 → AC 테스트 4건 + Regression 테스트 2건 생성 완료
```

### [ACE 3/6] 구현

Alex에게 위임하여 설계 + 구현. 큰 변경 시 설계를 사용자에게 승인 요청.

**위임 시 필수 컨텍스트 패키지:**
```
1. [목적] 무엇을 달성해야 하는가
2. [범위] 변경 대상 파일 + 영향 범위
3. [데이터 특성] 비즈니스 로직 제약 (null 가능 여부, 데이터 분포 등)
4. [기대 결과] AC 목록
5. [이전 시도] 실패한 접근법과 이유 (있다면)
```

```
[ACE 3/6] 구현 — 파일 3개 수정 완료
```

### [ACE 4/6] 검증 루프 (dev 환경)

**검증은 dev(`https://localhost:5177`)에서 수행한다. prd에서 검증하지 않는다.**

자동으로 반복. 사용자 개입 없음. 최대 3회.

```
루프 1회차:
  [ACE 4/6] 검증 루프 (1/3) — 단위 테스트 실행 중...
  [ACE 4/6] 검증 루프 (1/3) — 단위 테스트 PASS. Gini 검수 중...
  [ACE 4/6] 검증 루프 (1/3) — Gini PASS. 배포 중...
  [ACE 4/6] 검증 루프 (1/3) — 배포 완료. E2E 테스트 실행 중...
  [ACE 4/6] 검증 루프 (1/3) — E2E FAIL (AC#2 실패). 수정 중...

루프 2회차:
  [ACE 4/6] 검증 루프 (2/3) — 수정 완료. 단위 테스트 재실행...
  ...

3회 실패 시:
  [ACE 4/6] 검증 루프 (3/3) — 3회 실패. 자동 중단.
  → 사용자에게 실패 사유 보고 + 방향 재논의 요청
  → 토의 후 방향 수정 → 다시 [ACE 3/6]부터 재시작 (루프 3회 리셋)
```

**경로별 검증 수준:**

| 경로 | 검증 내용 |
|------|-----------|
| S | 빌드 + Gini 코드리뷰만 |
| M | 빌드 + (단위테스트 ∥ Gini) → 로컬 PASS 후 배포 1회 → E2E |
| L | 빌드 + (단위테스트 ∥ Gini) → 배포 → (E2E ∥ Mira) |

∥ = 병렬 실행

**L 경로 검증 순서 (현행):**
1. 단위 테스트 (`npm test` / `pytest`) — Gini 코드 검수와 병렬
2. Gini 코드 검수 (증거 포함) — 단위 테스트와 병렬
3. 배포
4. E2E 테스트 (AC 기반, Playwright/curl/DB) — Mira 시각 검증과 병렬

**실패 시:**
- 실패 원인 분석 → 자동 수정 → 다음 루프
- 수정 내용을 로그로 표시

### [ACE 5/6] Jude 감사 + 증거 보고

ACE 4/6 통과 후, **Jude(AC 독립 감사관)**에게 최종 검증을 위임한다.
Claude는 Jude의 판정 결과를 사용자에게 전달만 한다.

#### Jude 감사 프로세스 (증거 감사 방식)

Jude는 **4/6의 증거 패키지를 먼저 검토**하고, 의심 항목만 직접 재실행한다.
"항상 처음부터 전부 다시"가 아니라, **"증거를 거부할 자유 + 선택적 재실행"**이 독립성이다.

근거: IEEE 1028 Audit/Inspection 분리, DO-178C 독립 검증자 패턴

```
[ACE 5/6] Jude 감사 — 증거 패키지 수령 → 3단계 감사
  Phase A: 테스트 코드 감사 — "이 테스트가 AC를 제대로 검증하는가?"
  Phase B: 증거 검토 — 스크린샷/API 응답을 직접 열어서 확인
  Phase C: 선택적 재실행 — 의심 항목 + 무작위 1건만 Playwright
  → 최종 PASS/FAIL 판정
```

**Jude에게 전달할 컨텍스트:**
```
1. AC 목록 (행동 + 기대 결과 + 검증 방법)
2. 증거 패키지 경로 (/d/tmp/ace-evidence/)
   — summary.md, 테스트 코드, 스크린샷, API 응답, Gini 리포트
3. 배포 URL / 접속 정보
4. 변경된 파일 목록 (git diff --name-only)
```

#### Jude 감사 3단계

**Phase A: 테스트 코드 감사 (Jude의 핵심 가치)**
- 4/6에서 실행한 테스트 코드를 읽고, AC의 기대 결과를 제대로 검증하는지 판단
- "이 테스트가 통과해도 AC가 충족되지 않는 경우가 있는가?" 를 검토
- 예: AC가 "가나다순 정렬"인데 테스트가 첫 번째 항목만 확인 → FAIL

**Phase B: 증거 검토**
- 스크린샷을 Read로 직접 열어서 시각 확인
- API 응답을 직접 읽어서 기대값과 대조
- Gini 리포트 검토

**Phase C: 선택적 재실행**
- Phase A/B에서 의심이 드는 AC만 Playwright로 직접 재실행
- 추가로, **무작위 1건**을 반드시 직접 재실행 (완전 스킵 방지)
- 의심 근거 없이 전체 재실행하지 않는다

#### Jude PASS 시 → 사용자 보고

```markdown
## ACE 완료 보고

### Jude 감사 결과: PASS

| # | AC | 감사 방법 | Jude 판정 | 증거 |
|---|-----|----------|:---------:|------|
| 1 | "이름" 클릭 → 가나다순 | 증거 검토 | ✅ PASS | [4/6 스크린샷 확인] |
| 2 | 다시 클릭 → 역순 | 직접 재실행 | ✅ PASS | [Jude 독립 스크린샷] |
| 3 | 빈값 위치 올바름 | 증거 검토 | ✅ PASS | [4/6 API 응답 확인] |

테스트 코드 감사: 3건 모두 AC와 일치
직접 재실행: AC#2 (무작위 선택)
검증 루프: 2회차에서 PASS
변경 파일: 3개
→ 브라우저 하드 리프레시(Ctrl+Shift+R) 후 확인해 주세요
```

#### Jude FAIL 시 → 해당 AC만 수정 후 재감사

```

  → Claude가 거절 노트 분석
  → AC#2만 수정
  → AC#2 테스트만 재실행 + 새 증거 수집
  → Jude 재감사: AC#2만 재검증

     + 영향 있는 AC만 추가 재검증 (전체 재실행 아님)
  → 최대 2회 재감사 (총 3회)
  → 3회 FAIL 시 사용자에게 보고 + 방향 재논의
```

#### 보고 규칙

- 결론을 맨 위에 (Jude PASS/FAIL)
- PASS 항목은 한 줄, FAIL 항목만 상세 전개
- 증거는 **감사 방법에 따라 다름**: 증거 검토 = 4/6 증거 참조, 직접 재실행 = Jude 독립 증거
- 미검증 항목이 있으면 "미검증" 섹션에 명시. 미검증 0건이어야 완료
- **사용자가 "완료됐나?" 물어볼 일이 없도록, Claude가 먼저 능동적으로 보고**
- **프론트엔드 변경 시 "브라우저 하드 리프레시(Ctrl+Shift+R) 후 확인해 주세요" 안내 필수**
- **배포 완료 + Jude PASS + 확인 방법을 한 번에 보고. 쪼개서 보고하지 않음**
- **사용자에게 테스트를 시키지 않는다. 모든 검증을 Jude가 완료한 뒤, 최종 확인만 요청한다**

## PASS 등급

| 등급 | 의미 | 사용자 보고 가능 |
|------|------|:--------------:|
| Code PASS | 코드 리뷰 + 빌드 성공 | ❌ |
| Mock PASS | 단위 테스트 통과 | ❌ |
| **Data PASS** | 실제 데이터로 검증 통과 | ✅ |

**Data PASS만 사용자에게 보고 가능.**
Jude는 Data PASS 여부를 독립적으로 판정한다. Claude의 Data PASS 주장을 그대로 수용하지 않는다.

## 주의사항

- AC 승인 없이 구현 시작 금지
- Data PASS 없이 완료 보고 금지
- 미확인은 "미확인"으로 명시. 뒤늦은 고백 금지
- 프로세스는 모든 작업 유형(프론트엔드/백엔드/파이프라인/배치)에 동일 적용. 검증 도구만 다름
- **AC에 명시된 검증 방법을 반드시 준수한다. "Playwright"로 명시된 AC를 "Code PASS"로 대체하는 것은 절대 금지. AC 조건이 잘못된 경우(물리적으로 검증 불가능 등)에만 사용자에게 사전 보고 후 검증 방법 변경을 요청할 수 있다. 사후 변경 금지.**
- **모든 AC가 PASS되어야 완료 보고 가능. 일부 PASS + 일부 Code PASS로 완료 보고하는 것은 사기이며 CRITICAL 위반이다.**
- **Jude FAIL 시 사용자에게 보고 금지. 수정 후 Jude 재감사를 거쳐야 한다.**
- **Claude는 Jude의 판정을 조작하거나 무시할 수 없다. Jude FAIL = 보고 불가.**
- **E2E 테스트 로그인: e2e-tester 에이전트에게 카카오 로그인을 위임하지 않는다. Claude가 직접 Playwright MCP로 로그인 완료 후 테스트를 위임하거나 직접 수행한다. 로그인 정보는 `D:\aims\.playwright-mcp\kakao.txt`에 있다.**
