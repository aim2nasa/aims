# ACE Process

**AC first, Code, Evidence** — 기준 먼저, 코드, 증거.

2026-03-28 전원 합의. 모든 구현 작업에 적용하는 자동화 개발 프로세스.

## 트리거

- `/ace` (사용자 호출)
- 기능 개발, 버그 수정, 리팩토링 등 코드 변경이 수반되는 작업 요청 시 자동 적용

## 워크플로우

```
[ACE 1/5] AC 작성 — 완료 기준 정의
[ACE 2/5] 테스트 생성 — AC 기반 테스트 코드 자동 생성 (구현 전)
[ACE 3/5] 구현 — Alex 설계 + 구현
[ACE 4/5] 검증 루프 — 단위 테스트 → Gini 검수 → 배포 → E2E (최대 3회)
[ACE 5/5] 증거 보고 — AC-증거 대조표 + 화면 캡처
```

## 상세

### [ACE 1/5] AC 작성

사용자의 요구사항을 행동 기반 완료 기준(AC)으로 변환한다.

**형식:**
```markdown
## 완료 기준 (AC)

| # | 행동                  | 기대 결과              | 검증 방법      |
|---|----------------------|----------------------|--------------|
| 1 | "이름" 칼럼 클릭        | 가나다순 정렬           | Playwright   |
| 2 | 한 번 더 클릭          | 역순 정렬              | Playwright   |
| 3 | 빈값("-") 문서 위치     | asc 맨 뒤, desc 맨 앞  | API 검증      |

범위 밖: (건드리지 않는 것 명시)
→ 승인 / 수정 요청
```

**규칙:**
- 전문용어 최소화. "누르면 이렇게 된다" 식 행동 기반
- 검증 방법을 AC 단계에서 미리 명시
- 범위 밖을 명확히 적어 scope creep 방지
- **사용자 승인 후 다음 단계로 진행. 승인 없이 구현 시작 금지**

### [ACE 2/5] 테스트 생성

AC 각 항목을 테스트 코드로 자동 변환한다. **구현 전에 테스트를 먼저 만든다.**

작업 유형별 테스트 도구:
- 프론트엔드: Playwright E2E
- 백엔드 API: Jest/Pytest + curl
- 파이프라인: Pytest + DB 쿼리
- 배치/크론: DB 사전/사후 비교

```
[ACE 2/5] 테스트 생성 — AC 3건 → 테스트 6건 생성 완료
```

### [ACE 3/5] 구현

Alex에게 위임하여 설계 + 구현. 큰 변경 시 설계를 사용자에게 승인 요청.

**위임 시 필수 컨텍스트 패키지:**
```
1. [목적] 무엇을 달성해야 하는가
2. [범위] 변경 대상 파일 + 영향 범위
3. [데이터 특성] 비즈니스 로직 제약 (null 가능 여부, 데이터 분포 등)
4. [기대 결과] AC 목록
5. [이전 시도] 실패한 접근법과 이유 (있다면)
```

```
[ACE 3/5] 구현 — 파일 3개 수정 완료
```

### [ACE 4/5] 검증 루프

자동으로 반복. 사용자 개입 없음. 최대 3회.

```
루프 1회차:
  [ACE 4/5] 검증 루프 (1/3) — 단위 테스트 실행 중...
  [ACE 4/5] 검증 루프 (1/3) — 단위 테스트 PASS. Gini 검수 중...
  [ACE 4/5] 검증 루프 (1/3) — Gini PASS. 배포 중...
  [ACE 4/5] 검증 루프 (1/3) — 배포 완료. E2E 테스트 실행 중...
  [ACE 4/5] 검증 루프 (1/3) — E2E FAIL (AC#2 실패). 수정 중...

루프 2회차:
  [ACE 4/5] 검증 루프 (2/3) — 수정 완료. 단위 테스트 재실행...
  ...

3회 실패 시:
  [ACE 4/5] 검증 루프 (3/3) — 3회 실패. 자동 중단.
  → 사용자에게 실패 사유 보고 + 방향 재논의 요청
  → 토의 후 방향 수정 → 다시 [ACE 3/5]부터 재시작 (루프 3회 리셋)
```

**검증 순서:**
1. 단위 테스트 (`npm test` / `pytest`)
2. Gini 코드 검수 (증거 포함)
3. 배포
4. E2E 테스트 (AC 기반, Playwright/curl/DB)

**실패 시:**
- 실패 원인 분석 → 자동 수정 → 다음 루프
- 수정 내용을 로그로 표시

### [ACE 5/5] 증거 보고

AC와 검증 결과를 1:1 대조표로 보고한다.

**형식:**
```markdown
## ACE 완료 보고

| # | 약속한 것             | 결과 | 증거           |
|---|---------------------|------|---------------|
| 1 | "이름" 클릭 → 가나다순 | PASS | [스크린샷 링크] |
| 2 | 다시 클릭 → 역순      | PASS | [스크린샷 링크] |
| 3 | 빈값 위치 올바름       | PASS | [API 응답]     |

검증 루프: 2회차에서 PASS (1회차 AC#2 실패 → 수정 후 통과)
변경 파일: 3개
→ 확인 완료 / 문제 발견
```

**규칙:**
- 결론을 맨 위에 (PASS/FAIL)
- PASS 항목은 한 줄, FAIL 항목만 상세 전개
- 증거(스크린샷/API 응답/DB 결과)를 반드시 첨부
- 미검증 항목이 있으면 "미검증" 섹션에 명시. 미검증 0건이어야 완료

## 서브에이전트 결과 검증 규칙

Claude는 서브에이전트(Alex, Gini, e2e-tester)의 PASS를 그대로 전달하지 않는다.

```
서브에이전트 PASS 수신 시 체크리스트:
[ ] 실제 데이터 값(JSON/스크린샷)이 첨부되어 있는가?
[ ] 예상값이 먼저 정의되어 있는가?
[ ] 경계값/역케이스를 테스트했는가?

2개 이상 "없음" → PASS 전달 금지, 재검증 요청
```

## PASS 등급

| 등급 | 의미 | 사용자 보고 가능 |
|------|------|:--------------:|
| Code PASS | 코드 리뷰 + 빌드 성공 | ❌ |
| Mock PASS | 단위 테스트 통과 | ❌ |
| **Data PASS** | 실제 데이터로 검증 통과 | ✅ |

**Data PASS만 사용자에게 보고 가능.**

## 진행 상황 표시

모든 단계에서 전체 계획 대비 현재 위치를 표시한다:

```
[ACE 1/5] AC 작성 — 사용자 요구사항 분석 중
[ACE 2/5] 테스트 생성 — AC 3건 → 테스트 코드 생성 중
[ACE 3/5] 구현 — 파일 2/3 수정 완료
[ACE 4/5] 검증 루프 (1/3) — E2E 테스트 실행 중 (AC 2/3 통과)
[ACE 5/5] 증거 보고 — AC-증거 대조표 작성 중
```

문제 없으면 조용히 진행. 1분 이상 걸리는 단계는 진입 알림 1회.
실패 시 즉시 표시: `[ACE 4/5] 검증 루프 (1/3) — FAIL (AC#2). 수정 중...`

## 주의사항

- AC 승인 없이 구현 시작 금지
- Data PASS 없이 완료 보고 금지
- 미확인은 "미확인"으로 명시. 뒤늦은 고백 금지
- 프로세스는 모든 작업 유형(프론트엔드/백엔드/파이프라인/배치)에 동일 적용. 검증 도구만 다름
