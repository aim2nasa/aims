# AIMS 개인정보보호 리스크 검토

> **목적:** 사업계획서에는 포함하지 않지만, 심사·사업 운영 과정에서 반드시 대비해야 할 개인정보보호 이슈를 정리한다.
> **상태:** 검토 중 (법률 자문 필요)
> **최종 수정:** 2026-02-26

---

## 1. 이슈 발생 배경

### 현재 설계사들의 고객 정보 보관 실태

설계사들은 고객의 보험증권, 건강정보, 계약서 등 민감한 개인정보를 다음과 같은 방법으로 보관하고 있다:

| 보관 방법 | 보안 수준 |
|-----------|-----------|
| 카카오톡 나에게 보내기 | 암호화 없음, 접근 제어 없음, 감사 로그 없음 |
| 스마트폰 사진첩 | 기기 분실 시 데이터 노출, 백업 시 클라우드 자동 업로드 |
| 개인 클라우드 (네이버/구글) | 기본 보안만, 금융 데이터 전용 보안 아님 |
| PC 로컬 폴더 | 암호화 없음, 백업 없음, PC 교체 시 유실 |
| 종이 파일 | 물리적 보안만, 분실·도난 리스크 |

**현재 상태는 사실상 개인정보보호법 위반 소지가 크다.** 그러나 이를 사업계획서에서 공격 포인트로 활용하면, "그럼 AIMS는 이 문제에서 자유로운가?"라는 역공을 받게 된다.

### 전략적 판단

- **사업계획서에서는 이 이슈를 언급하지 않는다** (칼을 꺼내지 않는다)
- **대신 방패를 철저하게 준비한다** (질문 받았을 때 완벽하게 답변할 수 있도록)
- AIMS를 "문제를 없애는 것"이 아니라 "관리 가능한 상태로 바꿔주는 것"으로 포지셔닝

## 2. 핵심 쟁점 — 3개 층위

### 2-1. 1층: 법적 근거 (가장 위험)

> **"설계사가 고객 정보를 제3자 플랫폼(AIMS)에 저장해도 되는가?"**

기술적 보안이 아무리 좋아도, 법적 근거가 없으면 의미가 없다. 이것이 가장 먼저 해결해야 할 문제.

#### 데이터 유형별 분석

| 데이터 유형 | 원래 출처 | AIMS 저장 시 쟁점 |
|-------------|-----------|-------------------|
| 보험사에서 다운로드한 PDF (증권, Annual Report 등) | 보험사 시스템 | 보험사 데이터를 제3자 플랫폼으로 이전하는 것. 보험사 이용약관 위반 가능성 |
| 고객이 설계사에게 직접 전달한 서류 | 고객 본인 | 고객 동의 범위 문제. "설계사에게 준 것"과 "설계사가 쓰는 플랫폼에 저장"은 다를 수 있음 |
| 설계사가 직접 작성한 메모·상담 기록 | 설계사 본인 | 상대적으로 리스크 낮음. 단, 고객 식별 정보 포함 시 개인정보에 해당 |
| 건강정보, 질병 이력 | 고객/의료기관 | 민감정보(신용정보법상 특별 보호 대상). 가장 높은 리스크 |

#### 소유권 문제

- **보험사 입장:** "우리 시스템에서 생성된 데이터는 우리 것이다. 외부 반출은 약관 위반."
- **설계사 입장:** "내 고객이고, 내가 영업 활동을 위해 수집한 정보다."
- **고객 입장:** "내 개인정보다. 누가 어디에 저장하는지 알 권리가 있다."
- **AIMS 입장:** 이 삼자 관계에서 어떤 법적 지위를 갖는가?

#### 법적 관계 구조 (검토 필요)

```
고객 ──(개인정보 제공)──► 설계사 ──(업무 위탁?)──► AIMS
                              │
                         보험사 시스템
                        (PDF 다운로드)
```

- AIMS는 「개인정보 처리 위탁」을 받는 **수탁자**인가?
- 아니면 설계사가 자발적으로 사용하는 **도구**에 불과한가?
- 위탁이라면 위탁 계약 체결이 필요하고, 도구라면 설계사 본인 책임
- **이 구분이 AIMS의 법적 책임 범위를 결정한다**

### 2-2. 2층: 보안 조치

> **"저장한다면 어떻게 보호하는가?"**

법적 근거가 확보되었다는 전제 하에, 기술적·관리적 보호 조치:

| 영역 | 조치 | 현재 상태 | 필요 조치 |
|------|------|-----------|-----------|
| 전송 암호화 | HTTPS/TLS | ✅ 적용됨 | - |
| 저장 암호화 | AES-256 등 | ❓ 확인 필요 | MongoDB 암호화 설정 검토 |
| 접근 제어 | 설계사별 데이터 격리 | ✅ 사용자별 분리 | 다중 인증(MFA) 검토 |
| 감사 로그 | 접근·수정·삭제 이력 | ❓ 확인 필요 | 로그 보존 기간 정책 수립 |
| 데이터 보존 | 보존 기간·삭제 정책 | ❌ 미수립 | 정책 수립 필요 |
| 백업·복구 | 정기 백업, 복구 절차 | ❓ 확인 필요 | 절차 문서화 |

### 2-3. 3층: 사고 대응

> **"유출되면 어떻게 하는가?"**

- 개인정보 침해 사고 발생 시 72시간 내 개인정보보호위원회 및 정보주체에 통지 의무
- 사고 대응 절차(Incident Response Plan) 수립 필요
- 배상 책임 범위 정의

## 3. 예상 공격 시나리오와 대응 논리

### 시나리오 A: 심사위원 질문

> "고객 동의 없이 제3자 플랫폼에 개인정보를 저장하는 거 아닌가요?"

**대응 논리:**
- 설계사가 업무상 고객 정보를 정리하는 행위 자체는 보험업법상 정당한 영업 활동
- AIMS는 설계사의 업무 도구이며, 이는 의사가 EMR을 사용하는 것과 같은 맥락
- 개인정보 처리방침을 통해 고객에게 고지하고, 필요 시 동의를 받는 절차를 갖출 예정
- (이 논리가 법적으로 유효한지 자문 필요)

### 시나리오 B: 보험사 반발

> "우리 시스템에서 다운로드한 데이터를 외부 서비스에 저장하는 건 약관 위반입니다."

**대응 논리:**
- 설계사가 PDF를 다운로드하는 행위는 보험사가 허용한 기능
- 다운로드한 파일을 PC에 저장하든, 클라우드에 저장하든, AIMS에 저장하든 설계사의 업무 환경 선택
- 단, 보험사별 이용약관에 "외부 반출 금지" 조항이 있는지 실제 확인 필요
- AutoClicker로 자동 다운로드하는 부분은 별도 리스크 (이용약관상 자동화 금지 가능성)

### 시나리오 C: 고객 문제 제기

> "제 보험 정보가 왜 다른 서비스에 있나요?"

**대응 논리:**
- "담당 설계사님이 고객님의 보험을 체계적으로 관리하기 위해 사용하는 업무 도구입니다"
- "고객님의 데이터는 암호화되어 안전하게 보관되며, 담당 설계사만 접근할 수 있습니다"
- 개인정보 처리방침에 AIMS 사용 사실을 명시하고, 열람·삭제 요청 절차를 안내

### 시나리오 D: 데이터 유출 사고

> 해킹 등으로 고객 정보가 유출된 경우

**대응:**
- 이 시나리오가 발생하면 법적 근거가 아무리 탄탄해도 타격이 큼
- 사전 예방이 최우선: 침투 테스트, 보안 감사, 최소 권한 원칙
- 사고 대응 계획(IRP) 수립, 사이버 배상 보험 가입 검토
- 72시간 내 통지 의무 이행 절차 사전 수립

## 4. AIMS의 포지셔닝 논리

### "카톡·사진첩 vs AIMS" 비교 (내부용, 대외 발표 아님)

| 항목 | 카톡·사진첩·PC 폴더 | AIMS |
|------|---------------------|------|
| 암호화 | ❌ 없음 | ✅ 전송·저장 암호화 |
| 접근 제어 | ❌ 기기 보유자 누구나 | ✅ 인증된 사용자만 |
| 감사 로그 | ❌ 없음 | ✅ 접근·수정 이력 기록 |
| 데이터 격리 | ❌ 개인 데이터와 혼재 | ✅ 업무 데이터 분리 |
| 분실·도난 대응 | ❌ 기기 분실 = 정보 유출 | ✅ 원격 접근 차단 가능 |
| 삭제 요청 대응 | ❌ 여러 기기에서 개별 삭제 | ✅ 중앙에서 일괄 삭제 |

**핵심 논리:** "개인정보를 다루는 것 자체는 피할 수 없다. 문제는 그것을 관리 없이 방치하느냐, 체계적으로 보호하느냐의 차이다."

## 5. 액션 아이템

### 즉시 (사업 초기)
- [ ] 개인정보 처리방침 초안 작성
- [ ] 이용약관에 데이터 보관·처리 범위 명시
- [ ] 주요 보험사(삼성·한화·DB·메리츠 등) 설계사용 시스템 이용약관에서 "외부 반출 금지" 조항 확인
- [ ] 설계사-AIMS 간 법적 관계 정의 (위탁 vs 도구)

### 단기 (6개월 내)
- [ ] 개인정보보호 전문 법률 자문 수행
- [ ] 고객 동의 절차 설계 (설계사가 AIMS에 고객 등록 시 고객 동의 프로세스)
- [ ] 데이터 보존·삭제 정책 수립
- [ ] 보안 조치 현황 점검 및 보완 (암호화, 감사 로그 등)

### 중기 (1년 내)
- [ ] 개인정보 영향평가 실시
- [ ] ISMS-P 인증 준비 (또는 ISO 27001)
- [ ] 사이버 배상 보험 가입 검토
- [ ] 침투 테스트 실시

## 6. 참고 법령

- **개인정보보호법** — 개인정보 수집·이용·제공·위탁의 법적 근거
- **신용정보의 이용 및 보호에 관한 법률** — 금융 분야 개인정보(신용정보) 특별법
- **보험업법** — 설계사의 영업 활동 범위, 고객 정보 관리 의무
- **금융분야 개인정보보호 가이드라인** (금융위원회) — 금융회사 및 금융 관련 사업자의 개인정보 보호 기준
- **전자금융거래법** — IT 시스템 보안 기준 (AIMS가 금융 서비스에 해당하는 경우)

---

> **결론:** 이 이슈는 "기술로 해결할 문제"가 아니라 "법률·사업 구조로 해결할 문제"이다.
> 기술적 보안은 필요조건이지 충분조건이 아니다. 법적 근거 확보가 최우선이며,
> 이를 위해 전문 법률 자문이 반드시 필요하다.
