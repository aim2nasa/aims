## 전체 평가: 완성도 높음

50만 라인 실 구축 + MDRT 자문역이라는 **강력한 차별점**이 있으며, 구조적으로 PSST 프레임워크에 잘 맞춰져 있다. 대부분의 지원자가 "만들겠다"고 계획을 제출하는 반면, AIMS는 50만 라인의 실 동작 시스템을 증거로 제시한다는 점이 압도적 차별점이다.

---

## 1. 문제인식 (Problem) — 강함

### 잘 된 점

- "계약을 보는 보험사 vs 고객을 보는 설계사" — 핵심 간극을 한 문장으로 명쾌하게 정의
- 설계사 현재 대응 수단(종이, 사진, 엑셀, 카톡 등) 구체적 나열
- 기존 디지털 도구 5개 카테고리 비교표 — 경쟁 환경을 입체적으로 분석
- "왜 아무도 못 풀었나" 4개의 벽 + "왜 지금" 기술 성숙 — 설득력 있는 타이밍 논거
- 생산성 손실 데이터(월 4~8시간, 정착률 47.3%) — 정량적 근거 확보

### 보완 제안

- [ ] **고객 인터뷰/고통 사례 추가**: "MDRT 설계사 현장 실측"만으로는 약함. 자문역 외 **2~3명의 설계사 인터뷰 결과**(익명이라도)를 추가하면 "현장의 목소리"가 됨
- [ ] **정착률과 AIMS의 연결 강화**: "문서 정리 시간 절감 → 본업 집중 → 이탈 감소"라는 연결 고리를 한 문장만 더 명시

---

## 2. 실현가능성 (Solution) — 매우 강함 (최대 강점)

### 잘 된 점

- **"계획이 아니라 증거"** — 이 한 줄이 평가위원을 끌어당기는 핵심
- 50만 라인, 8개 마이크로서비스, 395개 API, 4,982 커밋 — 압도적 정량 지표
- 기술 아키텍처 다이어그램 포함
- 모듈별 개발 현황표 (운영 중/개발 중 구분)
- 지식재산권 보유/예정 분리
- Phase 1~3 로드맵이 구체적이고 KPI가 명시됨

### 보완 제안

- [ ] **"발표 시 라이브 데모 가능"** 명시: 발표평가에서 실제 데모를 보여줄 수 있다는 점은 엄청난 무기
- [ ] **자문역 사용 전후 정량 비교 데이터 추가**: 예) "PDF 정리 시간 월 8시간 → 30분", "고객 미팅 준비 30분 → 5분" — "실현가능성"이 "실현됨"으로 승격됨
- [ ] **DGX Spark PoC 결과 추가**: 오픈소스 LLM(Llama 등)이 128GB 통합 메모리에서 실제로 원하는 성능을 낼 수 있는지 검증 데이터 필요

---

## 3. 성장전략 (Scale-up) — 양호, 보완 필요

### 잘 된 점

- TAM-SAM-SOM 분석 구조 충실
- 구독 + 크레딧 하이브리드 모델 — SaaS 업계 트렌드에 맞음
- 크레딧 환산표가 구체적
- 단위 경제성(LTV/CAC 11.8x) 제시
- 3개 시나리오 매출 전망 (보수/기본/낙관)
- MetLife 1등 지점 → 동심원 확장 GTM이 구체적
- 리스크 4개 항목 + 대응 전략

### 보완 제안

- [ ] **가격 지불 의향(WTP) 조사 결과 추가**: 월 49,000원이 설계사에게 적절한지 피드백 인용. 설계사 평균 월 소득 대비 비용 비율 제시도 방법
- [ ] **SOM 29.4억원 근거 보강**: "디지털 수용층 5,000명"의 출처 불분명. MetLife 1등 지점 → MetLife 전체 → GA 확산 경로별 전환율 추정이 더 설득력 있음
- [ ] **B2B AI Engine 구체화**: 타겟 업종 1~2개와 예상 수익을 간략히 추가. 현재는 "중장기"로만 언급되어 평가위원 질문 시 답변 부족
- [ ] **진입장벽(moat) 재정의**: "1인 개발 + AI 생산성"은 다른 기업도 활용 가능. **20년 MDRT의 도메인 지식 + 실사용 피드백 루프**가 진짜 moat이라는 점을 더 강조

---

## 4. 팀 구성 (Team) — 강하지만 약점 노출

### 잘 된 점

- 대표자 32년 경력이 상세하고 인상적
- 스타트업 창업 경험(래셔널) + 기술 수출 실적 — 창업 역량 입증
- 자문역의 MetLife 20년 + MDRT — 도메인 전문성 입증
- "배우자" 관계를 솔직히 밝히면서도 "이해관계 일치, 이탈 불가"로 장점 전환 — 좋은 전략
- 채용 계획 3명 (Phase별 배치)

### 보완 제안

- [ ] **1인 개발 리스크 대응 구체화**: 평가위원은 **"대표가 쓰러지면 사업이 끝나는 것 아니냐"**를 반드시 물을 것. 아래 내용 추가 필요:
  - 코드 문서화 수준 (CLAUDE.md, 아키텍처 문서 등)
  - CI/CD 자동화 수준
  - Phase 1 즉시 풀스택 개발자 채용 → 지식 이전 계획
- [ ] **사회적 가치 차별화**: "종이 절약", "일자리 창출"은 모든 사업계획서에 들어감. **"47만 프리랜서에게 대기업급 AI를 월 49,000원에 제공 — 기술 접근성의 민주화"**가 더 인상적

- [ ] **인건비 현실성 설명**: 파트타임 풀스택 개발자 8개월 1,000만원(월 125만원)으로 확보 가능한 근거 설명 필요

## 발표평가 대비 예상 질문

평가위원이 높은 확률로 물어볼 질문들:

1. **"대표 혼자인데, 건강 문제 등으로 개발 못하면 어떻게 하나요?"**
2. **"월 49,000원을 개인 설계사가 낼까요? 실제로 지불 의향 조사 해보셨나요?"**
3. **"보험사가 비슷한 기능을 만들면 어떻게 대응하나요?"**
4. **"개인정보보호법상 리스크는 어떻게 관리하나요?"**
5. **"DGX Spark에 1,500만원인데, 실제로 원하는 성능이 나오는지 검증했나요?"**
6. **"자문역이 배우자라고 하셨는데, 객관적인 시장 검증은 어떻게 하셨나요?"**
7. **"50만 라인이라고 하셨는데, AI가 생성한 코드 비율은 어느 정도인가요?"**

## 총평

보완이 필요한 부분은 대부분 **"이미 있는 것을 더 잘 보여주는"** 수준의 작업이다. 특히 **자문역의 실사용 정량 데이터**만 추가해도 "문제인식 + 실현가능성 + 팀 구성" 세 항목이 동시에 강화된다.

**가장 시급한 조치**: 자문역에게 AIMS 사용 전후 업무 변화를 정량적으로 정리해달라고 요청하는 것. 이 한 장의 데이터가 사업계획서 전체의 설득력을 한 단계 끌어올린다.
