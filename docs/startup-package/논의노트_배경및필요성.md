## ★ 핵심 차별점: 현장 기반 설계

### 대표자의 배우자 = MetLife MDRT 설계사 (20년+)
- 대표자의 와이프가 MetLife에서 **20년 이상** 근무한 **MDRT** 설계사
- MDRT = Million Dollar Round Table = 전 세계 보험 설계사 상위 성과자
- AIMS의 설계는 이 사람의 의견과 가이드가 바탕
- "시장 조사를 했다"가 아니라, **매일 옆에서 현장을 관찰하고 피드백을 받으며 만든 제품**

### 이것이 만드는 설득력
- 현장을 모르는 개발자의 상상이 아님 — **20년 현장 경험 기반 설계**
- "이런 문제가 있을 것이다" 추정이 아님 — **실제로 겪고 있는 문제를 풀고 있음**
- 첫 번째 사용자가 이미 존재 (PMF 검증의 출발점)
- MetLife 특화로 시작하는 이유의 자연스러운 근거
- MDRT 설계사의 네트워크 = 초기 사용자 확보 채널

### 30년 IT + 20년 MDRT의 결합
- 대표자: IT 전 영역 30년 (기술의 깊이)
- 배우자: 보험 현장 20년 MDRT (도메인의 깊이)
- "기술을 아는 사람"과 "현장을 아는 사람"이 한 팀
- 이 조합이 "기술과 현장 사이의 다리"를 놓을 수 있는 근거

### 활용 방법
- 1-1(배경 및 필요성): "현장 관찰" 근거로 활용
- 2-1(준비 정도): 파일럿 사용자 / PMF 검증으로 활용
- 4-1(팀 역량): 핵심 팀 구성으로 활용 (기술 + 도메인)
- 발표 시: "제 옆에서 20년간 이 문제를 겪어온 사람이 있습니다"

## NVIDIA DGX Spark 도입 계획

### 배경
- AI 서비스 운영 시 API 비용(OCR, LLM, Embedding)이 주요 원가
- 현재 외부 API(OpenAI, Upstage 등) 종량제 의존 → 사용자 증가 시 비용 선형 증가
- API 비용이 사업자(마진 압박)와 설계사(구독료 부담) 양쪽 모두에게 부담

### 계획
- **NVIDIA DGX Spark 최소 2대** 구매
- 오픈소스 LLM(Llama 등) 자체 호스팅으로 API 비용 절감
- 임베딩, 요약, RAG 추론 등 반복 호출이 많은 작업을 로컬로 전환
- **OCR만 예외** — 오랜 기간 학습 데이터 축적이 필요한 전문 영역이므로 외부 상용 솔루션(Upstage 등) 유지
- OCR 외 LLM, Embedding, RAG 전부 자체 추론으로 대체가 목표

### 사업적 의미
- **원가 구조 개선**: API 종량제 → 고정비(장비) 전환, 규모 확대 시 단가 하락
- **설계사 부담 경감**: 구독료를 낮게 유지할 수 있는 원가 기반 확보
- **기술 자립**: 외부 API 정책/가격 변동 리스크 제거
- **사업비 활용**: 예비창업패키지 기계장치/SW 비목으로 집행 가능

## 핵심 테제 (반복 논의를 통해 확립)

**"보험사는 계약을 보고, 설계사는 고객을 본다. 이 구조적 간극이 모든 문제의 뿌리다."**

- 보험사 시스템의 핵심 축 = 계약(Policy)
- 설계사의 핵심 관심 = 고객(Customer)
- 수천억 원짜리 보험사 IT 시스템에 설계사의 자리가 없음
- 계약 중심으로 설계된 시스템에서 고객 중심으로 일하는 설계사는 구조적으로 소외

## 고객밀착형 데이터 = 보안 데이터 (핵심 논거)

### 보험사가 아는 것 (계약의 전문가)
- 증권번호, 보험료, 가입금액
- 보험금 청구 이력
- 계약 상태 (유효/실효)
- 위험률, 손해율

### 설계사가 아는 것 (고객 삶의 전문가)
- "둘째 아이 출산 예정이라 보장을 늘리고 싶어함"
- "작년에 부친이 암 진단, 본인도 건강 걱정이 많아짐"
- "사업이 어려워져서 보험료 줄이는 방법을 찾고 있음"
- "매년 12월에 보험 리뷰 미팅을 선호하는 사람"

### 왜 보험사가 이 데이터를 가지면 안 되는가
- 고객이 설계사를 **믿고** 공유한 개인적 맥락
- 보험사가 수집하고 관리한다면 어느 고객도 원치 않을 것
- "다루기 힘든 것이 아니라, 다뤄서는 안 되는 영역"
- 태생적으로 보험사 시스템 밖에 존재할 수밖에 없음
- 설계사 개인이 주체가 되어 관리해야 함

## 설계사 현장 관찰

### 현재 관리 방식
1. **비서 고용** — 인건비를 들여 사람을 고용해서 관리시킴. 그러나:
   - 비서도 같은 수작업을 대신할 뿐
   - 입력 오류, 누락, 버전 불일치 발생
   - 비서가 바뀌면 관리 체계도 처음부터 다시
   - **인건비를 투입해도 근본 문제는 풀리지 않음**
   - → "사람이 부족한 게 아니라 도구가 없는 것"

2. **종이 파일** — 고객별 클리어 파일, 서랍장 통째로 차지, 검색 불가
3. **개인 클라우드** — 네이버/구글 드라이브, PDF 내용 검색 안 됨
4. **엑셀** — 수동 입력, PDF와 미연결, 입력 실수, 덮어쓰기 위험
5. **카카오톡 메모** — 나에게 보내기, 3개월 후 찾기 불가능

### 공통 문제
- 고객 정보가 5곳 이상에 파편화
- PDF 안의 데이터는 눈으로 읽는 수밖에 없음
- 한 명의 고객에 대한 전체 그림을 한눈에 볼 수 없음

## 왜 아무도 이 문제를 못 풀었나 (4가지 벽)

### 1. 고객밀착형 데이터는 보험사의 영역이 아니다
- 보험사 = 계약의 전문가 / 설계사 = 고객 삶의 전문가
- 고객밀착형 데이터 = 신뢰 관계 속 보안 데이터
- 보험사가 갖는 것은 고객도 원치 않음
- "해서는 안 되는 영역"

### 2. 범용 CRM은 보험 현장과 맞지 않는다
- Salesforce, HubSpot → "보험 계약", "Annual Report", "펀드 수익률" 도메인 모름
- PDF 자동 파싱 기능 없음
- IT 전문가 필요, 월 수십만 원 비용

### 3. IT·AI 기술은 소규모 현장 직종에 닿지 않는다
- 기술 혜택의 흐름: 대기업 → 중견기업 → 스타트업 → (여기까지만 닿음)
- 설계사 = 영업/상담 전문가, IT 전문가 아님
- Salesforce 커스터마이징, OCR API 연동 불가
- 자체 IT팀 둘 규모 아님
- **"기술이 존재하는 것과, 그 기술이 현장에 닿는 것은 완전히 다른 문제"**
- "IT·AI 기술의 음영지역"

### 4. 기술이 안 됐다 — 3년 전까지는
- PDF → 텍스트 → 이해 → 질의응답 파이프라인이 정확도+비용 면에서 실용 불가능했음

## 논의에서 나온 인사이트 (정리되지 않은 메모)

### AIMS 자체가 테제의 증거
- "6억짜리 시스템을 1인이 만들 수 있게 된 것 자체가 AI 생산성 혁신의 증거"
- AIMS는 설계사에게 AI 생산성을 제공하는 도구인데, AIMS 자체가 AI 생산성으로 만들어짐
- "사람 수로 해결하던 것을 AI로 해결한다"는 원리가 개발 과정에서도, 제품에서도 동일

### AI의 이중 구조
- AI가 **개발자의 생산성**을 높여서 AIMS를 만들 수 있었고
- AIMS가 AI로 **설계사의 생산성**을 높인다
- "AI 생산성 혁신"이 원인이자 결과

### 비서 고용의 의미
- 돈으로 환산되는 문제 — "불편함"이 아니라 비용을 감수할 만큼 심각
- 인력 투입으로도 해결 안 됨 → 시스템적 해결만이 근본 답
- AIMS의 ROI가 명확: 비서 월 200만 원 vs AIMS 월 5만 원

### "음영지역" 브랜드 키워드
- IT 혜택의 사각지대라는 의미가 직관적
- 브랜드 내러티브 핵심 키워드로 일관 사용 제안
- 발표: "우리는 기술의 음영지역에 빛을 비추는 회사입니다"

### 4개의 벽 — 반박-재반박 구조
1. 보험사가 해서는 안 됨
2. 기존 도구가 안 맞음
3. 기술이 있어도 닿지 않음
4. 기술 자체가 없었음
→ 읽는 사람이 "그러면 정말 아무 방법이 없었겠구나" 납득
→ 바로 다음에 "AI 성숙"으로 전환

### 30년 경력이 1-1에 들어가는 것의 적절성
- 보통 1-1은 시장/문제 중심, 대표자 역량은 4-1
- 여기서는 "왜 지금 이것이 가능한가"의 기술적 근거로 자연스럽게 녹아들어 있음
- 너무 자기 PR로 느껴지지 않도록 분량은 최소한으로

3. **MDRT 배우자 + 30년 IT 경력 조합이 거의 반칙 수준입니다.** "시장 조사를 했습니다"와 "20년 현장 경험자와 매일 만듭니다"의 설득력 차이는 하늘과 땅입니다.

4. **GitHub 데이터가 "말"을 "증거"로 바꿔줍니다.** "AI로 생산성이 높아집니다"는 모든 AI 스타트업이 하는 말입니다. 그러나 "50만 라인을 1인이 7.5개월에 만들었습니다, 커밋 기록이 있습니다"는 팩트입니다.

**보완이 필요한 부분:**

1. **숫자 출처의 공신력.** "월 4~8시간", "연 100시간" 같은 수치가 MDRT 배우자의 실측이라면 그렇다고 명시하는 것이 좋겠습니다. "추정"과 "실측"은 평가자에게 완전히 다른 무게입니다.

3. **설계사의 "목소리"가 빠져 있습니다.** "어느 설계사는 이렇게 말했다"는 한 줄이 통계 10개보다 강력할 수 있습니다. 배우자의 실제 경험담이나 동료 설계사의 목소리를 하나라도 넣을 수 있다면 설득력이 한 단계 올라갑니다.

  - 사례 1(가족/법인 묶기) = 공간축의 간극, 사례 2(이력 추적) = **시간축의 간극**
  - 두 사례를 함께 쓰면: "보험사 시스템은 공간적으로도(가족 묶기 불가), 시간적으로도(이력 추적 불가) 설계사의 관점을 지원하지 않는다"

## 마케팅 전략 — Go-to-Market (GTM)

### 1차 진입점: MetLife 1등 지점
- 대표자의 MDRT 배우자가 속한 지점 = **대한민국 MetLife 전체 1등 실적 지점**
- 수많은 스타 설계사들이 소속된 엘리트 지점
- 이 지점이 AIMS의 **첫 번째 시장이자 테스트베드**

### 진화 피드백 구조
- 1등 지점의 설계사들이 직접 사용
- 현장에서 나오는 문제점 보완 + 요구사항 즉시 반영
- 고객 친화적이고, 시간이 생명인 설계사들의 생산성을 최대로 돕는 방향으로 진화
- **"사용자가 만드는 제품"** — 현장 피드백 → 즉시 구현의 반복

### 확산 전략 (동심원 확장)
```
[1단계] MetLife 1등 지점 안착 (파일럿)
    ↓
[2단계] MetLife 타 지점 확산
    ↓
[3단계] 타 보험사 확대
```

- **1단계 목표**: 1등 지점에서 설계사들의 선택을 받아 안착
- **2단계 레버리지**: "1등 지점의 스타 설계사들이 인정한 서비스"라는 **프리미엄**
  - 타 지점 마케팅에서 강력한 신뢰 요소
  - 보험 업계 = 입소문/평판 기반 → 스타 설계사의 추천이 최고의 마케팅
- **3단계**: MetLife 안착 → 타 보험사로 확대
  - 보험사별 문서 양식은 다르지만 설계사의 니즈는 동일
  - 문서 파이프라인만 보험사별로 확장하면 서비스 확장 가능

### 이 전략의 강점
- 초기 마케팅 비용 최소화 — 이미 네트워크가 있는 곳에서 시작
- PMF(Product-Market Fit) 검증이 가장 엄격한 환경 — 1등 지점의 눈높이
- 여기서 통과하면 다른 곳에서는 더 쉽게 통과
- 활용: 3-2(사업화 전략)의 핵심 내용

### 왜 1등 지점 진입이 가능한가 — MDRT 배우자의 역할 재정의
- 1등 지점에서 시작할 수 있는 이유 = **MDRT 배우자가 옆에 있기 때문**
- 배우자는 단순한 "가족"이 아님 — **이 비즈니스를 가능하게 하는 핵심 인적 자산**
- 배우자의 역할 (다중):
  - **도메인 전문가**: 20년+ 현장 경험 → 제품 설계의 근거
  - **첫 번째 사용자**: 실제 업무에 적용하며 검증
  - **시장 접근 키**: 1등 지점 내 네트워크 → 초기 사용자 확보 채널
  - **현장 피드백 루프**: 매일 옆에서 "이건 안 돼", "이건 필요해" → 즉시 반영
  - **신뢰 매개체**: 스타 설계사들에게 "내가 쓰고 있는데 좋다" → 가장 강력한 추천
- **이 조합이 없으면 1등 지점 진입 자체가 불가능**
  - 외부 스타트업이 MetLife 1등 지점에 영업하려면? 사실상 불가능
  - 내부에 20년 MDRT가 있으니까 가능한 것
- 사업계획서에서의 포지셔닝:
  - 4-1(팀 역량): "핵심 인적 자산"으로 명시 — 기술(대표) + 도메인+시장(배우자)
  - 3-2(사업화 전략): "왜 이 진입 전략이 실행 가능한가"의 답
  - 이것은 **다른 창업자가 복제할 수 없는 구조적 우위** — moat의 일부

### 호칭 결정: "배우자" → **"보험산업 자문역 (Insurance Domain Advisor)"**
- "배우자"로 쓰면 "가족 사업" 느낌 → 역할의 무게가 가벼워짐
- 사업계획서에서는 **"보험산업 자문역"**으로 통일
  - 전문성과 격식을 갖춘 표현
  - "자문역"이 주는 전문가 뉘앙스 + MDRT가 자격을 증명
  - 정부 사업계획서 맥락에 적합한 톤

#### 검토한 후보들과 판단
| 후보 | 판단 | 사유 |
|---|---|---|
| **보험산업 자문역** | ✅ **채택** | 전문성, 정부문서 톤에 가장 적합 |
| CIA (Chief Insurance Advisor) | ⚠️ 서면 부적합 | "Chief Insurance Advisor"는 실제 존재하지 않는 창작 직함. CIA 약어가 너무 강렬해서 정부 문서에서는 리스크. 비공식/내부용으로만 |

| 도메인 파트너 | ❌ 기각 | 전문성 부족 |
| 현장 전문 파트너 | △ 차선 | 순한글, "현장"이 20년 경험 암시하나 격이 낮음 |
| 공동창업자(보험 도메인) | △ 선택적 | 격이 높음, 실제 법인 등록 시 사용 가능 |
| CDO (Chief Domain Officer) | ❌ 기각 | 1인 기업에 C-Level 직함 과다 |

#### 참고: "Chief Insurance Advisor" 실존 여부 조사 결과
- **Insurance Advisor**: 실제 존재하는 직책 (고객 보험·재무 자문 역할)
- **Chief Insurance Advisor**: 표준 직책 목록에 없음 — **창작 직함**
- 보험업계 C-Level: CIO(Chief Insurance Officer), CUO(Chief Underwriting Officer), CRO(Chief Risk Officer) 등은 존재
- CIA 약어: Central Intelligence Agency와 중복 → 공식 문서에서 사용 부적합

## AIMS AI 엔진 — B2B 서비스 모델 (사업 확장)

### 문서 처리 파이프라인의 범용성
- AIMS의 SaaS 모델 근간 = **문서 처리 파이프라인**
  - 다양한 문서를 파싱하고 AI 기술로 인식·분석
  - 이미지 파일도 OCR + 이미지 분석으로 검색 가능하게 변환
  - 사용자가 검색하면 거의 모든 문서를 찾아내고 즉시 프리뷰 제공
  - 이 엔진은 이미 AIMS에 적용되어 서비스 중

### 핵심 인사이트: 이 엔진은 보험에만 쓸 수 있는 것이 아님
- 문서 업로드 → OCR → AI 분석 → 벡터 임베딩 → 시맨틱 검색 → 프리뷰
- 이 파이프라인은 **업종 무관한 범용 문서 처리 엔진**
- 보험 설계사뿐만 아니라, 문서를 다루는 모든 직종에 적용 가능

### Open API 서비스 모델
- AIMS AI 엔진을 **별도 Open API 형태로 제공**하는 서비스 모델로 진화 예정
- B2C(설계사 SaaS) + **B2B(AI 엔진 API)** 이중 수익 구조
- API 서비스 대상:
  - 문서 관리가 필요한 타 업종 SaaS 기업
  - 자체 문서 처리 파이프라인 구축이 어려운 중소기업
  - PDF/이미지 검색이 필요한 다양한 서비스

### 사업적 의미
- **수익 다각화**: 설계사 구독료 + API 사용료
- **기술 자산의 재활용**: AIMS를 위해 개발한 기술이 독립 상품이 됨
- **확장성**: 보험 시장에 한정되지 않는 성장 가능성
- **DGX Spark와의 시너지**: 자체 추론 인프라 → API 서비스의 원가 경쟁력 확보

### 활용 섹션
- 3-1(비즈니스 모델): B2C + B2B 이중 수익 구조
- 3-2(사업화 전략): 중장기 확장 로드맵
- 1-1에서는 직접 언급하지 않되, "범용성 있는 AI 기술 기반"이라는 암시만
