**보험사는 "계약"을 보고, 설계사는 "고객"을 봅니다.** 이 구조적 간극 때문에 47만 설계사에게는 전용 디지털 도구가 없습니다. AIMS는 AI 기술로 이 간극에 다리를 놓아, 설계사가 문서 정리자에서 고객 전문가로 돌아갈 수 있게 하는 **AI 기반 지능형 고객·문서 관리 SaaS 플랫폼**입니다.

# 1. 문제인식

## 1-1. 창업아이템 배경 및 필요성

### "계약"을 보는 보험사, "고객"을 보는 설계사

보험사는 수천억 원을 들여 IT 시스템을 구축하지만, 그 핵심 축은 **계약(Policy)**입니다. 정작 현장에서 고객을 대면하는 설계사에게 필요한 것은 **고객(Customer) 중심의 정보**인데, 주어진 도구들은 계약과 상품을 중심으로 설계되어 있습니다.

보험사 시스템은 계약 심사·관리·정산이라는 본연의 목적에 맞게 설계되어 있으므로, 설계사가 현장에서 필요로 하는 **고객 중심의 정보 정리**와는 근본적으로 거리가 있습니다. 예를 들어 가족 전체의 보험 현황을 한눈에 보거나, 과거 해약 이력을 추적하려면 여러 화면의 정보를 **수기로 정리**해야 합니다. 업무량이 많은 설계사들은 이 정리 작업을 위해 비서를 고용하기도 합니다.

더 근본적인 문제는, 보험사 시스템이 태생적으로 **고객밀착형 데이터를 다루기 어렵다**는 점입니다. 설계사는 고객과의 관계를 위해 경조사, SNS 소식, 가족 상황 변화 등 개인적인 맥락까지 파악하고 챙깁니다. 그러나 이런 정보는 고객의 거부감과 개인정보보호 등 여러 이유로 **보험사가 체계적으로 수집하고 관리하기 어려운 영역**입니다. 결국 이 데이터는 설계사의 머릿속이나 개인 메모에만 존재하게 됩니다.

### 설계사의 현재 대응 수단

| 방법 | 한계 |
|------|------|
| 비서 고용 (월 200~300만원) | 같은 수작업 대행. 교체 시 체계 유실 |
| 종이 파일·클리어 파일 | 검색 불가. 고객이 늘수록 물리적 보관 공간 한계 |
| 엑셀 수동 입력 | 원본 문서와 미연결. 덮어쓰기 한 번이면 끝 |
| 개인 클라우드 (네이버/구글) | 파일명으로만 검색 가능, 문서 내용 기반 검색 불가 |
| 스마트폰 사진 촬영 | 사진이 쌓일수록 필요한 서류를 찾기 어려워짐 |
| 카카오톡 나에게 보내기 | 시간이 지나면 대화에 묻혀 찾기 어려움 |

**공통 문제:** PDF, 이미지, HWP, 엑셀 등 다양한 형식의 서류가 여러 곳에 흩어져 있지만, 파일명으로만 검색할 수 있어 내용을 확인하려면 직접 열어봐야 합니다. 보관량이 늘수록 검색에 드는 시간이 커지고, 갖고 있는 자료도 찾지 못하거나 같은 문서가 중복 저장되는 일이 반복됩니다.

### 생산성 손실의 실체

| 손실 유형 | 규모 | 출처 |
|---|---|---|
| 매월 고객별 PDF 수동 다운로드 (100명 기준) | **월 4~8시간** | MDRT 설계사 현장 실측 |
| 미팅 전 고객 정보 수집·정리 | 1건당 **20~30분** | 자문역 설계사 실측 |
| 연간 문서 수집·정리 시간 | **100시간 이상** | 자문역 설계사 실측 |
| 자료 검색 실패·중복 보관 | 같은 문서를 여러 곳에 저장, 필요 시 찾지 못해 재수집 | 자문역 설계사 실측 |
| 보험 관련 금융기관 제재 비중 (2024년) | 전체 47건 중 **36건(76.6%)** | 금감원 제출자료 (김재섭 의원실, 2025) |
| 설계사 13개월 정착률 (2023년 말) | **47.3%** — 이직 시 데이터 전량 유실 | 금감원·보험업계 종합 (이투데이 보도) |

연 100시간 절약 → 고객 미팅 200건 추가 가능(1건 30분). 계약 전환율 10%만 잡아도 20건 신규 계약. **"편리하다"가 아니라 "돈이 된다"는 것이 설계사에게 AIMS의 가치입니다.**

### 왜 아무도 이 문제를 풀지 못했는가 — 4개의 벽

| 벽 | 핵심 이유 |
|---|---|
| **1. 보험사의 영역이 아니다** | 고객밀착형 데이터(가족상황, 상담이력 등)는 설계사가 신뢰 기반으로 수집한 것. 보험사가 관리할 수도, 해서도 안 되는 영역 |
| **2. 범용 CRM은 안 맞는다** | Salesforce·HubSpot은 보험 도메인(Annual Report, 펀드 수익률, 계약 이전 등)을 모름. 커스터마이징에 IT 전문가 필요, 월 수십만 원 비용 |
| **3. 기술이 현장에 안 닿는다** | AI 혜택은 대기업→중견→스타트업 순. 개인사업자 수준의 설계사에게까지 도달하지 않음 |
| **4. 3년 전까지 기술 자체가 안 됐다** | PDF 파싱 정확도, LLM 비용, 벡터 검색 — 2022년 이전에는 정확도·비용 양면에서 실용 불가 |

### 왜 지금 — AI 기술의 성숙

| 기술 | 3년 전 (2022) | 현재 (2026) |
|---|---|---|
| OCR 정확도 | 85~90% | **99%+** (실무 적용 가능) |
| LLM 비용 (1K 토큰) | $0.06 | **$0.00015** (400배 하락) |
| 벡터 검색 | 대기업 전용 | **오픈소스** (누구나 구축) |

3년 전에는 수억 원이 들었던 AI 문서 처리가, 지금은 **월 수만 원**으로 가능합니다. 2025년 NVIDIA DGX Spark 출시로 **소규모 기업도 자체 LLM을 운영할 수 있는 로컬 AI 시대**가 열렸고, AI 기반 개발 도구의 등장으로 **1인 기업도 구축할 수 있는 시대**가 되었습니다.

### AIMS: 설계사의 두 번째 뇌

| 설계사의 고통 | AIMS의 해결 |
|---|---|
| 매달 수백 건 PDF 수동 다운로드 | **자동 수집 엔진**: 보험사 시스템에서 전 고객 PDF 자동 다운로드 |
| PDF 열어서 숫자 하나하나 확인 | **AI 문서 파싱**: 계약정보·보험료·펀드를 자동 추출 |
| 고객 정보가 5곳에 흩어져 있음 | **통합 고객 관리**: 연락처·문서·계약·메모를 한 화면에 |
| "이 고객 보험료가 뭐였지?" | **AI 어시스턴트**: 자연어 질문 → 3초 내 답변 |
| 가족/법인 단위 조회 불가 | **관계 기반 조회**: 가족·법인 한 화면 통합 |
| 과거 해약 이력 조회 불가 | **시계열 추적**: 연도별 보험 현황 자동 누적·비교 |
| 이직하면 데이터 전부 유실 | **설계사 소유 플랫폼**: 보험사와 무관하게 본인이 데이터 소유 |

## 1-2. 창업아이템 목표시장 현황 분석

### 47만 명 — 전용 디지털 도구가 없는 한국 최대 프리랜서 집단

의사에게는 EMR, 변호사에게는 사건관리 시스템, 세무사에게는 회계 소프트웨어가 있습니다. **47만 명의 보험 설계사에게는 전용 디지털 도구가 없습니다.**

> ※ 47만 명 = 전속 설계사 약 18.7만 + GA 설계사 약 28.8만 (방카슈랑스 제외). 금융감독원 「보험설계사 등록현황」 2024년 말 기준. 방카슈랑스 포함 시 총 약 65만 명.

### TAM-SAM-SOM 분석

```
TAM (전체 시장): 글로벌 IDP(지능형 문서 처리) 시장
  └─ 2025년 $3.0B → 2035년 $54.7B (CAGR 33.4%)
     출처: Research Nester, "Intelligent Document Processing Market" (2025)
     ※ 별도 보고서(Fortune Business Insights)에서는 2032년 $66.7B으로 추정.
        본 계획서는 보수적 수치인 Research Nester 기준 사용.

SAM (유효 시장): 국내 보험 설계사 문서·고객 관리 시장
  └─ 보험 설계사 약 47만 명 × 연 약 59만원(월 49,000원 Pro 기준) = 약 2,760억원

SOM (목표 시장): 초기 3년 내 획득 가능 시장
  └─ MetLife 시작 → GA 디지털 수용층 5,000명 × 월 49,000원
  └─ 연 약 29.4억원
```

### GA(법인대리점) 시장의 구조적 기회

- GA 소속 설계사 **약 29만 명** (2024년 말, 금감원) — 전속 설계사의 약 1.5배
- 여러 보험사 상품을 취급 → **정보 통합 필요성이 가장 큼**
- 보험사 시스템에 종속되지 않아 **독립적 도구 선택이 가능**

### 세대교체와 규제 순풍

- MZ세대 설계사 유입 증가 → SaaS 구독 모델에 익숙
- 마이데이터 보험 분야 확대 (2025~): 계약 정보 API 표준화
- 불완전판매 제재 강화: 완전한 정보 기반 상담의 필요성 증대
- 금융위원회 「디지털 보험 로드맵」(2024): 비대면 채널 확대 정책

### 경쟁 환경 분석

| 구분 | 보험사 내부 시스템 | 범용 CRM | 인슈어테크 | **AIMS** |
|------|----------------|---------|-----------|----------|
| **타겟** | 전속 설계사 | 대기업/중견 | 보험사 본사 | **개인 설계사/GA** |
| **가격** | 무료(전속 한정) | 월 수십만원+ | B2B 계약 | **월 29,000~49,000원** |
| **관점** | 계약 중심 | 범용 | 보험사 효율 | **고객 중심** |
| **문서 AI** | 없음 | 없음 | 부분적 | **OCR→요약→RAG 풀스택** |

# 2. 실현가능성

## 2-1. 창업아이템 현황 (준비 정도)

### "계획"이 아니라 "증거"를 가지고 옵니다

IT 업계에서 펌웨어부터 엔터프라이즈 시스템, 보안까지 전 영역을 개발해 온 **30년 경력의 전문가**가 AI를 활용하여, 프로덕션 운영 가능 수준의 시스템을 이미 구축했습니다.

### 핵심 수치

| 지표 | 수치 |
|------|------|
| 총 소스코드 | **509,427 라인** (약 50만) |
| 마이크로서비스 | **8개** + 데스크톱 앱 1개 |
| 프론트엔드 앱 | **5개** (메인, 어드민, 엑셀, 보험상품, 모바일) |
| REST API 엔드포인트 | **395개** |
| 테스트 파일 | **401개** (141,047 라인) |
| Git 커밋 | **4,982회** (~7.5개월) |

### 기존 방식 대비 생산성 비교

| 항목 | 기존 방식 (AI 없이) | AIMS 실제 |
|---|---|---|
| 필요 인력 | 10~12명 | **1명** |
| 개발 기간 | 12개월+ | **7.5개월** |
| 예상 인건비 | 6~7억 원 | — |

**AIMS 자체가 "AI 생산성 혁신"의 증거입니다.** 설계사에게 AI 생산성을 제공하는 도구를, AI 생산성으로 만들었습니다.

### 개발 완료 현황

| 모듈 | 상태 | 설명 |
|------|------|------|
| **프론트엔드 (aims-uix3)** | 운영 중 | React 19 + TypeScript, 278,479 라인 |
| **백엔드 API** | 운영 중 | Node.js/Express, 319개 REST 엔드포인트 |
| **문서 파이프라인** | 운영 중 | FastAPI, 업로드→OCR→요약→임베딩 자동화 |
| **AI 어시스턴트 (MCP)** | 운영 중 | 34개 도구, 자연어 고객정보 질의 |
| **RAG 검색** | 운영 중 | 하이브리드(벡터+키워드) 검색 |
| **AR/CRS 파서** | 운영 중 | pdfplumber 기반 테이블 자동 추출, 자가복구 |
| **AutoClicker** | 운영 중 | PDF 자동 다운로드 데스크톱 앱 |
| **인증 시스템** | 운영 중 | JWT + 소셜 로그인 (카카오/네이버/구글) |
| **모바일 앱** | 개발 중 | React Native |

### DGX Spark 도입 — 원가 구조 혁신과 AI 기능 해방

#### 왜 로컬 AI 추론이 필요한가

AIMS의 AI 기능은 현재 외부 API(OpenAI, Upstage 등) 종량제에 의존합니다. **비용 부담으로 인해 최저가 모델(gpt-4.1-mini)만 사용하고 있으며, 고급 AI 기능(고객 인사이트 리포트, 자동 보장분석 등)은 비용 제약으로 미구현 상태입니다.**

DGX Spark를 도입하면 비용 제약 없이 AI 기능을 풀로 제공할 수 있고, 고객 금융 데이터가 외부 API로 전송되지 않아 **보안도 강화**됩니다.

#### 비용 절감 효과 (상세 분석: 별첨 2 참조)

AIMS 어드민 시스템(aims_analytics DB)에서 측정된 개발·테스트 환경 데이터를 기반으로 상용화 시 비용을 추정했습니다. 개발 환경(사용자 1명, 고객 784명)의 단가를 기준으로 사용자 수에 따라 선형 확장한 추정치이며, 실제 상용 환경에서는 사용 패턴에 따라 달라질 수 있습니다.

> **핵심 결론**: 개발 환경 실측 기반 추정 시, 사용자 300명 이상에서 약 5.5개월 만에 DGX Spark 장비 투자비(₩1,500만원)를 회수합니다. 500명 기준으로는 약 3.1개월 만에 회수됩니다.

#### DGX Spark의 3가지 전략적 가치

| 가치 | 설명 |
|------|------|
| **1. AI 기능 해방** | 토큰 비용 걱정 없이 고객 인사이트 리포트, 자동 보장분석 등 고급 기능 제공 가능 |
| **2. 데이터 보안** | 고객의 보험·금융 데이터가 외부 API 서버로 전송되지 않음 |
| **3. 원가 예측 가능성** | 변동비(API 종량제) → 고정비(장비) 전환으로 스케일링 시 마진 확대 |

#### 전환 대상 AI 영역

10개 AI 영역 중 **9개를 자체 추론으로 전환**, OCR만 외부 API(Upstage) 유지합니다.

| 전환 대상 (9개) | 유지 (1개) |
|----------------|-----------|
| 문서 요약, 문서 분석·분류, 벡터 임베딩, RAG 검색, MCP AI 어시스턴트, 이미지 분석, 음성 처리, 계약 추출, 인사이트 리포트 | **OCR** (Upstage API — 특화 엔진 정확도 우위) |

> **왜 2대인가**: 1대는 실시간 추론 전용(채팅·RAG 등 사용자 요청 즉시 처리), 1대는 배치 처리 전용(문서 임베딩·대량 요약 등 비동기 작업). 단일 장비로는 실시간 응답과 배치 처리를 동시 수행 시 성능 저하가 발생합니다.

### 기술적 리스크 및 대응 방안

### Go-to-Market: MetLife 1등 지점에서 시작하는 동심원 확장

```

    ↓  ← 문서 파이프라인만 보험사별로 확장

```

#### 1단계: MetLife 1등 지점 (0~6개월)

대표자의 **보험산업 자문역**이 속한 지점은 **대한민국 MetLife 전체 1등 실적의 지점**으로, 스타 설계사들이 소속된 엘리트 지점입니다. 이 지점에서 직접 사용하며 피드백을 받는 **진화 피드백 구조**로 제품을 고도화합니다.

| 채널 | 방법 | 목표 |
|------|------|------|
| 보험산업 자문역 네트워크 | 1등 지점 내 직접 소개 + 파일럿 | 파일럿 사용자 30명 |
| 현장 피드백 루프 | "이건 안 돼" → 다음 날 구현 | 주간 업데이트 |
| 입소문 | 스타 설계사 "내가 쓰고 있는데 좋다" | 자연 확산 |

#### 2단계: MetLife 타 지점 확산 (6~12개월)

"1등 지점의 스타 설계사들이 인정한 서비스"라는 **프리미엄**이 핵심 신뢰 요소. 보험 업계는 입소문/평판 기반이므로, 스타 설계사의 추천이 최고의 마케팅입니다.

#### 3단계: 타 보험사 확대 (12개월~)

보험사별 문서 양식은 다르지만 설계사의 니즈는 동일. 문서 파이프라인만 보험사별로 확장하면 서비스 확장 가능.

### 왜 이 전략이 실행 가능한가 — 구조적 경쟁 우위

AIMS는 **1인 기업의 속성으로 고객과 바로 직접 맞닿아** 있습니다. 보험산업 자문역이 "이 기능이 필요해" → AI를 활용하여 **수일 이내** 구현·배포. 보험사가 50억을 투자해도 "현장 설계사 옆에서 매일 피드백 받으며 개발"은 불가능합니다. **스타트업의 민첩성 + AI의 생산성 = 대기업이 따라올 수 없는 moat.**
