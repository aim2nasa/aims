# AI 분류/제목 생성 신뢰도 지표 도입 검토 보고서

> 작성일: 2026-03-26
> 대상: AIMS 문서 파이프라인 — `summarize_text()`, `generate_title_only()`

---

## 1. 배경 및 목적

AIMS는 업로드된 문서에 대해 AI(gpt-4o-mini)를 사용하여 **22개 유형 분류**와 **제목(별칭) 생성**을 수행한다.
현재 이 결과가 "충분한 근거를 바탕으로 한 판단"인지, "정보 부족으로 환각 위험이 높은 판단"인지 구분하는 체계가 없다.

본 보고서는 **AI 결과의 근거 충분성 지표(confidence)** 도입 가능성을 검토한다.

---

## 2. 현재 상태 (As-Is)

### 2.1 이미 존재하는 confidence 필드

`summarize_text()`는 AI에게 JSON 응답으로 `confidence` 값을 요청하고 있으며, 0.0~1.0으로 정규화하여 MongoDB에 저장한다.

```python
confidence = parsed.get("confidence", 0.0)
confidence = max(0.0, min(1.0, float(confidence)))
```

**문제점**: 이 값은 **AI의 자기 평가**다. 모델이 "나는 90% 확신한다"고 말해도 실제 정확도와 상관관계가 낮을 수 있다 (calibration 문제). 특히 gpt-4o-mini는 자기 확신이 높은 편이어서, 대부분 0.8~0.95 범위에 몰리는 경향이 있다.

### 2.2 품질 검증 — 텍스트 길이 체크만 존재

| 항목 | 현재 구현 | 한계 |
|------|----------|------|
| 최소 길이 | 10자 이상이면 AI 호출 | 10자는 환각 위험 매우 높음 |
| 최대 길이 | 10,000자 절단 | 절단 사실은 기록하나 신뢰도 반영 안 함 |
| 텍스트 품질 | 미검증 | OCR 깨짐, 인코딩 오류 감지 없음 |
| 분류 확신도 | AI 자기 평가만 | logprobs 미사용 |
| 제목 근거성 | 프롬프트 규칙만 | 생성 후 원문 대조 없음 |

### 2.3 `generate_title_only()`는 confidence 자체가 없음

제목만 생성하는 경량 함수는 신뢰도 지표를 일절 반환하지 않는다.

---

## 3. 제안: 3계층 신뢰도 체계

AI 결과의 근거 충분성을 **입력 → 추론 → 출력** 3단계에서 각각 측정한다.

### 3.1 Layer 1: 입력 품질 점수 (Input Quality Score)

> AI에게 전달되는 재료의 품질. AI 호출 전에 측정 가능.

| 지표 | 측정 방법 | 비용 |
|------|----------|------|
| **텍스트 길이** | 단순 `len()`. 10~50자/50~200자/200자+ 단계화 | 무료 |
| **텍스트 깨짐률** | 비정상 문자 비율 (연속 특수문자, 제어문자, 깨진 유니코드) | 무료 |
| **핵심 키워드 존재** | 보험사명, 날짜, 금액 패턴 등 도메인 키워드 매칭 | 무료 |
| **OCR confidence** | Upstage OCR에서 이미 반환하는 값 (현재 저장은 하지만 미활용) | 무료 (이미 있음) |

**산출 예시**:
```
텍스트 200자+, 깨짐 0%, 키워드 3개 → input_quality: "high"
텍스트 30자, 깨짐 5%, 키워드 0개  → input_quality: "low"
```

**효과**: 가장 확실하고 비용 없음. "AI에게 좋은 재료를 줬는가"를 객관적으로 판단.

### 3.2 Layer 2: 추론 확신도 (Classification Confidence)

> AI가 분류할 때 얼마나 확신했는가. logprobs 기반.

**방법**: OpenAI API의 `logprobs` 옵션 활용

```python
response = await client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[...],
    logprobs=True,          # 추가
    top_logprobs=5,         # 상위 5개 토큰 확률
    response_format={"type": "json_object"},
)
```

**핵심 지표**: 분류 유형 토큰의 **1위-2위 확률 격차**

```
"policy" 확률 92%, "insurance_etc" 5%  → 격차 87%p → "high"
"general" 확률 35%, "personal_docs" 28% → 격차 7%p  → "low"
```

| 장점 | 한계 |
|------|------|
| AI 자기 평가보다 객관적 | JSON 모드에서 logprobs 해석이 복잡 (토큰 단위) |
| 추가 비용 없음 (API 옵션) | 분류에만 유효, 제목 생성에는 적용 어려움 |
| "찍은 것"과 "확신한 것"을 구분 가능 | 모델 변경 시 calibration 재검증 필요 |

**제약**: `response_format=json_object`와 `logprobs`를 함께 사용할 때, 분류 유형에 해당하는 토큰의 확률을 정확히 추출하려면 파싱 로직이 필요하다.

### 3.3 Layer 3: 출력 근거성 검증 (Output Grounding Check)

> 생성된 결과가 원문에 근거하는가. AI 호출 후 사후 검증.

#### 제목(별칭) 검증

생성된 제목의 주요 단어가 원문 텍스트에 실제로 존재하는지 확인:

```
원문: "메트라이프생명 종신보험 보험증권 ... 홍길동 ..."
생성 제목: "홍길동 메트라이프 종신보험 증권"
→ 모든 단어 원문에 존재 → grounding: 100%

원문: "가나다라마바사 ... (깨진 텍스트)"
생성 제목: "삼성화재 건강보험 계약서"
→ 해당 단어 원문에 없음 → grounding: 0% → 환각
```

**구현**: 형태소 분석 없이도 단순 문자열 포함 체크로 충분

#### 분류 검증

분류 유형에 대응하는 키워드가 원문에 존재하는지:

```
분류: "policy" → "보험증권", "증권번호", "보험료" 등 키워드 확인
→ 1개도 없으면 → 환각 가능성
```

| 장점 | 한계 |
|------|------|
| 환각을 직접 탐지 | 동의어/유사 표현은 놓칠 수 있음 |
| AI 불필요, 비용 0 | 제목이 추상적일 때 (예: "기타 서류") 의미 없음 |
| 구현 단순 | 완벽한 검증은 아님 (필요충분 아닌 필요조건) |

## 4. 통합 신뢰도 등급

3개 Layer를 조합하여 최종 등급을 산출한다:

```
┌──────────────┬──────────────┬──────────────┐
│  Layer 1     │  Layer 2     │  Layer 3     │
│  입력 품질   │  추론 확신도 │  출력 근거성 │
│  (AI 호출 전)│  (logprobs)  │  (AI 호출 후)│
└──────┬───────┴──────┬───────┴──────┬───────┘
       │              │              │
       ▼              ▼              ▼
    ┌─────────────────────────────────────┐
    │         통합 신뢰도 등급            │
    │    HIGH / MEDIUM / LOW / SUSPECT    │
    └─────────────────────────────────────┘
```

| 등급 | 조건 (예시) | 의미 |
|------|------------|------|
| **HIGH** | 입력 200자+, 분류 격차 50%p+, 제목 grounding 80%+ | 충분한 근거 기반 판단 |
| **MEDIUM** | 입력 50~200자, 분류 격차 20~50%p | 합리적 판단이나 일부 불확실 |
| **LOW** | 입력 50자 미만 또는 깨짐 5%+ | 정보 부족, 결과 검토 권장 |
| **SUSPECT** | 제목 grounding 30% 미만 또는 분류 키워드 0개 | 환각 의심, 수동 확인 필요 |

---

## 5. 활용 방안

### 5.1 파이프라인 내부 (자동화)

| 등급 | 동작 |
|------|------|
| HIGH | 현재와 동일 (자동 저장) |
| MEDIUM | 자동 저장하되, `needs_review: true` 플래그 |
| LOW | 분류를 `general`로 폴백, 제목은 파일명 유지 |
| SUSPECT | AI 결과 폐기, 수동 분류 대기 상태로 전환 |

### 5.2 프론트엔드 표시

- 문서 목록에서 신뢰도 등급 아이콘 표시 (예: 초록/노랑/빨강 점)
- LOW/SUSPECT 문서를 별도 필터로 모아볼 수 있는 뷰
- 사용자가 분류/제목을 수정하면 `manually_verified: true`로 전환

### 5.3 품질 모니터링

- 신뢰도 등급별 분포를 대시보드로 추적
- LOW/SUSPECT 비율이 높아지면 → OCR 품질 문제 또는 새로운 문서 유형 출현 감지

## 6. 구현 난이도 및 우선순위

| Layer | 난이도 | 소요 | API 비용 변화 | 권장 순서 |
|-------|--------|------|-------------|----------|
| **Layer 1: 입력 품질** | 낮음 | 0.5일 | 없음 | **1순위** — 즉시 가능, 효과 확실 |
| **Layer 3: 출력 근거성** | 낮음 | 0.5일 | 없음 | **2순위** — 환각 직접 탐지 |
| **Layer 2: logprobs** | 중간 | 1~2일 | 없음 | **3순위** — JSON 모드 파싱 복잡 |
| **통합 등급 + DB 저장** | 낮음 | 0.5일 | 없음 | Layer 구현 후 |
| **프론트엔드 표시** | 중간 | 1일 | 없음 | 최종 |

## 7. 리스크 및 주의점

### 7.1 과신 리스크
신뢰도 "HIGH"라고 해서 반드시 정확한 것은 아니다. 사용자에게 "이 수치는 참고 지표"임을 명확히 전달해야 한다.

### 7.2 logprobs의 JSON 모드 제약
`response_format=json_object` 사용 시 logprobs가 JSON 구조 토큰(중괄호, 콜론 등)을 포함한다. 분류 유형에 해당하는 토큰만 정확히 추출하는 파싱이 필요하며, 유형명이 여러 토큰으로 분할될 수 있다 (예: `coverage` + `_` + `analysis`).

### 7.3 모델 변경 시 재보정
gpt-4o-mini에서 다른 모델로 변경하면 확률 분포가 달라진다. 등급 기준값(임계치)을 모델별로 관리해야 한다.

### 7.4 성능 영향
Layer 1, 3은 성능 영향 무시 가능. logprobs 응답은 데이터량이 증가하나 처리 시간 차이는 미미하다.

## 8. 결론

| 질문 | 답변 |
|------|------|
| AI 결과에 신뢰도를 제공할 수 있는가? | **가능하다** |
| OCR confidence처럼 정밀한가? | **아니다.** 참고 지표 수준 |
| 환각 탐지에 유효한가? | **Layer 1(입력 품질) + Layer 3(출력 근거성)은 유효하다** |
| 도입 비용 대비 효과는? | **높다.** Layer 1+3은 API 비용 증가 없이 0.5~1일 구현 |

**권장**: Layer 1(입력 품질) + Layer 3(출력 근거성)을 우선 도입하고, 실측 데이터를 축적한 후 Layer 2(logprobs) 도입 여부를 결정한다.
