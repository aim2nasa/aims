# AI 근거성 검증 (Evidence Verification) 기획서

> AIMS 문서 파이프라인에서 AI가 생성한 분류(document_type)와 별칭(displayName)이
> 충분한 근거를 바탕으로 도출되었는지를 객관적으로 판별하는 체계

---

## 1. 개요

### 1.1 문제 정의

AIMS는 업로드된 문서에 대해 AI(gpt-4o-mini)로 **22개 유형 분류**와 **별칭 생성**을 수행한다.
그러나 현재 이 결과가 "충분한 근거에 기반한 판단"인지, "정보 부족으로 인한 추측"인지 구분하는 체계가 없다.

현재 유일한 지표인 `meta.confidence`는 AI의 자기 평가(self-reported)이며, **58.2%가 0을 반환**하여 실질적으로 미작동 상태다.

### 1.2 목표

**AI 결과에 대해 "이 결과를 그냥 써도 되는가, 한번 봐야 하는가"의 판단 근거를 제공한다.**

100% 정확한 검증이 아니다. 일기예보처럼 — 완벽하지는 않지만, 우산을 챙길지 말지의 판단 근거로서 실용적 가치를 제공하는 것이 목표다.

### 1.3 명칭

| 구분 | 명칭 |
|------|------|
| 기능명 (한글) | AI 근거성 검증 |
| 기능명 (영문) | Evidence Verification |
| 내부 약칭 | EV |
| DB 필드 접두어 | `evidence.*` |
| 모듈명 | `evidence_scorer` |

### 1.4 "신뢰도"가 아닌 "근거성"인 이유

| 용어 | 의미 | 문제 |
|------|------|------|
| 신뢰도 (Confidence) | AI가 스스로 얼마나 확신하는가 | AI의 주관적 자기 평가. 틀려도 높은 값을 반환 |
| 근거성 (Evidence) | 결과가 충분한 증거 위에 서 있는가 | 입력 재료와 출력의 원문 대조라는 외부 기준으로 측정 |

## 2. 현재 상태 (As-Is)

### 2.1 기존 체계의 한계

| 항목 | 현재 구현 | 한계 |
|------|----------|------|
| `meta.confidence` | AI 자기 평가 (0.0~1.0) | 58.2%가 0 반환. 실질 미작동 |
| 텍스트 길이 체크 | 10자 이상이면 AI 호출 | 10자는 환각 위험 매우 높음 |
| 텍스트 품질 | 미검증 | OCR 깨짐, 인코딩 오류 감지 없음 |
| 출력 검증 | 없음 | 생성된 별칭이 원문에 근거하는지 확인하지 않음 |

### 2.2 환각의 근본 원인: 프롬프트

현재 프롬프트가 파일명 참조를 **장려**하고 있다:

- `summarize_text()` — "텍스트가 부실해도 파일명에서 유형을 추론 가능하면 반드시 해당 유형으로 분류!"
- `generate_title_only()` — "파일명에 유용한 정보가 있다면 반영하세요"

이것이 환각의 주요 경로다. EV 도입과 별개로, **프롬프트 개선이 선행되어야** 한다. 단, 오버피팅(특정 케이스를 잡으려다 정상 케이스가 깨지는 것)에 주의해야 한다.

### 2.3 실측 근거

> 상세 시뮬레이션 데이터: [2026-03-26_AI_CONFIDENCE_REVIEW_REPORT.md](2026-03-26_AI_CONFIDENCE_REVIEW_REPORT.md)

캐치업코리아(곽승철) 문서 1,169건 시뮬레이션 결과:

| 구분 | 수치 |
|------|------|
| 전체 SUSPECT (환각 의심) | 196건 (16.8%) |
| 별칭 환각 | 47건 (4.1%) |
| Layer 1만으로 SUSPECT 탐지 | 93% (182/196건) |
| Layer 1+3 조합 시 환각 별칭 탐지 | ~80% |
| 기존 AI confidence 미반환 | 58.2% |

## 3. EV 체계 설계

### 3.1 3계층 구조

```
┌──────────────────┬──────────────────┬──────────────────┐
│  Layer 1         │  Layer 2         │  Layer 3         │
│  입력 품질       │  추론 확신도     │  출력 근거성     │
│  (AI 호출 전)    │  (logprobs)      │  (AI 호출 후)    │
│  비용: 무료      │  비용: 무료      │  비용: 무료      │
│  난이도: 낮음    │  난이도: 중간    │  난이도: 낮음    │
└────────┬─────────┴────────┬─────────┴────────┬─────────┘
         │                  │                  │
         ▼                  ▼                  ▼
      ┌──────────────────────────────────────────┐
      │          Evidence Grade (EV 등급)        │
      │       HIGH / MEDIUM / LOW / SUSPECT      │
      └──────────────────────────────────────────┘
```

### 3.2 Layer 1: 입력 품질 (Input Quality)

> AI에게 전달되는 재료의 품질을 AI 호출 전에 측정한다.

| 지표 | 측정 방법 | 비용 |
|------|----------|------|
| 텍스트 길이 | `len()`. 10~50자 / 50~200자 / 200자+ 단계화 | 무료 |
| 텍스트 깨짐률 | 비정상 문자 비율 (아래 정의 참조) | 무료 |
| 핵심 키워드 존재 | 보험사명, 날짜 패턴, 금액 패턴 등 도메인 키워드 매칭 | 무료 |
| OCR confidence | Upstage OCR 반환값 (현재 저장만 하고 미활용) | 무료 (이미 존재) |

**깨짐률 정의 (구체화 필요)**:
- 유니코드 대체문자(U+FFFD) 비율
- 연속 제어문자(U+0000~U+001F, U+007F~U+009F) 비율
- 주의: `㈜`, `㎡`, `②`, 한자(`漢字`) 등은 정상 문자로 취급해야 함

### 3.3 Layer 2: 추론 확신도 (Classification Confidence)

> AI가 분류할 때 얼마나 확신했는가를 logprobs로 측정한다.

- OpenAI API의 `logprobs=True, top_logprobs=5` 옵션 활용
- 핵심 지표: 분류 유형 토큰의 1위-2위 확률 격차
- JSON 모드에서 의미 있는 토큰을 특정하는 파싱이 핵심 난점
- **Phase 1에서는 도입하지 않음.** Layer 1+3의 실측 데이터를 축적한 후 판단

### 3.4 Layer 3: 출력 근거성 (Output Grounding)

> 생성된 결과가 원문에 근거하는가를 사후 검증한다.

#### 별칭(displayName) 검증

생성된 별칭을 토큰화하고, 각 토큰이 원문 텍스트에 존재하는지 확인:

```
원문: "메트라이프생명 종신보험 보험증권 ... 홍길동 ..."
별칭: "홍길동 메트라이프 종신보험 증권"
→ 모든 단어 원문에 존재 → grounding: 100%

원문: "가나다라마바사 ... (깨진 텍스트)"
별칭: "삼성화재 건강보험 계약서"
→ 해당 단어 원문에 없음 → grounding: 0% → 환각
```

**한국어 매칭 보완 규칙**:
- 별칭을 공백/특수문자로 토큰화 후, 각 토큰이 원문에 부분 문자열로 존재하는지 확인
- 2자 이하 토큰(조사, 숫자 등)은 검증 skip
- 날짜 정규화: `YYYY년 MM월` / `YYYY-MM-DD` / `YYYY.MM.DD` 등을 통일 포맷으로 변환 후 비교
- 고객명(`customer_name`)은 DB에서 주입하므로 grounding check 예외 처리

#### 분류(document_type) 검증

분류 유형별 기대 키워드가 원문에 존재하는지 확인:

```
분류: "policy" → "보험증권", "증권번호", "보험료" 중 1개 이상 확인
→ 0개면 → 환각 가능성
```

### 3.5 통합 등급

| 등급 | 조건 (예시) | 의미 |
|------|------------|------|
| **HIGH** | 입력 200자+, grounding 70%+, 분류 키워드 1개+ | 충분한 근거 기반 |
| **MEDIUM** | 입력 50~200자, 또는 grounding 50~70% | 합리적이나 일부 불확실 |
| **LOW** | 입력 50자 미만 또는 깨짐 5%+ | 정보 부족 |
| **SUSPECT** | grounding 30% 미만 또는 분류 키워드 0개 또는 텍스트 없음 | 환각 의심 |

## 4. EV가 잡을 수 있는 것과 없는 것

이 체계의 한계를 명확히 인식해야 한다.

| 탐지 가능 | 탐지 불가 |
|-----------|-----------|
| 텍스트 없이 파일명으로 추정한 별칭 | 원문 단어로 구성됐지만 의미가 틀린 별칭 |
| AI가 원문에 없는 단어를 생성한 환각 | 입력 충분 + AI 규칙 혼동 (분류 오류) |
| 파일명을 그대로 복사한 저품질 별칭 | "맞지만 너무 모호한" 별칭 (예: "보험 관련 자료") |
| 텍스트 부족 상태에서의 AI 추정 결과 | 동의어/유사 표현으로 구성된 별칭 |

**핵심**: EV는 "근거 없이 만들어낸 것을 걸러낸다." "AI가 틀린 모든 경우를 잡는다"는 목표가 아니다.

## 5. 프론트엔드 표시

### 5.1 설계 원칙

에이전트 리뷰(Alex, Gini, Ari, Sora, Dana)를 통해 도출된 핵심 원칙:

| 원칙 | 근거 |
|------|------|
| **4단계 → 2단계 매핑** | 설계사에게 HIGH/MEDIUM/LOW/SUSPECT 구분은 과다. "괜찮음/확인 필요" 둘이면 충분 (Sora, Dana) |
| **예외 기반 표시** | 정상(~79%)은 아무 표시 없음. 문제 있는 것에만 작은 아이콘 (Sora: "잘 된 건 신경 안 써요") |
| **"신뢰도" 용어 미사용** | 사용자에게 점수나 등급을 직접 노출하지 않음. "AI 자동 분류" / "확인 필요"로 표현 (Dana) |
| **색상 신호등 금지** | 빨강은 보험 맥락에서 위험 연상. WCAG 접근성 위반. 차분한 중립 톤 사용 (Sora, Dana) |

### 5.2 백엔드 ↔ 프론트엔드 매핑

```
백엔드 DB (4단계, 분석/모니터링용)     프론트엔드 (2단계, 사용자용)
─────────────────────────────────     ──────────────────────────
HIGH                              →   표시 없음 (기본 상태)
MEDIUM                            →   "확인 필요" 아이콘
LOW                               →   "확인 필요" 아이콘
SUSPECT                           →   "확인 필요" 아이콘
manually_verified: true           →   표시 없음
```

### 5.3 표시 방식

- **기본 뷰 (문서 목록)**: "확인 필요" 문서에만 작은 인라인 아이콘 (12px, 중립 톤)
- **호버/탭**: 툴팁 — "AI가 자동 분류한 문서입니다. 확인하거나 수정할 수 있습니다."
- **필터**: 기존 전체문서보기에 "확인 필요 문서만" 토글 추가 (별도 뷰 아님)
- **수정 UX**: 기존 RenameModal 확장 + "이대로 확인" 원탭 옵션 + 일괄 확인 모드

### 5.4 사용자 수정 시 동작

사용자가 별칭/분류를 수정하거나 "이대로 확인"하면:
- `manually_verified: true`로 전환
- "확인 필요" 아이콘 사라짐
- 수정 기록 축적 → 피드백 루프 (6.3 참조)

## 6. 파이프라인 동작

### 6.1 등급별 파이프라인 동작

| 등급 | 분류 | 별칭 | 플래그 |
|------|------|------|--------|
| HIGH | AI 결과 자동 저장 | AI 결과 자동 저장 | — |
| MEDIUM | AI 결과 저장 | AI 결과 저장 | `needs_review: true` |
| LOW | `general`로 폴백 | 원본 파일명 유지 | `needs_review: true` |
| SUSPECT | AI 결과 저장 + `needs_review` | AI 결과 저장 (있는 경우) 또는 파일명 유지 | `needs_review: true` |

**SUSPECT도 AI 결과를 폐기하지 않는다.** 빈칸보다 틀린 답이라도 있는 것이 사용자가 수정하기 편하다 (Sora). 단, 텍스트가 전혀 없어서 AI 결과 자체가 없는 경우는 파일명을 유지한다.

### 6.2 DB 스키마

기존 `meta.confidence`는 하위 호환을 위해 유지하고, `evidence` 필드를 추가한다:

```javascript
// files 컬렉션에 추가
"evidence": {
    "input_score": 0.85,         // Layer 1: 입력 품질 (0~1)
    "grounding_score": 0.90,     // Layer 3: 출력 근거성 (0~1)
    "classification_gap": null,  // Layer 2: logprobs 격차 (Phase 2에서 추가)
    "grade": "HIGH",             // 통합 등급
    "needs_review": false,       // 프론트엔드용 플래그
    "manually_verified": false,  // 사용자 확인 여부
    "details": {                 // 디버깅/분석용 상세
        "text_length": 2450,
        "garble_ratio": 0.02,
        "keyword_found": ["보험증권", "보험료"],
        "ungrounded_words": []
    }
}
```

- `evidence.grade`에 인덱스 추가 (needs_review 문서 조회용)
- 기존 문서에 `evidence` 필드가 없으면 "레거시"로 취급 (프론트엔드에서 아무 표시 안 함)

### 6.3 피드백 루프

사용자가 별칭/분류를 수동 수정한 기록을 축적하여 EV 체계 자체의 정확도를 검증한다:

- EV가 HIGH로 판정한 문서를 사용자가 수정 → False Negative (놓친 것)
- EV가 SUSPECT로 판정한 문서를 사용자가 그대로 확인 → False Positive (과잉 탐지)
- 이 데이터를 분석하여 임계값 튜닝, 프롬프트 개선에 활용
- **피드백 루프가 없으면 EV가 잘 작동한다는 착각에 빠질 수 있다** (Gini)

| 파일 | 변경 내용 |

| Layer | 추가 시간 | 근거 |
|-------|-----------|------|

| 합계 | < 10ms | 현재 병목(OCR 수초, AI 1~3초) 대비 무시 가능 |

## 8. 구현 로드맵

### Phase 1: Layer 1 + Layer 3 (백엔드)

- `evidence_scorer.py` 모듈 생성
- 입력 품질 측정 (텍스트 길이, 깨짐률, 키워드)
- 출력 근거성 검증 (별칭 토큰 매칭, 분류 키워드 매칭)
- 통합 등급 산출 + DB 저장
- 프롬프트 환각 경로 개선 (파일명 참조 정책 조정, 오버피팅 주의)

### Phase 2: 프론트엔드 표시

- 문서 목록에 "확인 필요" 인라인 아이콘
- "확인 필요 문서만" 필터 토글
- 수동 확인/수정 UI (RenameModal 확장 + 일괄 확인)
- `manually_verified` 플래그 연동

### Phase 3: Layer 2 (logprobs) — 선택적

- Phase 1+2의 실측 데이터 축적 후 도입 여부 결정
- `summarize_text()`에 `logprobs=True` 추가
- JSON 모드에서 분류 토큰 파싱 로직 구현
- 통합 등급에 Layer 2 반영

### Phase 4: 피드백 루프 + 모니터링

- 사용자 수정 기록 축적
- EV 등급별 분포 추적
- False Negative/Positive 분석
- 임계값 자동 튜닝

## 9. 리스크 및 주의사항

### 9.1 과신 금지

HIGH 등급은 "충분한 근거가 있다"이지, "정확하다"가 아니다. 입력이 좋아도 AI가 틀릴 수 있으며, 이 경우 EV로 탐지할 수 없다. 이것은 AI 기술의 본질적 한계다.

### 9.2 프롬프트 개선과의 병행

EV는 사후 탐지(detection)이고, 프롬프트 개선은 예방(prevention)이다. 둘 다 필요하며, 예방만으로 100% 환각 제거는 불가능하므로 탐지가 반드시 병행되어야 한다. 프롬프트 개선 시 오버피팅에 주의한다.

### 9.3 깨짐률 정의 명확화

구현 시 "깨진 문자"의 정의를 코드에 명확히 명시해야 한다. 한자, 특수기호(`㈜`, `㎡`, `②`), 법률용어가 정상 문자인데 깨짐으로 오탐되면 안 된다.

### 9.4 보안

- Layer 3 grounding 검증 로그에 고객명이 평문으로 남을 수 있음 → 로깅 시 마스킹 적용
- "확인 필요" 문서 조회 API에 소유자 격리(`ownerId`) 필수 적용

### 9.5 모델 변경 시

gpt-4o-mini에서 다른 모델로 변경하면 EV 임계값 재검증이 필요하다. 특히 Layer 2(logprobs) 도입 후에는 모델별 확률 분포 차이에 주의한다.
