## 4. 오류 유형 1: AC 콘솔 테이블 혼입 (9명) — 파서 버그

### 현상

주소 필드에 `No 고객명 구분 휴대폰 상태` 헤더와 고객 목록 테이블 데이터가 삽입됨.

### 핵심 발견: Upstage OCR의 비결정적 HTML 구조

`customer_results_ㄱ.json`의 `customer_detail.home_address`에 **이미** 테이블 데이터가 포함되어 있음.
→ `generate_reports.py`의 변환 과정이 아닌 **OCR 파싱 단계** (`read_customer.py`)에서 발생.

### 근본 원인 — OCR + 파서 복합 원인 (재현 검증 완료)

**재현 방법**: `D:/ac/ㄱ/dev/detail_김성희.png`에 Upstage OCR + `parse_customer_info()` 실행 → 오류 재현됨.
**대조군**: `D:/ac/ㄱ/dev/detail_곽혜원.png` → 정상 파싱됨.

#### 원인 1: Upstage OCR의 비결정적 `<figure>` 생성

모든 detail 스크린샷에는 우측에 **AC 콘솔 창**(고객 목록 테이블)이 보입니다.
Upstage OCR는 이 AC 콘솔 영역을 **두 가지 방식**으로 처리합니다:

**정상 (곽혜원)** — AC 콘솔을 `<figure>` 태그로 캡슐화:
```html
<td colspan="7">
  ...자택주소 13957 Q 경기 안양시 만안구 예술공원로51번길
  자택전화 031 472 1194 11, A동 401호(안양2동,삼영2차아파트)
  <figure><img alt="...(AC 콘솔 이미지 설명)..." /></figure>
  직장주소 Q 직장전화 선택 ▼ ...
</td>
```
→ `extract_text_from_html()`이 `<figure>` 내부를 **정상 스킵** → 주소 깨끗함.

**오류 (김성희)** — AC 콘솔 텍스트를 `<td>` 안에 직접 노출:
```html
<td colspan="3">
  ...자택주소 Q 자택전화 선택
  No 고객명 구분 휴대폰 상태 1 김성희 계약 [처리중] 2 김세영 계약 010-4433-0428
  3 김세현 계약 010-8468-0997 4 김세희 계약 010-4285-3242
  직장주소 직장전화 선택 ▼
  5 김소영 계약 010-8775-0997 6 김소희 계약 010-4321-3139 ...
</td>
```
→ `<figure>` 없음 → AC 콘솔 텍스트가 **그대로 추출** → `find_between("자택주소", ["직장주소"])` 구간에 포함 → **주소 오염**

#### 원인 2: 파서의 방어 부재

`clean_address_section()` (L224)은 폼 라벨(`자택전화`, `직장전화`, `팩스번호`)만 제거하며, **AC 콘솔 테이블 패턴(`No 고객명 구분 휴대폰 상태`)은 감지하지 않음**.

```
추출 텍스트 (정상 — figure 스킵됨):
  자택주소 13957 Q 경기 안양시 ... 직장주소 ...
                  ✓ 깨끗한 주소

추출 텍스트 (오류 — figure 없이 직접 노출):
  자택주소 Q 자택전화 선택 No 고객명 구분 휴대폰 상태 1 김성희 계약 [처리중] ... 직장주소 ...
                          ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                          AC 콘솔 테이블 데이터 (파서가 제거하지 못함)
```

## 8. 수정 방안

### 방안 A (권장): `clean_address_section()`에 AC 콘솔 테이블 패턴 감지 추가

```python
def clean_address_section(raw: str) -> str:
    # ... 기존 로직 ...

    # AC 콘솔 테이블 혼입 감지 — "No 고객명 구분 휴대폰 상태" 패턴
    # 이 패턴 이전의 텍스트만 주소로 사용
    table_match = re.search(r'No\s+고객명\s+구분\s+휴대폰\s+상태', s)
    if table_match:
        s = s[:table_match.start()]

    # ... 기존 로직 계속 ...
```

장점: 간단하고 즉시 적용 가능, 기존 구조 변경 최소화.
단점: 테이블 헤더 뒤에 이어지는 주소 나머지 부분 복원 불가 (해당 주소는 null 처리).

### 방안 B: OCR 캡처 영역 크롭 (근본 해결)

`MetlifeCustomerList.py`에서 detail 스크린샷 캡처 시, AC 콘솔 영역을 제외한 좌측 영역만 크롭:

```python
# 현재: capture(SCREEN)  → 전체 화면 (1920x1080)
# 수정: capture(Region(0, 0, 1370, 1080))  → AC 콘솔 제외
```

장점: OCR 입력에서 AC 콘솔이 완전히 제거됨 → `<figure>` 비결정성 문제 자체 소멸.
단점: 좌표 의존 (모니터 해상도 변경 시 조정 필요).

### 방안 C: 파싱된 주소 후처리 검증

파싱 완료 후, 결과 주소에 오염 패턴이 포함되면 null로 처리:

```python
def validate_address(addr: str) -> str:
    if not addr:
        return None
    # AC 콘솔 테이블 오염 감지
    contamination_patterns = ['No 고객명', '[처리중]', '[완료]', '계약 010-']
    if any(p in addr for p in contamination_patterns):
        return None  # 오염된 주소 → null 처리
    return addr
```

장점: 방어적. 단점: 오염된 주소에서 실제 주소를 복원하지 못함.

### 권장 조합: A + B

- **즉시 적용**: 방안 A (파서 방어) — 오염 패턴 감지 후 해당 부분 제거
- **근본 해결**: 방안 B (캡처 크롭) — AC 콘솔 영역을 OCR 입력에서 제외
