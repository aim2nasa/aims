## 1. 배경

AIMS의 외부 AI API 의존성(OpenAI, Upstage)을 로컬 추론으로 전환하기 위해 하드웨어 도입을 검토한다.

### 1.1 도입 동기

| 동기 | 상세 |
|------|------|
| 데이터 보안 | 고객 보험 문서가 외부 API 서버로 전송되는 문제 해소 |
| 외부 종속성 제거 | OpenAI/Upstage 장애 시 AIMS 전체 AI 기능 마비 방지 |
| 비용 구조 개선 | 월 과금 → 일회성 투자 전환 (현재 월 $10 수준이나 확장 시 증가) |
| 확장 대비 | 설계사 100명, 동시 사용자 10~20명 규모 대응 |

### 1.2 ROI 분석 (교차검증에서 추가)

현재 월 $10(연 ₩17만) 수준이므로, 비용 절감만으로는 투자 정당화가 어렵다.

| 시나리오 | 월 API 비용 | ₩5M 투자 손익분기 | ₩10M 투자 손익분기 |
|---------|-----------|-----------------|------------------|
| 현재 (소수 사용자) | ₩14,000 | 29년 | 59년 |
| 100명 보수적 | ₩70,000 | 6년 | 12년 |
| 100명 적극적 | ₩420,000 | 12개월 | **24개월** |

**투자 정당화 근거는 비용 절감이 아닌 데이터 보안 + 외부 종속성 제거**에 있다. 금융당국의 AI 데이터 역외 처리 규제가 강화되면 필수 투자가 된다.

## 4. 워크로드별 비교 분석

### 4.1 LLM 추론 — 실측 벤치마크 기반 (v4 전면 수정)

> **v3까지의 오류**: "TPS = 대역폭 ÷ 모델 크기"로 단순 계산했으나, NVFP4/MLX 등
> 하드웨어 가속 양자화에서는 이 공식이 적용되지 않는다. v4에서 실측 데이터로 전면 교체.

#### NVFP4란?

NVIDIA의 Blackwell Tensor Core 전용 4비트 양자화 포맷. 일반 Q4와 달리 **GPU 하드웨어가
디퀀타이즈+연산을 동시 처리**하여 메모리 대역폭 병목을 우회한다.
- FP8 대비 40% 메모리 절감, 정확도 <1% 손실
- Q4/AWQ 대비 **20~100% 빠름** (NVIDIA 공식)

#### MLX란?

Apple의 Metal 네이티브 ML 프레임워크. llama.cpp Metal 백엔드 대비 Apple Silicon에서
**2~3배 빠른 추론 성능**을 보인다 (Qwen3 계열 벤치마크 기준).

#### 32B급 모델 실측 TPS (토큰 생성)

| 하드웨어 | 프레임워크 | 모델 | **실측 TPS** | 출처 |
|---------|-----------|------|------------|------|
| Mac Mini M4 Pro 64GB | llama.cpp Q4 | Qwen 2.5 32B | **11-12** | 커뮤니티 벤치 |
| Mac Mini M4 Pro 64GB | llama.cpp (최신) | Qwen3 Next Q4_K_M | **~24** | GitHub #19366 |
| Mac Mini M4 Pro 64GB | **MLX 4bit** | Qwen3 Next | **~60** | GitHub #19366 |
| **DGX Spark (단일)** | **vLLM NVFP4** | Qwen3 14B | **22.71** | NVIDIA 공식 |
| DGX Spark (단일) | vLLM NVFP4 | Llama 3.1 8B | **38.65** | NVIDIA 공식 |
| DGX Spark (단일) | vLLM NVFP4 | Nemotron-30B-A3B | **56.11** | NVIDIA 공식 |
| DGX Spark (단일) | FlashInfer NVFP4 | Qwen3-Next-80B-A3B | **20→35** | NVIDIA 포럼 |
| **DGX Spark 2대 NVLink** | **vLLM MXFP4** | Qwen3.5-35B-A3B | **70.68** | NVIDIA 포럼 |
| DGX Spark 2대 NVLink | vLLM MXFP4 | gpt-oss-120B | **80** | NVIDIA 포럼 |
| DGX Spark 2대 NVLink | NVFP4+SpecDec | Qwen-235B | **11.73** (FP8 대비 2.6배) | NVIDIA 공식 |

#### 핵심 비교 (동일 조건 근사)

32B급 모델, 단일 사용자 기준:

| 하드웨어 | 최적 프레임워크 | 실측 TPS | 가격 |
|---------|--------------|---------|------|
| Mac Mini M4 Pro 64GB | MLX 4bit | **~60** | ₩3M |
| Mac Mini M4 Pro 64GB | llama.cpp Q4 | ~12-24 | ₩3M |
| **DGX Spark 단일** | **vLLM NVFP4** | **22-56** | **₩5M** |
| DGX Spark 2대 NVLink | vLLM MXFP4 | **70-80** | ₩10M |
| Mac Studio M3 Ultra | MLX 4bit | ~90+ (추정) | ₩15M+ |

> **중요 발견**: Mac에서 MLX를 사용하면 llama.cpp 대비 2.5배 빠르다.
> DGX Spark에서 NVFP4를 사용하면 일반 Q4 대비 2~3배 빠르다.
> **양쪽 모두 최적 프레임워크를 사용해야 공정한 비교가 된다.**

#### Prefill(프롬프트 처리) vs Decode(토큰 생성) 차이

| 단계 | 병목 | Spark 강점 | Mac 강점 |
|------|------|-----------|---------|
| **Prefill** (프롬프트 입력 처리) | **연산력** | **★★★★★** (1 PFLOP FP4) | ★★★ |
| **Decode** (토큰 생성) | **대역폭** | ★★★ (273GB/s + NVFP4 가속) | **★★★★** (273GB/s + MLX) |

Spark는 긴 프롬프트 처리(MCP 도구 40개 컨텍스트 등)에서, Mac은 긴 답변 생성에서 각각 유리하다.

### 4.2 Diffusion 모델 — NVFP4 실측 (v4 수정)

| 작업 | DGX Spark (NVFP4 실측) | Mac (Metal 추정) | 차이 |
|------|----------------------|-----------------|------|
| **FLUX.1 12B** (1K) | **2.6초/장** (분당 23장) | ~30-60초 | **10~20배** |
| SDXL 1.0 (1K) | **8.6초/장** (분당 7장) | ~30초 | 3~4배 |

> v3에서 "1.5~2.5배"로 수정했으나, NVFP4 실측 결과 **FLUX.1에서 2.6초/장**이 확인되었다.
> Mac에서 FLUX.1은 30~60초 수준이므로 실제 차이는 **10~20배**에 달한다.
> Diffusion 워크로드에서 Spark의 NVFP4 가속은 압도적이다.

### 4.3 OCR VLM

VLM OCR은 이미지 인코딩(연산력 병목) + 텍스트 생성(대역폭 병목)이 혼합된 워크로드.
**실측 없이 어느 쪽이 유리한지 단정 불가.** PoC에서 양쪽 비교 필수.

### 4.4 파인튜닝

AIMS 규모(설계사 100명)에서 비현실적. 학습 데이터 수천~수만 건 라벨링, 대기업 AI팀 수준의 작업. **도입 근거에서 제외.**

#### 동시 사용자 체감 속도 비교 (핵심)

클러스터의 진짜 강점은 "각 노드가 독립적으로 요청을 처리"하는 것이다.
고가 단일 장비는 대역폭을 나눠 쓰지만, 클러스터는 각 노드가 풀 스피드 유지.

Qwen3-32B 기준, 각 사용자가 체감하는 TPS (Mac: MLX 4bit, Spark: NVFP4):

| 동시 사용자 | Mini 1대 (MLX) | Mini 2대 | Mini 3대 | Mini 4대 | Spark 1대 (NVFP4) | Spark 2대 NVLink |
|-----------|--------------|---------|---------|---------|------------------|-----------------|
| 1명 | **60** | 60 | 60 | 60 | 22-56 | 70-80 |
| 3명 | 20 | **60** | **60** | **60** | 15-30 | 25-30 |
| 5명 | 12 | 30 | **60** | **60** | 10-20 | 15-20 |
| 10명 | 6 | 15 | 20 | **30** | NIM 배치 | NIM 배치 |
| 20명 | 3 | 7.5 | 10 | **15** | NIM 배치 | NIM 배치 |

> **MLX 기준으로 Mini 1대의 단일 TPS가 ~60으로, 동시 3명까지는 Mini 1대로도 충분하다.**
> 동시 5명 이상에서 Mini 클러스터의 노드 분산이 효과를 발휘한다.
> Spark NIM의 continuous batching은 동시 10명+에서 효율적이나, 정확한 per-user TPS는 모델/배치 설정에 따라 다르다.
> 이는 각 Mini가 독립 노드로 요청을 분담하기 때문이다.

#### 클러스터 토폴로지

**2대 (TB5 직결):**
```
┌────────┐  TB5 직결  ┌────────┐
│ Mini #1 │◄────────►│ Mini #2 │
│ 64GB    │  80Gbps   │ 64GB    │
└────────┘           └────────┘
```

**3대 (TB5 데이지체인):**
```
┌────────┐  TB5  ┌────────┐  TB5  ┌────────┐
│ Mini #1 │◄────►│ Mini #2 │◄────►│ Mini #3 │
│ 64GB    │      │ 64GB    │      │ 64GB    │
└────────┘      └────────┘      └────────┘
```

**4대 (TB5 스타/메시):**
```
┌────────┐  TB5  ┌────────┐
│ Mini #1 │◄────►│ Mini #2 │
└───┬────┘      └────┬───┘
    │ TB5            │ TB5
┌───┴────┐      ┌────┴───┐
│ Mini #3 │◄────►│ Mini #4 │
└────────┘  TB5  └────────┘
```

> 참고: TB5 데이지체인은 3대 이상에서 대역폭 경합이 발생할 수 있다. 4대는 TB5 허브 또는 메시 구성 권장. EXO 프레임워크가 토폴로지를 자동 감지하여 최적 라우팅한다.

#### 대형 모델 분산 실행

RDMA를 통해 메모리를 합쳐 단일 노드에 안 올라가는 대형 모델도 실행 가능:

| 모델 | 메모리 (Q4) | 1대 64GB | 2대 128GB | 3대 192GB | 4대 256GB |
|------|-----------|---------|----------|----------|----------|
| Qwen3-32B | 19GB | ✅ | ✅ | ✅ | ✅ |
| Qwen3-72B | 40GB | ✅ | ✅ | ✅ | ✅ |
| Llama 3.1 70B | 40GB | ✅ | ✅ | ✅ | ✅ |
| DeepSeek V3 671B | ~350GB | ❌ | ❌ | ❌ | ❌ (6대 필요) |
| Llama 3.1 405B | ~230GB | ❌ | ❌ | ❌ | ✅ (빠듯) |

단, 분산 실행 시 단일 요청 TPS는 하락함 (노드 간 통신 오버헤드). **동시 사용자 분산이 클러스터의 핵심 가치.**

#### 비용 효율성 비교

| 지표 | Mini 4대 (₩12M) | Spark 2대 NVLink (₩10M) | M3 Ultra (₩15M) |
|------|---------------|------------------------|-----------------|
| 총 메모리 | 256GB | 256GB | 256GB |
| 동시 10명 체감 TPS | **~6/인** | ~2-3/인 | ~3.6/인 |
| 동시 20명 체감 TPS | **~3/인** | ~1-1.5/인 | ~1.8/인 |
| CUDA | ❌ | ✅ | ❌ |
| 이미지 생성 | 느림 | **빠름** | 느림 |
| 장애 격리 | **✅ 최고 (노드 단위)** | ❌ | ❌ |
| 확장 단위 | **+₩3M/대** | +₩5M | 불가 |
| 전력 (전체) | ~200W | ~280W | ~150W |

#### Mac Mini 클러스터의 한계

- **CUDA 없음**: 이미지/영상 생성이 필요하면 Spark 추가 필수
- **단일 요청 TPS는 노드 수와 무관**: 1명 사용 시 ~60 tok/s (MLX) 고정. 클러스터 추가로 올라가지 않음
- **macOS 서버 운영**: 자동 업데이트/SIP 등 관리 필요 (노드 수만큼 관리 대상 증가)
- **EXO 프레임워크 성숙도**: 프로덕션 안정성 검증 필요
- **TB5 대역폭 경합**: 4대 이상에서 허브 구성 시 대역폭 감소 가능

### 5.3 동시 사용자 처리 방식 비교

| 방식 | 원리 | 장점 | 단점 |
|------|------|------|------|
| CUDA continuous batching | GPU 내부에서 요청 병합 | 메모리 효율적 (PagedAttention) | CUDA 전용 |
| NVLink Tensor Parallel | 모델을 2노드에 분할 | 대형 모델 가능 | 단일 요청 오버헤드, 장애 격리 불가 |
| **Mac 클러스터 (EXO/RDMA)** | **각 노드가 독립 요청 처리** | **선형 확장, 장애 격리** | 단일 요청 속도 향상 없음 |

**AIMS 특성 (동시 10~20명, 독립적 채팅)에는 Mac 클러스터의 "독립 노드 분산" 방식이 가장 적합.**

### Phase 2: 상호 보완 장비 추가 (₩3~5M 추가, 누적 ₩8M)

**어느 쪽이든 Phase 2에서 Spark + Mini 조합(₩8M)으로 수렴한다.**

```
└──────────────┘          └──────────┘
```

- Mini: LLM 채팅 전담 (MLX ~60 tok/s)
- Spark: 이미지 생성 (2.6초) + VLM OCR + STT + TTS + 동시 서빙 백업
- VLM OCR 검증 후 Upstage 해지 검토

**Phase 3 트리거:**
- [ ] 동시 10명+ 지연 발생, 또는
- [ ] 70B 모델 상시 필요

## 10. 결론 (v4)

### 가성비 최적안: 2가지 경로

**경로 A — DGX Spark 1대 (₩5M)로 시작** (서비스 서버 우선)
- NVFP4로 22-56 tok/s + FLUX 2.6초 + NIM 동시 서빙
- CUDA 풀스택, Linux, 128GB
- 이미지 생성이 필요하거나 동시 사용자가 중요하면 이 경로

**경로 B — Mac Mini 1대 (₩3M)로 시작** (최저 비용 우선)
- MLX 4bit로 ~60 tok/s (단일 TPS 최고)
- ₩3M 최저 비용으로 진입
- 1명 전용 AI 서버로 충분하거나, 비용 최소화가 최우선이면 이 경로

**어느 경로든 Phase 2에서 Spark+Mini 조합(₩8M)으로 수렴.** 최종적으로는 동일 구성.

### 최적 성능안: Mac Studio M3 Ultra + DGX Spark (₩20M+)

MLX로 ~90+ TPS + CUDA NVFP4 + FLUX 2.6초. 예산 무관 시 최고 성능.

### 핵심 원칙

- **Phase 0에서 트리거 조건 모니터링** → 조건 충족 시에만 투자
- **각 Phase는 이전 Phase 안정 검증 후 진입** (PoC 필수 원칙 준수)
- **외부 API 폴백 항상 유지** → AI Gateway 이중화 구조
- **최적 프레임워크 사용 필수** → Mac: MLX, Spark: vLLM NVFP4 (llama.cpp Q4 사용 금지)
- **이미지/영상/TTS는 비즈니스 필요성 확인 후 검토**
