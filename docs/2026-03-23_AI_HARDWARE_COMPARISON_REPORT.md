## 1. 배경

AIMS의 외부 AI API 의존성(OpenAI, Upstage)을 로컬 추론으로 전환하기 위해 하드웨어 도입을 검토한다.

### 1.1 도입 동기

| 동기 | 상세 |
|------|------|
| 데이터 보안 | 고객 보험 문서가 외부 API 서버로 전송되는 문제 해소 |
| 외부 종속성 제거 | OpenAI/Upstage 장애 시 AIMS 전체 AI 기능 마비 방지 |
| 비용 구조 개선 | 월 과금 → 일회성 투자 전환 (현재 월 $10 수준이나 확장 시 증가) |
| 확장 대비 | 설계사 100명, 동시 사용자 10~20명 규모 대응 |

### 1.2 ROI 분석 (교차검증에서 추가)

현재 월 $10(연 ₩17만) 수준이므로, 비용 절감만으로는 투자 정당화가 어렵다.

| 시나리오 | 월 API 비용 | ₩5M 투자 손익분기 | ₩10M 투자 손익분기 |
|---------|-----------|-----------------|------------------|
| 현재 (소수 사용자) | ₩14,000 | 29년 | 59년 |
| 100명 보수적 | ₩70,000 | 6년 | 12년 |
| 100명 적극적 | ₩420,000 | 12개월 | **24개월** |

**투자 정당화 근거는 비용 절감이 아닌 데이터 보안 + 외부 종속성 제거**에 있다. 금융당국의 AI 데이터 역외 처리 규제가 강화되면 필수 투자가 된다.

## 4. 워크로드별 비교 분석

### 4.1 LLM 추론 (병목: 메모리 대역폭)

토큰 생성(decode)은 연산력이 아닌 메모리 대역폭이 결정한다.
**TPS = 메모리 대역폭 ÷ 모델 크기** (양자화 수준에 따라 크게 변동)

Qwen3-32B **Q4 양자화 (~19GB)** 기준:

| 하드웨어 | 대역폭 | 이론 TPS | 실효 TPS (~85%) |
|---------|--------|---------|----------------|
| Mac Mini M4 Pro | 273GB/s | 14.4 | ~12 |
| DGX Spark | 273GB/s | 14.4 | ~12 |
| Mac Studio M4 Max (16코어) | 546GB/s | 28.7 | **~24** |
| Mac Studio M3 Ultra | 819GB/s | 43.1 | **~36** |
| DGX Spark 2대 NVLink (TP=2) | ~450-500GB/s* | 23.7-26.3 | **~20-22** |

*NVLink TP 효율 80~90% 적용. GB10에서 TP 지원 자체가 미확인 상태 (리스크)

### 4.2 Diffusion 모델 (병목: 연산력 FLOPS)

| 작업 | DGX Spark (GB10) | Mac M3 Ultra | 차이 |
|------|------------------|-------------|------|
| FLUX.1 1024×1024 | 빠름 | 느림 | **1.5~2.5배** (GB10 기준 수정) |
| SD3.5 1024×1024 | 빠름 | 느림 | 1.5~2배 |

> 교차검증 수정: 초기 "3~5배" 추정은 데이터센터 GPU(H100) 기준이었다. GB10은 통합 SoC이므로 **1.5~2.5배가 현실적** 추정이다. GB10의 정확한 FP16 TFLOPS는 미공개 상태.

### 4.3 OCR VLM

VLM OCR은 이미지 인코딩(연산력 병목) + 텍스트 생성(대역폭 병목)이 혼합된 워크로드.
**실측 없이 어느 쪽이 유리한지 단정 불가.** PoC에서 양쪽 비교 필수.

### 4.4 파인튜닝

AIMS 규모(설계사 100명)에서 비현실적. 학습 데이터 수천~수만 건 라벨링, 대기업 AI팀 수준의 작업. **도입 근거에서 제외.**

#### 동시 사용자 체감 속도 비교 (핵심)

클러스터의 진짜 강점은 "각 노드가 독립적으로 요청을 처리"하는 것이다.
고가 단일 장비는 대역폭을 나눠 쓰지만, 클러스터는 각 노드가 풀 스피드 유지.

Qwen3-32B Q4 기준, 각 사용자가 체감하는 TPS:

| 동시 사용자 | Mini 1대 | Mini 2대 | Mini 3대 | Mini 4대 | M3 Ultra 1대 | Spark 2대 NVLink |
|-----------|---------|---------|---------|---------|-------------|-----------------|
| 1명 | 12 | 12 | 12 | 12 | **36** | 20-22 |
| 3명 | 4 | **12** | **12** | **12** | 12 | 7-8 |
| 5명 | 2.4 | 6 | **12** | **12** | 7 | 4-5 |
| 10명 | 1.2 | 3 | 4 | **6** | 3.6 | 2-3 |
| 15명 | 0.8 | 2 | **4** | **4** | 2.4 | 1.5-2 |
| 20명 | 0.6 | 1.5 | 2.7 | **3** | 1.8 | 1-1.5 |

> **동시 5명 이상에서 Mini 3~4대 클러스터가 ₩15M M3 Ultra보다 체감 속도가 빠르다.**
> 이는 각 Mini가 독립 노드로 요청을 분담하기 때문이다.

#### 클러스터 토폴로지

**2대 (TB5 직결):**
```
┌────────┐  TB5 직결  ┌────────┐
│ Mini #1 │◄────────►│ Mini #2 │
│ 64GB    │  80Gbps   │ 64GB    │
└────────┘           └────────┘
```

**3대 (TB5 데이지체인):**
```
┌────────┐  TB5  ┌────────┐  TB5  ┌────────┐
│ Mini #1 │◄────►│ Mini #2 │◄────►│ Mini #3 │
│ 64GB    │      │ 64GB    │      │ 64GB    │
└────────┘      └────────┘      └────────┘
```

**4대 (TB5 스타/메시):**
```
┌────────┐  TB5  ┌────────┐
│ Mini #1 │◄────►│ Mini #2 │
└───┬────┘      └────┬───┘
    │ TB5            │ TB5
┌───┴────┐      ┌────┴───┐
│ Mini #3 │◄────►│ Mini #4 │
└────────┘  TB5  └────────┘
```

> 참고: TB5 데이지체인은 3대 이상에서 대역폭 경합이 발생할 수 있다. 4대는 TB5 허브 또는 메시 구성 권장. EXO 프레임워크가 토폴로지를 자동 감지하여 최적 라우팅한다.

- **CUDA 없음**: 이미지/영상 생성이 필요하면 Spark 추가 필수
- **단일 요청 TPS 향상 없음**: 1명 사용 시 12 tok/s 고정 (M3 Ultra의 36에 비해 낮음)
- **macOS 서버 운영**: 자동 업데이트/SIP 등 관리 필요 (노드 수만큼 관리 대상 증가)
- **EXO 프레임워크 성숙도**: 프로덕션 안정성 검증 필요
- **TB5 대역폭 경합**: 4대 이상에서 허브 구성 시 대역폭 감소 가능

### 5.3 동시 사용자 처리 방식 비교

| 방식 | 원리 | 장점 | 단점 |
|------|------|------|------|
| CUDA continuous batching | GPU 내부에서 요청 병합 | 메모리 효율적 (PagedAttention) | CUDA 전용 |
| NVLink Tensor Parallel | 모델을 2노드에 분할 | 대형 모델 가능 | 단일 요청 오버헤드, 장애 격리 불가 |
| **Mac 클러스터 (EXO/RDMA)** | **각 노드가 독립 요청 처리** | **선형 확장, 장애 격리** | 단일 요청 속도 향상 없음 |

**AIMS 특성 (동시 10~20명, 독립적 채팅)에는 Mac 클러스터의 "독립 노드 분산" 방식이 가장 적합.**

## 10. 결론

### 가성비 최적안: Mac Mini M4 Pro 64GB ×2 TB5 클러스터 (₩5M)로 시작

1. **₩2.5M 단위의 점진적 확장**이 가능하여 과잉 투자 리스크 최소화
2. **RDMA over Thunderbolt 5**로 동시 사용자 분산 처리 (AIMS 패턴에 최적)
3. **장애 격리** 기본 제공 (한 대 고장 시 나머지 운영)
4. Phase 2에서 Spark 추가로 CUDA 워크로드 대응
5. Phase 3에서 Mini 추가로 선형 확장

### 최적 성능안: Mac Studio M3 Ultra + DGX Spark (₩20M+)

예산 무관 시 최고 성능. 819GB/s LLM TPS + CUDA Diffusion.

### 핵심 원칙

- **Phase 0에서 트리거 조건 모니터링** → 조건 충족 시에만 투자
- **각 Phase는 이전 Phase 안정 검증 후 진입** (PoC 필수 원칙 준수)
- **외부 API 폴백 항상 유지** → AI Gateway 이중화 구조
- **이미지/영상/TTS는 비즈니스 필요성 확인 후 Phase 2에서 검토**
