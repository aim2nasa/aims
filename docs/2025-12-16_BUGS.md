# AR 파싱 관련 이슈 목록 (2025-12-16)

## 배경
- AIMS 시스템의 Annual Report(AR) 파싱 기능 관련 버그
- AR 문서: 보험사에서 고객에게 발송하는 연간 보유계약 현황 PDF
- AR 파싱: OpenAI API로 PDF에서 계약 정보 추출 → MongoDB `customers.annual_reports[]`에 저장

## 관련 파일
- **Backend**
  - `backend/api/annual_report_api/main.py` - AR 파싱 큐 워커
  - `backend/api/annual_report_api/routes/background.py` - `parse_single_ar_document()` 함수
  - `backend/api/annual_report_api/services/queue_manager.py` - 큐 상태 관리
  - `backend/api/aims_api/server.js` - 문서 삭제 API
- **Frontend**
  - `frontend/aims-uix3/src/features/customer/views/CustomerDetailView/tabs/AnnualReportTab.tsx`
  - `frontend/aims-uix3/src/features/customer/views/CustomerDetailView/tabs/DocumentsTab.tsx`

## 미해결 버그

### 1. 문서 탭 실시간 새로고침 안 됨
- **증상**: AR 문서 업로드 직후 문서 탭에서 해당 문서가 보이지 않음. F5 새로고침 필요.
- **재현 방법**:
  1. 고객 상세 페이지 → 문서 탭
  2. AR PDF 파일 드래그&드롭 업로드
  3. 업로드 완료 후 문서 목록에 새 문서가 안 보임
  4. F5 누르면 보임
- **예시**: 2025-12-16 테스트 - 송유미 고객에게 AR 4개 업로드 후 문서 탭에 안 보임 → F5 후 보임
- **원인**: DocumentsTab에 폴링 로직이 없어서 업로드된 문서가 자동 반영되지 않음
- **해결**: DocumentsTab에 10초 폴링 추가 (Page Visibility API로 백그라운드 탭에서는 중지)
- **관련 파일**: `DocumentsTab.tsx`
- **상태**: ⚠️ 코드 수정됨 - **테스트 필요**

---

### 2. AR 상태 동기화 버그
- **증상**: 동일한 AR 문서가 두 곳에 서로 다른 상태로 동시 표시됨
  - Annual Report 탭 테이블: "실패" 배지
  - Annual Report 탭 상단 "분석 중" 섹션: "대기 중" 표시
- **재현 방법**:
  1. AR 문서 업로드 → 파싱 실패 (Rate limit 등)
  2. Annual Report 탭 확인
  3. 테이블에는 "실패"로 표시되는데, 상단 "분석 중" 섹션에도 "대기 중"으로 표시됨
- **예시**: 안영미 AR 문서가 테이블에서 "실패" + 상단에서 "대기 중" 동시 표시
- **원인 분석**:
  - `files.ar_parsing_status`가 파싱 시작 시 'processing'으로 업데이트되지 않았음
  - 큐 워커가 작업 시작해도 files 컬렉션 상태는 'pending'으로 유지
- **해결**:
  - `parse_single_ar_document()` 시작 시 상태를 'processing'으로 설정 추가
- **관련 파일**: `background.py`
- **상태**: ⚠️ 수정됨 - 검증 필요 (배포됨 2025-12-16)

### 3. 재시도 후 Annual Report 탭 자동 새로고침 안 됨
- **증상**: "실패" 상태에서 재시도 버튼 클릭 → 파싱 완료되어도 화면이 자동 갱신되지 않음. F5 필요.
- **재현 방법**:
  1. AR 파싱 실패 상태 확인
  2. 재시도 버튼 클릭
  3. 파싱 완료 대기 (약 30초)
  4. 화면이 자동 갱신되지 않음
  5. F5 누르면 "완료" 상태로 변경됨
- **시도한 수정**:
  - `AnnualReportTab.tsx`에 폴링 로직 추가
  - `hasProcessingReports` memo로 processing/pending 상태 AR 감지
  - 10초마다 `loadAnnualReports()` 호출
- **예상 원인**: #2(상태 동기화 버그)가 해결되지 않아서 폴링해도 상태가 안 바뀜
- **관련 파일**: `AnnualReportTab.tsx`
- **상태**: ⚠️ 코드 추가했으나 검증 필요 (폴링 로직)

### 4. AR 업로드 후 Annual Report 탭 폴링 안 됨
- **증상**: AR 문서 업로드 직후 Annual Report 탭이 "Annual Report가 없습니다" 상태에서 자동 갱신 안 됨. F5 필요.
- **재현 방법**:
  1. 고객 상세 페이지 → Annual Report 탭 (비어있는 상태)
  2. 문서 탭에서 AR PDF 업로드
  3. Annual Report 탭으로 돌아와도 "Annual Report가 없습니다" 표시
  4. F5 누르면 "분석 중" 표시됨
- **예시**: 송유미 고객에 AR 업로드 후 Annual Report 탭 변화 없음
- **근본 원인 (발견됨)**:
  - 폴링 조건이 `pendingCount > 0 || hasProcessingReports`
  - 업로드 직후에는 둘 다 0/false라서 **폴링이 아예 시작 안 됨**
  - `loadPendingDocuments()`가 초기 로드 후 다시 호출되지 않음
- **관련 파일**: `AnnualReportTab.tsx`
- **해결**: 폴링에서 항상 `loadPendingDocuments()` 호출하도록 수정
- **상태**: ✅ 수정 완료 (2025-12-16)

### 6. Rate Limit 오류
- **증상**: 여러 AR 문서를 동시에 파싱하면 OpenAI Rate limit 오류 발생
- **에러 메시지**: `Rate limit reached for gpt-4.1... Limit 30000, Used 15011, Requested 26470... Please try again in 22.962s`
- **근본 원인 (발견됨)**:
  - **재시도 버튼이 큐 시스템을 우회하고 직접 실행!**
  - `background.py:retry_ar_parsing()`이 `BackgroundTasks`로 직접 파싱 실행
  - 30초 대기는 `main.py` 큐 워커에만 적용되어 재시도에는 적용 안 됨
- **시도한 해결**:
  1. `main.py` 큐 워커에 30초 대기 추가
  2. `background.py` 재시도 엔드포인트가 큐 시스템 사용하도록 변경
  3. `parse_single_ar_document()` 시작 시 상태를 'processing'으로 설정
- **2025-12-16 테스트 결과**:
  - 큐 순차 처리는 동작하지만 **30초 대기가 부족함**
  - 로그: `Used 18437, Requested 26470` → 1분 TPM 제한(30000) 초과
  - 대형 AR 문서(26000+ 토큰)가 연속될 때 Rate limit 발생
  - 재시도 메커니즘으로 결국 성공하지만 **실패가 너무 자주 발생**
- **추가 조치 필요**:
  - 30초 → **60초**로 대기 시간 증가 검토
  - 또는 토큰 사용량 기반 동적 대기 시간 적용
- **관련 파일**: `main.py`, `background.py`
- **상태**: ⚠️ 부분 해결 - 30초 대기가 부족, 추가 조치 필요

## 해결된 이슈

### ✅ 재시도 버튼 호버 없이 안 보임 (Bug #4)
- **증상**: "실패" 배지 옆의 재시도 버튼이 마우스 호버 없이는 보이지 않음
- **원인**: CSS 변수 `--color-primary`가 정의되지 않아 버튼 배경이 투명
- **해결**: CSS 변수 fallback 추가 (`--color-primary-500, #3b82f6`)
- **수정 파일**: `AnnualReportTab.css`
- **검증 완료**: 2025-12-16

---

### ✅ AR 삭제 시 파싱 결과 자동 삭제
- **증상**: AR 문서를 삭제해도 `customers.annual_reports[]` 배열에서 파싱 결과가 삭제되지 않음
- **원인**: 기존 코드가 `issue_date`만으로 매칭하여 같은 날짜의 다른 고객 AR도 삭제될 수 있었음
- **해결**: `customer_name + issue_date` 조합으로 정확히 매칭하도록 수정
- **수정 파일**: `backend/api/aims_api/server.js`
- **검증 완료**: 2025-12-16
