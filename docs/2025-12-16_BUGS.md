# AR 파싱 관련 이슈 목록 (2025-12-16)

## 배경
- AIMS 시스템의 Annual Report(AR) 파싱 기능 관련 버그
- AR 문서: 보험사에서 고객에게 발송하는 연간 보유계약 현황 PDF
- AR 파싱: OpenAI API로 PDF에서 계약 정보 추출 → MongoDB `customers.annual_reports[]`에 저장

## 관련 파일
- **Backend**
  - `backend/api/annual_report_api/main.py` - AR 파싱 큐 워커
  - `backend/api/annual_report_api/routes/background.py` - `parse_single_ar_document()` 함수
  - `backend/api/annual_report_api/services/queue_manager.py` - 큐 상태 관리
  - `backend/api/aims_api/server.js` - 문서 삭제 API
- **Frontend**
  - `frontend/aims-uix3/src/features/customer/views/CustomerDetailView/tabs/AnnualReportTab.tsx`
  - `frontend/aims-uix3/src/features/customer/views/CustomerDetailView/tabs/DocumentsTab.tsx`

## 미해결 버그

### 1. 문서 탭 실시간 새로고침 안 됨
- **증상**: AR 문서 업로드 직후 문서 탭에서 해당 문서가 보이지 않음. F5 새로고침 필요.
- **재현 방법**:
  1. 고객 상세 페이지 → 문서 탭
  2. AR PDF 파일 드래그&드롭 업로드
  3. 업로드 완료 후 문서 목록에 새 문서가 안 보임
  4. F5 누르면 보임
- **예시**: 정부균 AR 문서 업로드 후 문서 탭에 안 보임 → F5 후 보임
- **예상 원인**: 업로드 후 `refresh()` 호출이 안 되거나, 업로드 컴포넌트와 문서 목록 간 상태 동기화 문제
- **관련 파일**: `DocumentsTab.tsx`, 업로드 컴포넌트
- **상태**: ❌ 미해결

---

### 2. AR 상태 동기화 버그 (근본 원인)
- **증상**: 동일한 AR 문서가 두 곳에 서로 다른 상태로 동시 표시됨
  - Annual Report 탭 테이블: "실패" 배지
  - Annual Report 탭 상단 "분석 중" 섹션: "대기 중" 표시
- **재현 방법**:
  1. AR 문서 업로드 → 파싱 실패 (Rate limit 등)
  2. Annual Report 탭 확인
  3. 테이블에는 "실패"로 표시되는데, 상단 "분석 중" 섹션에도 "대기 중"으로 표시됨
- **예시**: 안영미 AR 문서가 테이블에서 "실패" + 상단에서 "대기 중" 동시 표시
- **예상 원인**:
  - `files.ar_parsing_status` (files 컬렉션)와 `customers.annual_reports[].status` (customers 컬렉션)가 동기화 안 됨
  - 파싱 실패 시 `files.ar_parsing_status`는 "error"로 업데이트되지만, `customers.annual_reports[]`에는 반영 안 됨
  - 또는 두 데이터 소스를 각각 다른 API에서 가져와서 불일치 발생
- **관련 파일**:
  - `background.py` - `parse_single_ar_document()` 함수에서 상태 업데이트
  - `queue_manager.py` - `mark_failed()` 함수
  - `AnnualReportTab.tsx` - 프론트엔드 데이터 조회
- **상태**: ❌ 미해결
- **중요도**: 🔴 높음 - 이 버그가 #3의 근본 원인일 가능성 높음

### 3. 재시도 후 Annual Report 탭 자동 새로고침 안 됨
- **증상**: "실패" 상태에서 재시도 버튼 클릭 → 파싱 완료되어도 화면이 자동 갱신되지 않음. F5 필요.
- **재현 방법**:
  1. AR 파싱 실패 상태 확인
  2. 재시도 버튼 클릭
  3. 파싱 완료 대기 (약 30초)
  4. 화면이 자동 갱신되지 않음
  5. F5 누르면 "완료" 상태로 변경됨
- **시도한 수정**:
  - `AnnualReportTab.tsx`에 폴링 로직 추가
  - `hasProcessingReports` memo로 processing/pending 상태 AR 감지
  - 10초마다 `loadAnnualReports()` 호출
- **예상 원인**: #2(상태 동기화 버그)가 해결되지 않아서 폴링해도 상태가 안 바뀜
- **관련 파일**: `AnnualReportTab.tsx`
- **상태**: ⚠️ 코드 추가했으나 검증 필요 (폴링 로직)

### 4. 재시도 버튼 호버 없이 안 보임
- **증상**: "실패" 배지 옆의 재시도 버튼이 마우스 호버 없이는 보이지 않음
- **재현 방법**:
  1. AR 파싱 실패 상태 확인
  2. "실패" 배지 확인 - 버튼이 안 보임
  3. 해당 행에 마우스 호버하면 버튼 나타남
- **시도한 수정**:
  - `AnnualReportTab.css`에서 상태 컬럼 너비 90px → 110px로 증가
  - 버튼이 잘려서 안 보이는 것으로 추정
- **관련 파일**: `AnnualReportTab.css`, `AnnualReportTab.tsx`
- **상태**: ⚠️ 수정했으나 검증 필요

### 5. Rate Limit 오류
- **증상**: 여러 AR 문서를 동시에 파싱하면 OpenAI Rate limit 오류 발생
- **에러 메시지**: `Rate limit reached for gpt-4.1... Limit 30000, Used 15011, Requested 26470... Please try again in 22.962s`
- **원인**:
  - OpenAI gpt-4.1 TPM 제한: 30,000 tokens/min
  - AR 파싱은 약 25,000 tokens 사용
  - 큐 워커가 연속으로 파싱 요청 → Rate limit 초과
- **시도한 수정**:
  - `main.py` 큐 워커에 30초 대기 추가
  ```python
  # 🔄 Rate Limit 방지: 다음 작업 전 30초 대기
  await asyncio.sleep(30)
  ```
- **관련 파일**: `backend/api/annual_report_api/main.py`
- **상태**: ⚠️ 코드 추가했으나 **서버 배포 필요**
- **배포 명령**:
  ```bash
  scp d:/aims/backend/api/annual_report_api/main.py tars.giize.com:/home/rossi/aims/backend/api/annual_report_api/
  ssh tars.giize.com 'cd /home/rossi/aims/backend/api/annual_report_api && ./deploy_annual_report_api.sh'
  ```

## 해결된 이슈

### ✅ AR 삭제 시 파싱 결과 자동 삭제
- **증상**: AR 문서를 삭제해도 `customers.annual_reports[]` 배열에서 파싱 결과가 삭제되지 않음
- **원인**: 기존 코드가 `issue_date`만으로 매칭하여 같은 날짜의 다른 고객 AR도 삭제될 수 있었음
- **해결**: `customer_name + issue_date` 조합으로 정확히 매칭하도록 수정
- **수정 파일**: `backend/api/aims_api/server.js`
- **검증 완료**: 2025-12-16
