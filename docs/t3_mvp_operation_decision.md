# T3 워크플로우 MVP 운영 판단 및 레드팀 검토 대응

---

## 1. Temp 파일 잔존 문제
- Temp 파일이 장기적으로 남아 있는 것은 **문제가 있었다는 증거**로 분석 대상.
- MVP 단계에서는 즉시 삭제보다 **왜 Temp 파일이 남는지 원인 파악**이 더 중요.
- 향후 운영 단계에서는 cron 등으로 주기적 정리 예정.

---

## 2. Temp 파일 삭제 자동화
- MVP에서는 **문제 원인 분석 우선**, 삭제 자동화는 **후속 과제**.
- 운영 고도화 시 cron/cleaner를 통한 주기적 삭제 적용.

---

## 3. 보안/노출 범위
- 현재 업로드 Webhook은 인증 없이 호출 가능 → **보완 필요성 인정**.
- 그러나 MVP 단계에서는 **기능 검증이 우선**, critical한 이슈 아님.
- 정식 운영 시 업로드 인증 및 권한 체계 추가 예정.

---

## 4. 대량 부하 대응
- MVP 예상 사용자는 최대 50명 수준, 동시접속은 더 적음.
- **부하 대응은 사용자 증가 전까지 보류**, 확장 전 반드시 대응 필요.
- 현재는 오버엔지니어링보다 빠른 검증이 더 중요.

---

## 5. DB 장애 및 재시도
- DB 장애 시 자동 재시도 대신 **즉시 실패 알림**이 MVP 운영에 더 적합.
- 장애 발생 시 관리자가 직접 원인 파악 및 즉시 대응.
- 후속 단계에서 재시도/큐잉 로직 검토 예정.

---

## ✅ 레드팀 결론
- **MVP 운영 수준에서는 타당한 판단.**
- 1인 운영, 낮은 트래픽, 빠른 검증이라는 현실적 조건에 부합.
- 다만 **운영 확장 전 반드시 보완해야 할 로드맵** 필요.
