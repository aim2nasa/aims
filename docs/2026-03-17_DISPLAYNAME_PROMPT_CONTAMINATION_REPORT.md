## 1. 문제 요약

AI 별칭(displayName) 생성 프롬프트에 포함된 예시 이름 "안영미"가 GPT-4o-mini 출력을 오염시켜, **문서 텍스트에 없는 엉뚱한 사람 이름이 별칭에 포함**되는 버그.

## 2. 근본 원인

### 오염 소스

`backend/api/document_pipeline/services/openai_service.py` 436줄:

```python
"- 문서 유형명만으로 된 제목 금지 (예: '진료비 계산서' -> '안영미 진료비 2011.09')",
```

이 예시에 포함된 **"안영미"라는 실제 사람 이름**이 Few-shot 앵커로 작동하여, 문서 텍스트에 사람 이름이 없는 경우 AI가 예시 이름을 차용합니다.

### 발생 메커니즘

1. 규칙 434줄: "문서에 등장하는 사람 이름이 있으면 포함" → AI에게 "이름을 넣어야 한다"는 강한 유인
2. 문서 텍스트에 사람 이름이 없음 (보장내용표, 깨진 텍스트 등)
3. **고객 이름이 프롬프트에 전달되지 않음** — AI가 참조할 실제 이름이 없음
4. 프롬프트에서 유일하게 참조 가능한 이름 = 예시의 "안영미"
5. GPT-4o-mini가 예시 이름을 실제 출력에 사용

### 보조 요인

- 원본 파일명에 이름이 있어도 (예: "윤두현제안서.xlsx") 공백 없이 붙어있어 토크나이징 불명확
- Few-shot 예시(규칙 상단)의 영향력 > 파일명 힌트(중간)의 영향력
- `temperature=0.3`이지만 0이 아니므로 편향 반영 가능
- `doc_display_name.py` 76줄 주석에도 동일한 예시 존재 (런타임 영향은 없으나 정리 필요)

## 4. 잠재적 변종 위험

### 4-1. 동일 프롬프트의 다른 하드코딩 값

현재 프롬프트에는 "안영미" 외에 다른 하드코딩된 값이 없으나, 예시의 **"2011.09"** 날짜도 오염 가능:
- "2011.09" 포함 displayName: 5건 → 모두 실제 문서 날짜와 일치 (정상)
- 현재까지 날짜 오염은 확인되지 않음

### 4-2. existing_aliases 자기증폭 경로 (Gini 발견)

`doc_display_name.py:285-293`에서 동일 고객의 기존 `displayName` 최대 30건이 프롬프트에 삽입됩니다. 이미 오염된 "안영미 XXX" 별칭이 `existing_aliases`로 다시 전달되면, **오염이 자기증폭(self-amplifying)**될 수 있는 구조적 경로가 존재합니다.

### 4-3. AI 이름 Hallucination 탐지 한계

본 조사는 "안영미"라는 특정 키워드 검색에 의존합니다. AI가 프롬프트 예시에서 영감을 받아 **전혀 다른 이름을 hallucinate**한 경우(예: 아예 새로운 이름 생성)는 이 방법으로 탐지할 수 없습니다. 이 한계는 프롬프트 레벨이 아닌 **출력 검증 로직**으로만 방어 가능합니다.

### 4-4. 구조적 재발 조건

다음 조건이 동시에 성립하면 재발:
1. 문서 텍스트에 사람 이름이 없거나 불명확
2. 원본 파일명에도 이름이 없거나 파싱 불가
3. 고객 이름이 프롬프트에 전달되지 않음 (현재 구조)
4. AI가 "이름 포함" 규칙을 적용하려고 시도

취약한 문서 유형:
- 보장내용표, 상품 비교표 (표 데이터만 존재)
- 깨진 인코딩 파일 (txt, 구버전 문서)
- 이미지 기반 OCR 실패 문서
- 고객명이 비어있는 문서 (customerId 없음)

## 5. `generate_title_only()` 호출 경로 분석 (Gini 발견)

현재 별칭 생성 함수는 **4개 경로**에서 호출되며, 각 경로의 파라미터 전달 상태가 다릅니다:

| # | 호출 지점 | 파일 | original_filename 전달 | customer_name 전달 |
|---|----------|------|----------------------|-------------------|
| 1 | displayName API (단건) | `doc_display_name.py:207` | O | **X** |
| 2 | displayName API (배치) | `doc_display_name.py:383` | O | **X** |
| 3 | OCR 워커 | `ocr_worker.py:325` | **X** | **X** |
| 4 | PDF 변환 워커 | `pdf_conversion_worker.py:298` | **X** | **X** |

**핵심 문제**: 4개 경로 모두 `customer_name`을 전달하지 않습니다. OCR/PDF 워커는 `original_filename`도 전달하지 않아 더 취약합니다.

## 6. 수정 방향 (권고)

### 필수 수정 (프롬프트)

1. **프롬프트 예시에서 실제 이름 제거** — 비인명 정보로 교체
   - 현재: `'진료비 계산서' -> '안영미 진료비 2011.09'`
   - 수정안: `'진료비 계산서' -> '삼성화재 진료비 2024.03'` (보험사명 사용)

2. **명시적 금지 규칙 추가**
   - "문서 텍스트, 파일명, 고객 정보에 없는 이름은 절대 사용 금지"
   - "이름을 찾을 수 없으면 이름 없이 생성"

3. **`doc_display_name.py` 76줄 주석도 동일하게 수정** (Alex 발견)

### 필수 수정 (코드 구조)

4. **`_build_title_prompt()`에 `customer_name` 파라미터 추가** (Gini 발견, 근본 해결)
   - 호출부 4곳 모두에서 고객명 조회 후 전달
   - 프롬프트에 "이 문서의 고객: {customer_name}" 명시

5. **OCR/PDF 워커에 `original_filename` 전달 추가** (Gini 발견)
   - `ocr_worker.py:325`, `pdf_conversion_worker.py:298`

6. **`temperature` 인하 검토** (Alex 발견)
   - displayName은 창의성보다 정확성 중요 → `0.3` → `0.1` 또는 `0` 검토

### 방어적 수정 (선택)

7. **출력 검증 후처리**: 생성된 displayName에 문서/파일명에 없는 고유명사가 포함되었는지 검증
8. **existing_aliases 삽입 시** "이하 목록의 이름을 새 별칭에 그대로 사용 금지" 명시

### 7-2. 버그/정상 분류 기준
각 문서에 대해 다음 3가지를 교차 확인:
- `meta.full_text.includes("안영미")` 또는 `ocr.full_text.includes("안영미")`
- `upload.originalName.includes("안영미")`
- `customerId === ObjectId("69b58751aa8b707b051ba4c8")` (안영미 고객)

3가지 모두 false인 경우만 "프롬프트 오염"으로 판정.

> **주의**: OCR 텍스트에서 "안영 미" (공백 분리)로 추출된 경우 `includes("안영미")`는 false이나, 실제 텍스트에 이름이 존재하므로 정상으로 분류함 (`69b2362f` 건)

### 7-3. 날짜 오염 확인

db.files.find({displayName: /2011\.09/}).count()  // → 5 (모두 정상)
