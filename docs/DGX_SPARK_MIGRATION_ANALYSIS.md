## 3. 기능별 대체 계획

### 3-1. AI 채팅 + MCP 40개 도구 호출

**현재:** `gpt-4.1-mini` (소형~중형 모델)

gpt-4.1-mini 수준의 function calling은 32B급 오픈소스 모델로 충분히 대체 가능하다.

| 로컬 모델 후보 | 파라미터 | 메모리 | DGX Spark 1대 출력 속도 |
|---------------|---------|--------|----------------------|
| Qwen3-14B (NVFP4) | 14B | ~10GB | 22.71 tok/s |
| Qwen3-32B | 32B | ~20GB | ~15-20 tok/s (추정) |
| GPT-OSS-20B (MXFP4) | 20B | ~15GB | 82.74 tok/s |

**근거:**
- BFCL v4 벤치마크에서 오픈소스 모델(GLM-4.5)이 1위 (70.85%)
- Qwen3는 MCP 공식 지원 (Qwen-Agent 프레임워크)
- gpt-4.1-mini는 gpt-4o보다 가벼운 모델이므로 대체 장벽이 낮음

**대체 난이도:** 중간
**예상 소요:** 2~3주 (테스트 + 프롬프트 최적화)

JSON mode 출력 필요. 8B 모델로 충분하다.

**대체 난이도:** 낮음
**예상 소요:** 2~3일 (RAG 답변과 동시 작업)

### 3-4. 문서 요약

**현재:** `gpt-4o-mini` (하드코딩)

문서 텍스트(최대 10,000자) → 3~5줄 요약 + 태그 생성.
8B~14B 모델로 가능. 한국어 요약 품질은 프롬프트 최적화로 보완.

**대체 난이도:** 낮음~중간
**예상 소요:** 3~5일

### 3-7. 문서 OCR (Upstage API 대체)

**현재:** Upstage Document OCR (`$0.0015/페이지`)
- 엔드포인트: `https://api.upstage.ai/v1/document-digitization`
- 용도: 스캔된/이미지 PDF에서 텍스트 추출 → 임베딩/검색/AR감지에 활용
- 관련 파일: `backend/api/document_pipeline/services/upstage_service.py`

#### PaddleOCR의 한국어 한계

PaddleOCR는 106개 언어를 지원하지만, 한국어 전용 학습 모델은 PP-OCRv3까지만 공개되어 있다.
최신 v4/v5에 한국어 전용 모델이 없어 **한국어 인식 정확도가 중국어/영어 대비 낮다.**

#### 권장 대안: VLM 기반 OCR (Qwen2.5-VL-7B)

전통 OCR 대신 Vision Language Model로 OCR을 수행하는 방식이 더 정확하다.

| 비교 항목 | PaddleOCR (전통 OCR) | Qwen2.5-VL-7B (VLM OCR) |
|----------|---------------------|------------------------|
| 한국어 정확도 | 중간 (v3 모델 기반) | **높음** (언어 이해 기반) |
| 인식 방식 | 글자 단위 패턴 매칭 | **문서 전체를 "이해"하며 텍스트 추출** |
| 문맥 보정 | 불가 | **가능** (예: "보헝"→"보험" 자동 교정) |
| 테이블 추출 | 별도 모듈 필요 | 구조 이해 포함 |
| 메모리 | ~1GB | ~5GB |
| DGX Spark 속도 | 매우 빠름 | 41.71 tok/s (NVIDIA 공식 벤치) |

**VLM이 한국어 OCR에 더 적합한 이유:**

```
PaddleOCR (전통 OCR):
  이미지 → 문자 검출 → 문자 인식 (글자 단위 패턴 매칭)
  → 한국어 학습 데이터 부족 = 인식률 저하

VLM (Qwen-VL):
  이미지 → "이 문서의 텍스트를 추출해줘" → 텍스트 출력
  → 한국어를 "언어"로 이해 (OCR이 아니라 문서 이해)
  → 문맥 기반 오류 보정 가능
```

#### 기타 후보

| 모델 | 한국어 | 특징 | 비고 |
|------|--------|------|------|
| PaddleOCR PP-OCRv5 | 지원 (정확도 낮음) | 경량, 빠름 | 파인튜닝으로 개선 가능하나 라벨링 노동 필요 |
| olmOCR 2 (8B) | 미검증 | 벤치마크 82.4점 (최상위) | 영어 중심, 한국어 테스트 필요 |
| EasyOCR | 지원 | 설치 간편 | 정확도는 PaddleOCR와 유사 |

#### 검증 필요 사항

Upstage에서 VLM 전환 전, **AIMS 실제 보험 문서 10~20건으로 품질 비교 테스트 필수:**
1. Upstage OCR 결과 vs Qwen2.5-VL-7B 결과 비교
2. 글자 인식률, 레이아웃 보존, 테이블 텍스트 추출 품질 비교
3. 통과 시 전환, 부족 시 Upstage 유지 또는 하이브리드

**대체 난이도:** 중간 (VLM 연동은 쉬우나 품질 검증 필요)
**예상 소요:** 1~2주 (테스트 + 파이프라인 수정)

## 8. 파인튜닝 (선택, 품질 향상용)

파인튜닝 없이도 작동할 가능성이 높지만, 하면 정확도를 더 높일 수 있다.

### 데이터 확보

AIMS에 이미 GPT-4.1-mini의 채팅 로그(도구 호출 포함)가 MongoDB에 축적되어 있다.
이 로그를 학습 데이터로 변환하면 "GPT-4.1-mini가 성공적으로 호출한 패턴"을 그대로 학습시킬 수 있다.

### 학습 환경

| 방법 | 가능 여부 | 비용 |
|------|----------|------|
| DGX Spark에서 Qwen3-32B QLoRA | 가능 (128GB 충분) | 무료 (전기세만) |
| 클라우드 GPU 렌탈 | 가능 | 1회 $50~200 |

### 기대 효과

- 파인튜닝 전: gpt-4.1-mini **동등** 수준
- 파인튜닝 후: AIMS 도메인에서 gpt-4.1-mini **이상** 수준 (전용 모델)
