## 2. GT(Ground Truth) 확보 전략

### GT 확보 가능 여부

| 코드 | GT 가능 | GT 소스 | 비고 |
|------|:-------:|---------|------|
| Q1 | ✅ | 키워드 검색 API | 100% 정확한 문서 목록 반환 |
| Q2 | ✅ | 키워드 검색 API | 존재 여부 = 검색 결과 유무 |
| Q3 | ❌ | 없음 | 정답 해석이 정의 불가 (open-ended generation) |
| Q4 | ✅ | MongoDB `customers.annual_reports` | 구조화된 계약 데이터 |
| Q5 | ✅ | MongoDB `customers` | 구조화된 고객 데이터 |
| Q6 | ✅ | MongoDB aggregation | 정확한 합계/건수 산출 |
| Q7 | ✅ | MongoDB 날짜 쿼리 | 정확한 범위 필터/정렬 |
| Q8 | ✅ | MongoDB `customer_relationships` | 구조화된 관계 데이터 |
| Q9 | ⚠️ | 개별 유형 GT 조합 | 개별 정보는 검증 가능, 통합 응답 품질은 검증 불가 |

### GT 소스별 분류

- **키워드 검색 API**: Q1, Q2 — 문서 검색 영역
- **MongoDB 직접 조회**: Q4, Q5, Q6, Q7, Q8 — 구조화 데이터 영역
- **자동 GT 생성 가능**: Q4~Q8은 DB 구조만 알면 자동 GT 생성 가능
- **GT 없음**: Q3 — supervised learning 적용 불가

## 3. Q3 (문서 내용 질의) 대응 전략

### 문제
- 문서 내용에 대한 "올바른 해석/요약"의 정답을 정의할 수 없음
- Supervised learning 적용 불가능한 open-ended generation 문제
- AI가 해석을 생성하면 오염(hallucination) 위험

### 결정된 접근법
- AI가 답변을 만들어내지 않음
- 대신 **키워드 검색 결과 + RAG 검색 결과**를 사용자에게 직접 제공
- 구글 검색처럼 "검색 결과를 보여주는 것"이 최선
- 이 이상 줄 수 있는 것은 없음

### 구현 시 미해결 과제
- LLM이 프롬프트 지시를 무시하고 해석을 끼워 넣을 수 있음
- 프롬프트 지시만으로는 100% 방지 불가
- **프롬프트 + 프론트엔드 후처리** 이중 방어 필요
- 복잡도가 높아 추가 설계 필요

## 4. Q9 (복합 질의) 대응 전략

### 문제
- 개별 GT는 확보 가능하나, 통합 응답 품질의 GT는 정의 불가
- 여러 도구를 올바른 순서로 호출하고 결과를 합치는 과정의 정답이 없음

### 결정된 접근법
- 복합 질의를 개별 질의로 분해
- 각각 답변 가능한 부분만 답변
- 답변 불가 부분(Q3 영역)은 "답변 불가"로 명시
- 개별 정보의 정확성만 GT로 검증

## 6. 실제 채팅 로그에서 발견된 문제 (2026-03-20)

### 문제 1: 집계 누락 (Q6)
- 질의: "캐치업코리아 보험료 얼마나 내고 있어?"
- 기대: 월 보험료 합계 포함
- 실제: 개별 목록만 나열, **합계 없음**

### 문제 2: 도구 활용 실패 (Q7)
- 질의: "가장 최근에 계약한 고객은?"
- 기대: 계약일 기준 정렬 후 답변
- 실제: "확인되지 않습니다"로 답변, 재질문 후에야 응답

### 문제 3: 필터링 실패 (Q4)
- 질의: "김보성 고객이 계약자로 되어 있는 계약들을 모두 보여줘"
- 기대: 계약자명으로 필터링
- 실제: "등록되어 있지 않습니다"로 잘못된 답변

### 원인 분석
| 원인 | 해당 문제 | 해결 방향 |
|------|----------|----------|
| 프롬프트에 집계/해석 가이드 부족 | 문제 1 | 시스템 프롬프트 튜닝 |
| 도구 파라미터/기능 부족 | 문제 2, 3 | MCP 도구 확장 |
