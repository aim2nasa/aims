# 고객별 문서함 검색 UX 개선 보고서

> 작성일: 2026-03-10
> 상태: **진행 중**

---

## 1. 문제 정의

### 1.1 검색창 placeholder와 실제 동작 불일치
- placeholder: 항상 "파일명으로 검색..."
- 요약 모드(초성 미선택): 실제로는 **고객명** 필터
- 초성 모드(초성 선택): 실제로는 **파일명+고객명** 필터
- → 사용자 혼란 유발

### 1.2 검색 모드 칩 레이블 불일치
- "파일명" 칩이 요약 모드에서 실제로 고객명 필터 동작
- "내용"/"AI 질문" 칩이 요약 모드에서 무의미 (문서 데이터 미로드)

### 1.3 요약 모드에서 파일명 검색 불가
- 고객명만 검색 가능, 파일명에 포함된 키워드로 문서/고객 찾기 불가
- 기술 원인: 요약 모드에서는 API가 고객 요약만 반환 (문서 데이터 없음)

### 1.4 검색 결과에서 고객 클릭 시 검색 컨텍스트 소실 (해결 완료)
- 커밋: `0ca22371`
- "곽" 검색 → "곽승철" 클릭 → 검색어 보존 + 해당 고객만 표시 + 필터 해제 시 복원

## 2. 관계자 회의 결과 (2026-03-10)

### 2.1 설계사(PM) — 사용자 니즈 (최우선)

> "단 하나의 검색창에 무엇을 입력하든 시스템이 알아서 의도를 파악하고 가장 관련성 높은 결과를 보여주는 것"

| 순위 | 시나리오 | 예시 | 빈도 |
|------|---------|------|------|
| P0 | 고객명 검색 | "곽" → 곽씨 고객들 | 매일 수십 번 |
| P1 | 복합 검색 | "곽승철 보험증권" | 낮지만 니즈 강함 |
| P2 | 파일명 검색 | "연금보험" → 해당 문서의 고객 | 보통 |

### 2.2 UX 에이전트 — 설계안
- 요약 모드: "고객명" / "문서 검색" 2칩 구조 (Progressive Disclosure)
- 초성 모드: "파일명" / "내용" / "AI 질문" 3칩 구조 (현행)
- placeholder를 모드별로 정확하게 표시

### 2.3 Alex — 기술 분석
- **권장**: explorer-tree API에 `search` 파라미터 추가
- 서버에서 파일명 매칭 문서의 고객만 필터 → 기존 UI 유지
- 기존 `getRecentDocuments`의 search regex 패턴 재활용
- 백엔드 ~30줄, 프론트엔드 ~25줄
