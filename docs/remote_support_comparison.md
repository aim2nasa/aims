## 1. 검토 배경 및 요구사항

### AIMS 특수 조건
- **자체 호스팅 필수**: 보험 고객 개인정보(계약자 정보, 주민번호, 보험료 납입 내역 등)가 외부 서버를 경유할 수 없음
- **개인정보보호법 준수**: 국외 이전 미동의 시 해외 공용 서버 사용 불가
- **설계사 친화적**: 비전문가인 설계사가 쉽게 사용 가능해야 함
- **보안 양호**: 알려진 미패치 취약점 없을 것

### 인프라 환경
- 서버: TARS (tars.giize.com, Ubuntu, 7.8GB RAM)
- MongoDB, nginx, Redmine, n8n, Qdrant, FastAPI 운영 중
- Tailscale VPN으로 내부 보안 통신

## 5. 검토한 다른 솔루션들

### 5-1. RustDesk 공식 클라우드 서버 (무료)
- **제외 이유**: 보험 고객 화면이 해외 공용 서버 경유 → 개인정보보호법 위반 위험, 금융감독원 보안 가이드라인 저촉 가능

### 5-2. Apache Guacamole
- 관리자 측 클라이언트 설치 불필요 (브라우저만으로 접속)
- 설계사 PC에 VNC 또는 RDP 서버 활성화 필요
- Windows 10 Home 에디션은 RDP 호스트 기능 없음
- **제외 이유**: 설계사 PC 설정 변경이 필요해 비전문가 대응 어려움

### 5-3. DWService
- 브라우저 기반, 에이전트 설치 없이 1회성 지원 가능
- **제외 이유**: 자체 호스팅 불가 (클라우드 전용) → 보험 데이터 외부 서버 경유 문제

### 5-4. Getscreen.me
- 링크 클릭만으로 즉시 연결, 자체 호스팅 옵션 있음
- **제외 이유**: 자체 호스팅은 유료 (가격 비공개), 오픈소스 아님

### 5-5. NoMachine
- NX 프로토콜로 화면 품질 업계 최고 수준
- **제외 이유**: 설계사 측 클라이언트 설치 필요, 기업 용도 유료

### ~~결론: MeshCentral 채택~~ (2026-03-17 초기 판단)

### 결론 (2026-03-18 최종): **RustDesk 채택 — "필요 시에만 포트 개방" 전략**

#### 결정 변경 사유

AIMS의 원격 지원 시나리오는 **"설계사가 전화 → 잠깐 도와주고 → 끝"** 패턴이다.
MeshCentral은 상시 모니터링/관리용으로 설계된 플랫폼으로, 이 용도에는 오히려 과도하다.

RustDesk의 서버 CVE(CVE-2026-30784) 문제는 **포트를 항상 열어두지 않는 운용 전략**으로 현실적 위험을 수용 가능 수준까지 낮출 수 있다.

#### 포트 온디맨드 개방 + 자동 닫기 전략

```
평소:   UFW 포트 닫힘 → hbbs/hbbr 외부 접근 불가 → 공격 표면 제로
지원 시: 포트 개방 → 원격 지원 수행 → 세션 종료 감지 → 자동 포트 차단
```

수동 닫기는 **관리자가 잊어버릴 위험**이 있다. 자동 닫기로 이 위험을 완전히 제거한다.

**TARS 서버 제어 스크립트:**

```bash
# ~/aims/scripts/rustdesk_port.sh

case "$1" in
  start)
    # 포트 개방
    sudo ufw allow 21115:21119/tcp
    sudo ufw allow 21116/udp
    echo "✅ RustDesk 포트 개방 — 원격 지원 가능"
    echo "⏳ 세션 종료 시 자동으로 포트가 닫힙니다."

    # 백그라운드에서 세션 종료 감지 → 자동 닫기
    (
      # hbbr(릴레이)에 활성 연결이 있는 동안 대기
      sleep 30  # 연결 수립까지 대기
      while ss -tn state established '( dport = 21117 or sport = 21117 )' | grep -q .; do
        sleep 10
      done
      # 연결 종료 확인 → 포트 차단
      sudo ufw delete allow 21115:21119/tcp
      sudo ufw delete allow 21116/udp
      echo "🔒 세션 종료 감지 — 포트 자동 차단 완료"
    ) &
    echo "감시 PID: $!"
    ;;
  close)
    # 수동 긴급 차단 (비상용)
    sudo ufw delete allow 21115:21119/tcp
    sudo ufw delete allow 21116/udp
    echo "🔒 RustDesk 포트 수동 차단 완료"
    ;;
  *)
    echo "Usage: $0 {start|close}"
    echo "  start  포트 열기 + 세션 종료 시 자동 닫기"
    echo "  close  긴급 수동 닫기"
    ;;
esac
```

#### 보안 평가

| 항목 | 평가 |
|------|------|
| 평소 노출 | 없음 ✓ |
| 지원 중 노출 시간 | 수십 분 이내 ✓ |
| CVE-2026-30784 위험 | 포트 열린 동안만, 단시간 ✓ |
| 서버 패치 전 운용 | 단시간 노출이라 현실적 수용 가능 △ |

> 공격자가 포트 스캔 → 취약점 악용까지 시간이 필요하다. 수십 분 단위로 닫으면 실질적 위험은 매우 낮다.

## 7. RustDesk 도입 계획

### Phase 1: 서버 설치 (TARS)

```bash
# Docker Compose로 hbbs/hbbr 설치
# 포트는 UFW에서 기본 차단 상태로 유지
mkdir -p /opt/rustdesk && cd /opt/rustdesk
```

```yaml
# docker-compose.yml
version: '3'
services:
  hbbs:
    image: rustdesk/rustdesk-server:latest
    container_name: rustdesk-hbbs
    command: hbbs
    volumes:
      - ./data:/root
    network_mode: host
    restart: unless-stopped

  hbbr:
    image: rustdesk/rustdesk-server:latest
    container_name: rustdesk-hbbr
    command: hbbr
    volumes:
      - ./data:/root
    network_mode: host
    restart: unless-stopped
```

| 항목 | 설정 | 이유 |
|------|------|------|
| 배포 방식 | Docker Compose | 기존 TARS 인프라와 일관성 |
| UFW 기본 상태 | 21115~21119 **차단** | 평소 공격 표면 제로 |
| 포트 개방 | 지원 시에만 스크립트로 열기/닫기 | CVE-2026-30784 위험 최소화 |
| 공개키 | 서버 기동 시 자동 생성 (`id_ed25519.pub`) | 클라이언트 설정에 사용 |

### Phase 2: 포트 자동 제어 스크립트 배포

```bash
# ~/aims/scripts/rustdesk_port.sh
# 사용:
#   ./rustdesk_port.sh start   포트 열기 + 세션 종료 시 자동 닫기
#   ./rustdesk_port.sh close   긴급 수동 닫기 (비상용)
```

- `start`: 포트 개방 후 백그라운드에서 릴레이 세션 감시 → 세션 종료 시 자동 차단
- `close`: 비상 상황 시 수동 즉시 차단
- **관리자가 닫기를 잊어도 자동으로 닫히므로 포트 방치 위험 제로**

향후 확장: AIMS 관리자 UI에서 원클릭 버튼으로 `start` 실행 (API 호출 → SSH 스크립트 실행)

### Phase 3: 클라이언트 배포 (설계사 PC)

1. RustDesk 클라이언트 **1.4.6+** 설치 (CVE 패치 완료 버전)
2. 클라이언트 설정:
   - ID/Relay Server: `tars.giize.com`
   - Key: 서버의 `id_ed25519.pub` 내용 입력
3. 설계사에게 안내: "도움 필요 시 → RustDesk 실행 → 화면의 ID 읽어주세요"

### Phase 4: 운용 프로세스

```
설계사 전화: "도와주세요"
  ↓
관리자: rustdesk_port.sh start (포트 개방 + 자동 닫기 감시 시작)
  ↓
설계사: RustDesk 실행 → ID 알려줌
  ↓
관리자: RustDesk로 접속 → 원격 지원
  ↓
지원 완료 → RustDesk 연결 종료
  ↓
[자동] 세션 종료 감지 → 포트 자동 차단 (관리자 조치 불필요)
```

**현재 기본 보호 (3중):** 서버 공개키(미설정 시 서버 사용 불가) + 세션 ID+비밀번호(모두 알아야 연결) + 포트 자동 닫기(노출 시간 최소화)

| 옵션 | 효과 | 적용 난이도 | 비고 |
|------|------|-------------|------|
| **A. 설계사 IP 화이트리스트** | 해당 IP 외 접근 완전 차단 | 낮음 | 유동 IP 시 매번 변경 필요 |
| **B. ISP 대역 제한** | 해외/봇 접근 차단, 국내 ISP만 허용 | 낮음 | KT/SKT/LG 대역만 허용 |
| **C. Tailscale 경유** | 포트 개방 자체 불필요 (VPN만으로 연결) | 중간 | 설계사 PC에 Tailscale 설치 필요 |

#### 옵션 A: 설계사 IP 화이트리스트

```bash
# 전체 개방 대신 특정 IP만 허용
sudo ufw allow from $설계사IP to any port 21115:21119 proto tcp
sudo ufw allow from $설계사IP to any port 21116 proto udp
```

설계사가 전화 시 IP를 확인하여 스크립트에 전달하는 방식. 유동 IP 환경에서는 매 세션마다 IP가 달라질 수 있어 운용이 번거로움.

#### 옵션 B: 국내 ISP 대역 제한

```bash
# 주요 ISP 대역만 허용 (예시)
sudo ufw allow from 210.0.0.0/8 to any port 21115:21119 proto tcp   # KT 등
sudo ufw allow from 121.0.0.0/8 to any port 21115:21119 proto tcp   # SKT 등
```

해외 봇/스캐너를 차단하면서 설계사 IP 변경에 영향받지 않음. 완벽한 차단은 아니지만 공격 표면을 크게 줄임.

#### 옵션 C: Tailscale VPN 경유

설계사 PC에 Tailscale을 설치하면 UFW 포트 개방 없이 `100.x.x.x` VPN IP로 직접 연결 가능. 포트 노출이 완전히 제거되어 CVE-2026-30784 위험이 제로. 단, 설계사 PC에 추가 소프트웨어 설치가 필요하여 초기 도입 단계에서는 부담.

**운용 흐름 (적용 시):**
1. 설계사 전화 → IP 확인 (설계사가 IP 확인 페이지 접속 또는 구두 전달)
2. `rustdesk_port.sh start 설계사IP` → 해당 IP만 포트 개방
3. 세션 종료 → 자동 차단

**고려사항:**
- 설계사 IP는 대부분 유동 IP (LTE, 가정용) → 매 세션마다 확인 필요
- 자동화 가능: 설계사가 특정 URL 접속 → 서버가 IP 캡처 → 자동 포트 개방
- 보안은 확실하지만 운용 단계가 하나 늘어남

### 주의사항
- 서버 hbbs/hbbr는 Docker로 항상 실행 중이지만, UFW가 외부 접근을 차단하므로 평소 노출 없음
- 자동 닫기는 릴레이(21117) 활성 연결을 감시 — 연결이 모두 끊어지면 30초 내 포트 차단
- `close` 명령은 비상용 수동 차단으로 항상 사용 가능
- RustDesk 서버 1.1.16+ 패치 출시 후에는 상시 포트 개방으로 전환 가능
