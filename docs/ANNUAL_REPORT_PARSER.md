### 1.3 출력 JSON 스키마
```typescript
interface AnnualReportSummary {
  insuredName: string;           // 피보험자명
  totalContracts: number;        // 보유계약 건수
  monthlyPremiumTotal: number;   // 월 보험료 총액
  contracts: Contract[];         // 보유계약 목록
  lapsedContracts: Contract[];   // 부활가능 실효계약
}

interface Contract {
  seq: number;                   // 순번
  policyNumber: string;          // 증권번호
  productName: string;           // 보험상품
  contractor: string;            // 계약자
  insured: string;               // 피보험자
  contractDate: string;          // 계약일 (YYYY-MM-DD)
  status: string;                // 계약상태
  coverageAmount: number;        // 가입금액 (만원)
  insurancePeriod: string;       // 보험기간
  paymentPeriod: string;         // 납입기간
  premium: number;               // 보험료 (원)
}
```

### 3.3 핵심 파싱 전략

#### 라인별 파싱 (Line-by-Line Parsing)
- 각 계약 데이터가 한 줄에 포함된 패턴 활용
- 정규표현식: `/^(\d{1,2})\s+(00\d{8})\s+(.+?)\s+(\d{4}-\d{2}-\d{2})\s+(.+?)\s+([\d,]+)$/`

#### 버퍼 메커니즘
- `productNameBuffer`: 상품명이 이전 줄에서 시작하는 경우 (예: "무배당...")
- `statusBuffer`: 상태가 여러 줄에 걸쳐 분리되는 경우 (예: "업무처" + "리중" → "업무처리중")

#### 특수 케이스 처리
- 법인 계약자명 병합 (예: "캐치업 코리아" → "캐치업코리아")
- 달러 상품 가입금액 소수점 처리 (예: 1,390.6)
- 상품명 접미사 병합 (예: "Ⅱ보험", "Plus")

## ⚠️ Phase 7: 오버피팅 문제 발견 및 근본적 해결 (2026-01-16)

### 7.1 문제 발견

기존 TypeScript 파서(`annualReportParser.ts`)는 **오버피팅(Overfitting)** 문제가 있었습니다.

**증상:**
- 5개 테스트 샘플에서 100% 정확도
- 하지만 새로운 데이터에서 실패 가능성 높음

**원인:** 하드코딩된 패턴
```typescript
// ❌ 특정 회사명 하드코딩
if (line === '캐치업') { result.contractorParts.push(line); }
if (line === '코리아') { result.contractorParts.push(line); }

// ❌ 특정 상태값 하드코딩
if (line === '업무처') { result.status = '업무처리중'; }

// ❌ 제한된 suffix 패턴
/^[ⅠⅡⅢ]?보험$/,  // Ⅳ, Ⅴ 등은?
```

### 7.2 근본적 원인 분석

```
PDF (테이블 구조 있음) → 텍스트 추출 (구조 손실) → 패턴 매칭 (추측)
                              ↑ 문제의 핵심
```

텍스트로 추출하는 순간 **테이블의 열 경계가 사라집니다**.
그래서 "상품명이 어디서 끝나고 계약자가 어디서 시작하는지"를 **추측**해야 하고,
이 추측이 하드코딩된 패턴이 됩니다.

### 7.3 근본적 해결책: 테이블 추출

```
PDF → pdfplumber TABLE 추출 (구조 보존) → 구조화된 JSON → 프론트엔드
                    ↑ 셀 경계 보존
```

pdfplumber의 `find_tables()` + `extract()` 사용:
- 각 셀이 완전한 단위로 추출됨
- 줄바꿈은 단순히 제거 (`cell.replace('\n', '')`)
- **하드코딩 불필요**

### 7.7 교훈

> **"100% 테스트 통과"는 의미 없다. 테스트 데이터에 오버피팅 되었을 수 있다.**
>
> 근본적인 질문: "새로운 데이터에서도 작동하는가?"
>
> 규칙 기반 파싱의 한계:
> - 구조 손실 후 패턴 매칭 = 추측
> - 추측이 하드코딩이 됨
> - 하드코딩은 새 데이터에서 실패
>
> 해결책:
> - **구조를 보존하는 방식 선택**
> - PDF → 테이블 추출 (구조 보존)
> - 또는 AI 기반 파싱 (semantic understanding)
