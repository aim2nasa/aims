- **작성일**: 2026-02-20
- **대상 파일**: `tools/metdo_reader/read_customer.py`

---

## 1. 문제 (Problem)

### 증상
AC v2가 MetLife 고객 상세화면을 캡처 → Upstage OCR로 인식할 때, **주소 필드에 파이프(`|`) 문자와 3중 반복 텍스트**가 나타남.

```
# TEXT 출력 예시 (남궁현 자택주소)
자택주소 18424 경기 화성시 동탄원천로 | 자택주소 18424 경기 화성시 동탄원천로 | 자택주소 18424...
```

### 근본 원인
MetLife는 **nexacro** UI 프레임워크를 사용하며, 화면 레이아웃이 **3컬럼 테이블 구조**로 렌더링됨.
Upstage OCR의 TEXT 출력은 이 구조를 `셀1 | 셀2 | 셀3` 형태로 파이프 구분하여 출력.
실제로는 `<td colspan="3">`으로 병합된 단일 셀인데, TEXT에서는 3번 반복되어 나타남.

```
# Upstage OCR 내부 해석
┌──────────────────────────────┐
│ 자택주소 18424 경기 화성시...  │  ← HTML: <td colspan="3"> (1회)
└──────────────────────────────┘
         ↓ TEXT 출력 시
자택주소 18424... | 자택주소 18424... | 자택주소 18424...  ← 3회 반복 + 파이프
```

### 영향
- 주소가 잘리거나 파이프 문자가 포함되어 저장됨
- 직장주소가 `| 직장주소 | 직장주소 | | | | | |` 형태로 깨짐
- 파이프 이후 텍스트가 다음 필드 파싱을 오염시킴

---

## 2. 해결 (Solution)

### 핵심 아이디어
Upstage OCR API는 동일 호출에서 **TEXT와 HTML 두 가지 출력**을 모두 반환함.
HTML 출력은 `<td colspan="3">`을 정확히 보존하므로 파이프 반복 문제가 원천적으로 없음.
**추가 API 비용 없이** HTML을 우선 파싱하면 문제 해결.

### 구현 내용

#### 2-1. `extract_text_from_html()` 함수 추가

Python 표준 라이브러리 `html.parser.HTMLParser` 기반으로 HTML에서 클린 텍스트를 추출.

- `<figure>`, `<figcaption>` 내부 텍스트 제외 (OCR 이미지 설명 노이즈)
- `<tr>`, `<br>` → 줄바꿈 삽입
- `<td>`, `<p>`, `<table>` 등 블록 요소 → 줄바꿈 삽입
- 연속 빈 줄 3개 이상 → 2개로 압축

#### 2-2. `parse_customer_info()` HTML 우선 로직

```
content.html 존재? ─── YES ──→ extract_text_from_html(html) 사용
       │
       NO
       │
       ▼
content.text 사용 (기존 방식, 폴백)
```

기존 파싱 함수(`parse_personal`, `parse_corporate`, `find_between`, `clean_address_section`)는 **전혀 수정하지 않음**.
입력 텍스트의 소스만 TEXT → HTML로 교체.

#### 2-3. 직장주소 `end_labels` 보강

```python
# 변경 전
find_between(text, "직장주소", ["직장명", "부서명", "세부정보"], max_chars=400)

# 변경 후
find_between(text, "직장주소", ["팩스번호", "직장명", "부서명", "세부정보"], max_chars=400)
```

나해정의 직장주소가 팩스번호 영역까지 침범하는 문제 해결.

#### 2-4. `sys.stdout` UTF-8 이동

모듈 레벨에서 `main()` 함수 내부로 이동하여 pytest 호환성 확보.
