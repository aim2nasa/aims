# ISSUE: 보험금청구서 document_type 분류 오류

- **발견일**: 2026-03-31
- **심각도**: Medium
- **상태**: Resolved (2026-03-31)
- **영향 범위**: 문서 분류 정확도, 고객별 문서함 트리 표시

---

## 현상

"계약자 DB손해보험 장기보험금청구서.pdf" 문서가 고객별 문서함에서 **기타 > 보험금청구서**로 분류됨.

정상 경로: **보험금청구 > 보험금청구서** (카테고리 `claim` > 소분류 `claim_form`)

## 원인 분석

### DB 상태
```json
{
  "_id": "69ca914f2b9e50dd4ad9ea7d",
  "document_type": "claim",
  "meta": {},
  "displayName": "계약자 DB손해보험 장기보험금청구서.pdf"
}
```

### 문제 구조

| 구분 | 값 | 역할 |
|------|-----|------|
| `claim` | 프론트엔드 **대분류 카테고리 코드** | 보험금청구 카테고리 전체를 의미 |
| `claim_form` | 백엔드 AI 분류 **소분류 값** | 보험금청구서 문서 유형 |

- `document_type: 'claim'`은 대분류 코드이지 소분류가 아님
- 프론트엔드 `getCategoryForType('claim')`에서 매핑 실패 → `etc`(기타)로 fallback
- 올바른 값은 `claim_form` → `getCategoryForType('claim_form')` → `claim` 카테고리

### 추가 의심점

- `meta: {}`가 비어 있음 → AI 분류(`summarize_text()`)가 정상 실행되지 않았거나, 별도 경로로 document_type이 설정된 가능성
- AI 분류의 `VALID_DOCUMENT_TYPES`에 `claim`은 없음 → `_validate_document_type()`에서 `general`로 변환되어야 하는데, `claim`으로 저장된 것은 AI 분류 외부에서 설정된 것으로 추정

### 관련 시스템 불일치

| DB DEFAULT_DOCUMENT_TYPES | AI VALID_DOCUMENT_TYPES | 상태 |
|---------------------------|-------------------------|------|
| `claim` (보험금청구서) | 없음 | 불일치 |
| `accident_cert` (사고증명서) | 없음 | 불일치 |
| `claim_form` | 없음 (DB에 미등록) | 불일치 |

DB의 `document-types-routes.js`에는 `claim`이 "보험금청구서"로 등록되어 있으나, AI 분류에서는 `claim_form`을 반환함. 두 시스템 간 용어가 통일되지 않음.

## 해결 방향

### 방안 1: 프론트엔드 방어 매핑 (즉시 적용 가능)
- `getCategoryForType('claim')` → `'claim'` 카테고리로 매핑 추가
- `claim`을 `claim_form`과 동일하게 "보험금청구서"로 표시

### 방안 2: DB 용어 통일 (근본 해결)
- `document-types-routes.js`의 `claim` → `claim_form`으로 변경
- 기존 `document_type: 'claim'` 문서를 `claim_form`으로 마이그레이션
- AI 분류와 DB 스키마 용어 완전 통일

### 방안 3: document_type 설정 경로 조사
- `meta: {}`인데 `document_type: 'claim'`이 설정된 경로 추적
- AI 분류 외부에서 document_type을 설정하는 코드 확인

## 관련 파일

## 해결 결과

### 적용된 방안: 방안 1 (프론트엔드 방어 매핑) + DB 데이터 보정 준비

#### 프론트엔드 수정 (`documentCategories.ts`)
- `LEGACY_TYPE_MAP` 추가: `{ 'claim': 'claim_form' }` 레거시→현행 매핑
- `prefetchDocumentTypes()` 2차 보정: 캐시 적재 시 레거시 항목의 category를 현행 값 기준으로 자동 보정
- `getCategoryForType()` / `getDocumentTypeLabel()`: 캐시 miss 시 LEGACY_TYPE_MAP fallback 추가
- 테스트 3건 추가 (41건 전체 PASS)

#### DB 데이터 보정 (미실행, 수동 확인 후 실행 예정)
```javascript
db.files.updateMany(
  { document_type: 'claim' },
  { $set: { document_type: 'claim_form' } }
)
```

#### 레거시 값 설정 경로 조사 결과
- AI 분류: `VALID_DOCUMENT_TYPES`에 `claim` 없음 → 이 경로 아님
- DocumentTypePickerModal: `isLegacy` 필터로 레거시 유형 선택 불가
- 결론: 신규 문서에 `claim`이 설정되는 경로 없음. 기존 DB 레거시 데이터만 존재
