# AutoClicker 배포 및 수익화 전략

> **상태**: Phase 0~2 구현 완료 (2026-02-13)
> **목적**: AC를 AIMS 고객에게 어떻게 제공하고, 정당한 대가를 받을 것인가

---

## 1. 현황

### AutoClicker란?
- AIMS 플랫폼의 **데스크톱 자동화 도구**
- 보험사 웹사이트에서 고객 정보/PDF를 자동 수집
- Python + SikuliX 기반, Windows 전용
- OCR 기능 내장 (크레딧 소모)

### 현재 구조
```
설계사 PC에서 직접 실행
├── Python 런타임 필요
├── SikuliX (Java) 필요
├── AIMS 서버와 연동 (문서 업로드, 크레딧 차감)
└── OCR 사용 시 크레딧 소모
```

### 비용 구조
| 항목 | 비용 발생 주체 |
|------|---------------|
| AC 개발/유지보수 | 우리 (개발 인력) |
| OCR API 호출 | 우리 (서버 비용) |
| 서버 인프라 | 우리 (AIMS 서버) |
| 문서 임베딩 | 우리 (AI API 비용) |

→ **단순 무료 다운로드 = 비용만 발생, 수익 없음**

## 2. 과거 기술과의 비교

### ActiveX 시대 (2000년대)
- 브라우저에서 `.cab` 다운로드 → 자동 설치/실행
- 보안 문제로 사라짐
- 하지만 **"웹에서 데스크톱 도구를 제공"**하는 개념은 유효

### 결론: AC는 "모던 ActiveX" 모델이다

2026-02-12 논의에서 도달한 핵심 인사이트:

> **"웹이 데스크톱 확장을 제어한다"는 ActiveX의 서비스 모델은 여전히 유효하다.**
> **ActiveX가 망한 건 모델이 아니라 보안이 없었기 때문이다.**

| | ActiveX (2000년대) | AC (2026) |
|---|---|---|
| **트리거** | 웹페이지가 로컬 컴포넌트 호출 | AIMS 웹이 `aims-ac://` URI로 AC 호출 |
| **설치** | `.cab` 다운로드 → 자동 설치 | Inno Setup 인스톨러 → 사용자 동의 설치 |
| **인증** | 없음 (코드 서명 신뢰 기반) | JWT 토큰 전달 + 서버 검증 |
| **권한** | 시스템 전체 접근 (보안 구멍) | OCR + 클릭으로 제한 |
| **업데이트** | 웹에서 새 버전 감지 → 재설치 | AC가 인스톨러 다운로드 → `/VERYSILENT` 자동 설치 (유저 모름) |
| **오프라인** | 가능 (로컬 실행) | 불가 (토큰 검증 필수) |
| **복제 방지** | 없음 | 토큰 없으면 실행 자체 불가 |

**현대 앱들도 동일 모델:**
- Zoom: `zoommtg://` — 웹에서 회의 참가 → 데스크톱 앱 실행
- Slack: `slack://` — 웹에서 Open in Slack → 데스크톱 앱 실행
- VS Code: `vscode://` — 웹에서 Open in VS Code → 에디터 실행

Custom URI Scheme은 ActiveX의 정신적 후계자이되, 보안 문제를 해결한 현대적 구현이다.

### 공통점 (유효한 것)
| 공통점 | 설명 |
|--------|------|
| 웹 → 데스크톱 브릿지 | 웹 플랫폼이 로컬 네이티브 도구를 호출 |
| 자동 배포 필요 | 비개발자가 쉽게 설치/업데이트 |
| 서비스 종속 | 웹 서비스 없이는 의미 없는 도구 |
| 라이선스 관리 | 정당한 사용자만 사용 가능해야 함 |

### 차이점 (개선된 것)
| 차이점 | ActiveX | AC |
|--------|---------|-----|
| 보안 | 무제한 접근 | 토큰 기반 인증, 제한된 동작 |
| 설치 동의 | 브라우저가 자동 설치 | 사용자가 인스톨러 직접 실행 |
| 브라우저 결합 | IE 종속 플러그인 | 브라우저 독립 (URI Scheme) |
| 크레딧 연동 | 없음 | 사용량 기반 과금 가능 |

## 3. 배포 방식 검토

### 3-1. 패키징

| 방식 | 장점 | 단점 |
|------|------|------|
| **PyInstaller (.exe)** | 단일 파일, Python 불필요 | 용량 큼, AV 오탐 가능 |
| **PyInstaller (폴더)** | 실행 빠름, SikuliX 번들 가능 | 배포 크기 큼 |
| **Nuitka** | 네이티브 컴파일, 성능 우수 | 빌드 복잡 |
| **MSI 인스톨러** | Windows 표준, 깔끔한 설치/제거 | 관리자 권한 필요할 수 있음 |

**권장**: PyInstaller (폴더) + SikuliX/JRE 번들 → ZIP 배포

### 3-2. 배포 채널

| 방식 | 장점 | 단점 |
|------|------|------|
| **AIMS 웹에서 다운로드** | 중앙 관리, 버전 통제 | 수동 업데이트 |
| **앱 내 자동 업데이트** | 항상 최신, 유저 부담 없음 | 구현 복잡 |
| **AIMS 웹 + 자동 업데이트** | 최초만 수동, 이후 자동 | 권장 조합 |

## 4. 수익화 모델

### 핵심 원칙
> **"AC는 AIMS의 부가가치 도구이며, 무료 배포 대상이 아니다."**

### 4-1. 라이선스 + 크레딧 하이브리드 모델

```
┌─────────────────────────────────────────────┐
│              AIMS 구독 플랜                    │
│                                              │
│  Basic     ─── AIMS 웹만 사용                 │
│  Pro       ─── AIMS 웹 + AC 사용권 + 월 N크레딧 │
│  Enterprise ── 전체 기능 + 대량 크레딧           │
└─────────────────────────────────────────────┘
```

| 과금 요소 | 방식 |
|-----------|------|
| AC 사용권 | 구독 플랜에 포함 (Pro 이상) |
| OCR 크레딧 | 사용량 기반 차감 |
| 문서 임베딩 크레딧 | 사용량 기반 차감 (기존 체계) |
| 초과 크레딧 | 추가 구매 가능 |

### 4-2. AC 실행 시 인증 흐름

```
AC 실행
  ├── 1. AIMS 서버에 라이선스 검증 요청
  │     POST /api/ac/verify-license
  │     { userId, machineId, acVersion }
  │
  ├── 2. 서버 응답
  │     { authorized: true, plan: "pro", credits: 150, minVersion: "0.2.0" }
  │
  ├── 3. 버전 체크
  │     현재 버전 < minVersion → 강제 업데이트 안내
  │
  ├── 4. 크레딧 체크
  │     credits <= 0 → "크레딧이 부족합니다" 안내
  │
  └── 5. 정상 실행
```

### 4-3. OCR 크레딧 차감

```
OCR 수행 시
  ├── 로컬에서 OCR 실행 (Tesseract 등)
  │     → 서버 비용 없음 → 크레딧 차감 소량 or 무료?
  │
  └── 서버 OCR API 호출
        → 서버 비용 발생 → 크레딧 차감
        → POST /api/ac/deduct-credit { userId, type: "ocr", count: 1 }
```

**토의 필요**: 로컬 OCR vs 서버 OCR에 따라 크레딧 정책이 달라짐

### 4-4. 크레딧 정책 (초안)

| 작업 | 크레딧 소모 | 비고 |
|------|------------|------|
| AC 실행 | 0 | 실행 자체는 무료 |
| 고객 1명 처리 | 0 | 자동화 자체는 무료 |
| OCR 1회 | ? | 로컬/서버 여부에 따라 결정 |
| PDF 다운로드 | 0 | 보험사 원본 다운로드 |
| 문서 임베딩 | 기존 정책 | AC와 무관, 업로드 시 차감 |

## 5. 기술 아키텍처 (배포 + 인증)

### 5-1. 전체 흐름

```
[AIMS 웹]                    [AC (설계사 PC)]              [AIMS 서버]
    │                              │                           │
    ├── AC 다운로드 ──────────────►│                           │
    │                              │                           │
    │                              ├── 라이선스 검증 ──────────►│
    │                              │◄── 인증 결과 ─────────────┤
    │                              │                           │
    │                              ├── 자동화 실행              │
    │                              │   (클릭, OCR, PDF 수집)    │
    │                              │                           │
    │                              ├── OCR 크레딧 차감 ────────►│
    │                              │                           │
    │                              ├── 문서 업로드 ─────────────►│
    │                              │                           │
    │                              ├── 버전 체크 ──────────────►│
    │                              │◄── 업데이트 URL ──────────┤
    │                              │                           │
```

### 5-2. 서버 API (신규 필요)

| 엔드포인트 | 용도 |
|-----------|------|
| `POST /api/ac/verify-license` | AC 실행 시 라이선스 + 크레딧 + 최소버전 검증 |
| `POST /api/ac/deduct-credit` | OCR 등 크레딧 차감 |
| `GET /api/ac/latest-version` | 최신 버전 정보 + 다운로드 URL |
| `POST /api/ac/usage-log` | 사용 통계 기록 (선택) |

### 5-3. AC 측 구현 필요 사항

| 기능 | 설명 |
|------|------|
| 로그인/인증 | AIMS 계정으로 로그인 (토큰 저장) |
| 라이선스 검증 | 실행 시 서버 호출, 오프라인 시 캐시된 토큰 사용 |
| 크레딧 표시 | GUI에 잔여 크레딧 표시 |
| 자동 업데이트 | 새 버전 감지 시 다운로드 + 교체 |
| 기기 바인딩 | machineId로 동시 사용 기기 수 제한 (선택) |

## 6. 유저 경험 원칙

### 최상위 철학
> **다운로드조차 하고 싶지 않다. 하지만 데스크톱 자동화라 어쩔 수 없이 하는 것이다.**
> **따라서 그 "어쩔 수 없는" 한 번을 최대한 매끄럽게 만들어야 한다.**

### 이상적 UX 계층 (위에서 아래로 갈수록 현실적)

| 등급 | 유저 경험 | 기술 | 가능 여부 |
|------|----------|------|----------|
| S | AIMS 웹에서 버튼 클릭 → 바로 동작 | 브라우저 자동화 | ❌ 데스크톱 GUI 제어 불가 |
| A | AIMS 웹에서 클릭 → AC 자동 실행 | Custom URI Scheme (`aims-ac://start`) | ✅ 가능 — 최초 1회 설치 후 |
| B | AIMS 웹에서 클릭 → 다운로드 → 더블클릭 → 끝 | 자동 설치 프로그램 | ✅ 가능 |
| C | 다운로드 → 압축 해제 → 더블클릭 | ZIP 배포 | ✅ 가능 (최소 구현) |

### 권장: A등급 — Custom URI Scheme

```
[AIMS 웹 — "자동수집 시작" 버튼 클릭]
         │
         ▼
  브라우저가 aims-ac://start?token=xxx&chosung=ㄱ 호출
         │
         ▼
  [AC 이미 설치됨?]──Yes──► AC 자동 실행 (설정 자동 적용)
         │
         No
         ▼
  "AutoClicker를 설치해주세요" 안내 + 원클릭 설치 버튼
         │
         ▼
  설치 프로그램 다운로드 → 실행 → 자동 설치 → URI Scheme 등록
         │
         ▼
  다음부터는 AIMS 웹에서 버튼만 클릭하면 AC 자동 실행
```

**현실 사례 (유저가 이미 익숙한 패턴):**
- Zoom: 웹에서 "회의 참가" → `zoommtg://` → Zoom 앱 자동 실행
- Slack: 웹에서 "Open in Slack" → `slack://` → Slack 앱 자동 실행
- VS Code: 웹에서 "Open in VS Code" → `vscode://` → VS Code 자동 실행

**AC도 동일:**
- AIMS 웹에서 "자동수집" → `aims-ac://` → AC 자동 실행 + 설정 자동 전달

### 유저가 하는 일
1. **최초 1회**: AIMS 웹에서 "AutoClicker 설치" 클릭 → 자동 설치
2. **이후**: AIMS 웹에서 "자동수집 시작" 클릭 → AC 자동 실행
3. **끝.**

### 유저가 절대 하지 않는 일
- ❌ Python, Java, SikuliX 등의 존재를 인식
- ❌ 환경변수, 명령 프롬프트, pip install
- ❌ 수동 업데이트 (자동 업데이트)
- ❌ AC 실행 파일을 직접 찾아서 실행

### 자동 업데이트 전략

```
AC 실행 시
  ├── AIMS 서버에 최신 버전 확인
  │     GET /api/ac/latest-version → { version: "0.2.0", url: "..." }
  │
  ├── 현재 버전 < 최신 버전?
  │     ├── Yes (선택적) → "업데이트 가능" 알림, 백그라운드 다운로드
  │     ├── Yes (강제)   → "필수 업데이트" → 자동 다운로드 + 재시작
  │     └── No           → 정상 실행
  │
  └── 업데이트 적용
        ├── 새 exe 다운로드 → 임시 폴더
        ├── AC 종료 → 교체 스크립트 실행 → 새 AC 실행
        └── 유저는 "잠시 업데이트 중..." 메시지만 봄
```

### 배포 패키지 구성 (유저에게 보이지 않는 내부 구조)

```
C:\Users\{user}\AppData\Local\AIMS\AutoClicker\
├── AutoClicker.exe          ← 실행 파일
├── runtime/
│   ├── python/              ← 임베디드 Python (~15MB, 시스템 설치 아님)
│   ├── jre/                 ← 임베디드 JRE (~50MB, 시스템 설치 아님)
│   └── sikulix/             ← SikuliX JAR
├── resources/               ← 이미지, 설정 등
└── version.json             ← 버전 정보
```

설치 경로가 `AppData\Local`이므로 **관리자 권한 불필요**.

## 8. 구현 우선순위 (2026-02-12 논의)

### 핵심 인사이트
> **"바로 유료 서비스가 아니라 시범 테스트 기간을 먼저 가진다."**
> **"하지만 시범 테스트라도 누가 실행하는지 구분되어야 한다."**
> **"AC를 독립 프로그램으로 두면 제어권을 잃는다. AC는 AIMS의 리모컨이다."**

### 아키텍처 원칙: AC는 AIMS의 확장이다

AC는 독립적인 데스크톱 프로그램이 **아니다**. AIMS 플랫폼의 데스크톱 확장 도구다.

```
❌ 독립 프로그램: AC.exe 더블클릭 → 알아서 실행 → 제어권 상실
✅ AIMS 확장:    AIMS 웹 로그인 → "자동수집" 클릭 → AC 실행 → AIMS가 통제
```

**독립 프로그램이면 안 되는 이유:**

| 위험 | 설명 |
|------|------|
| 무단 복제 | exe 파일만 복사해서 다른 PC에서 무단 사용 |
| 버전 통제 불가 | 구버전 계속 사용, 강제 업데이트 우회 |
| 크레딧 우회 | 서버 통신 없이 OCR만 돌리는 변종 가능 |
| 사용자 미식별 | 누가, 언제, 얼마나 쓰는지 파악 불가 |

**AIMS 종속 설계의 장점:**

| 장점 | 설명 |
|------|------|
| 인증 일원화 | AIMS 웹 로그인 = AC 인증 (이중 로그인 불필요) |
| 완전한 통제 | 서버에서 실행 허가/차단/버전 강제 가능 |
| 자연스러운 UX | "AIMS에서 버튼 누르면 동작" — 유저에게 별도 앱 느낌 없음 |
| 보안 | 토큰 없이는 실행 자체 불가, exe만으로는 무용지물 |

### 🔴 절대 원칙: AIMS 없이 AC 없다

```
AC 실행 시 첫 번째 동작 = AIMS 서버에 토큰 검증
  → 토큰 유효 → 정상 실행
  → 토큰 무효/만료 → "AIMS에서 다시 시작해주세요" → 종료
  → 서버 통신 불가 → "AIMS 서버에 연결할 수 없습니다" → 종료
```

**AC.exe를 USB로 복사해서 다른 PC에 가져가도:**
- AIMS 계정 없음 → 토큰 없음 → 실행 불가
- AIMS 미로그인 → 토큰 발급 안 됨 → 실행 불가
- 토큰 만료 → 재검증 실패 → 실행 중단

**exe는 껍데기일 뿐, AIMS 토큰이 열쇠다.**

### 우선순위 정리

| 순위 | 기능 | 난이도 | 이유 |
|:----:|------|:------:|------|
| **0** | **패키징 (배포용 빌드)** | 낮 | 시범 테스트의 물리적 전제 조건. 유저가 실행할 수 있어야 모든 것이 시작됨 |
| **1** | **사용자 식별 (인증)** | 중 | 3개 코드베이스 동시 변경 (AIMS 웹 + 서버 API + AC). 모든 기능의 전제 조건 |
| **2** | **자동 업데이트 (OTA)** | 낮~중 | 시범 테스트 유저에게 배포 후 버그 수정/기능 추가를 수동 전달 불가. 버전 체크 → 자동 교체 필수 |
| **3** | **크레딧 관리** | 높 | 단순하지 않음. 시범 테스트 기간에는 사용량 로깅만 먼저 하고, 데이터 기반으로 정책 설계 |

### 실행/인증 방식 결정 (A/B/C 비교)

논의 결과, **A 방식(AIMS 웹에서 실행)** 을 채택한다.

| 방식 | 실행 | 인증 | 설치 | 업데이트 | 판정 |
|:----:|------|------|------|---------|:----:|
| **A** | AIMS 웹 버튼 | 웹 토큰 전달 | 인스톨러 (URI Scheme 등록) | 사일런트 자동 업데이트 (`/VERYSILENT`) | **채택** |
| B | exe 직접 실행 | 브라우저 OAuth 연동 | ZIP or 인스톨러 | 복잡 (OAuth + 자체 업데이트) | 오버엔지니어링 |
| C | exe 직접 실행 | AC 자체 로그인 | ZIP | 매우 복잡 (실행 중 자기 교체) | 업데이트 불리 |

**A를 선택한 핵심 이유:**
- 업데이트가 투명: AC가 인스톨러 다운로드 → `/VERYSILENT` 실행 → 유저 모름
- 인증 일원화: AIMS 웹 로그인 = AC 인증 (별도 로그인 불필요)
- 인스톨러가 URI Scheme + 바탕화면 바로가기 + 프로그램 제거 전부 처리
- 비개발자 유저에게 가장 자연스러운 UX

### Phase 0: 패키징 + 인스톨러

#### 0-1. PyInstaller 빌드

AC GUI(Python)를 exe로 변환한다.

```
pyinstaller --onedir gui_main.py → dist/gui_main/
```

- AC GUI와 SikuliX는 이미 별도 프로세스로 분리되어 있음
- PyInstaller는 Python 부분만 패키징, SikuliX/JRE는 별도 번들

#### 0-2. Inno Setup 인스톨러

비개발자 유저가 설치할 수 있는 Windows 인스톨러를 만든다.

```
유저 경험: AIMS_AutoClicker_Setup.exe 더블클릭 → 다음 다음 완료 → 끝
```

**인스톨러가 하는 일:**
- `AppData\Local\AIMS\AutoClicker\`에 파일 설치 (관리자 권한 불필요)
- `aims-ac://` URI Scheme 레지스트리 등록 (HKCU)
- 바탕화면 바로가기 생성 (선택)
- "프로그램 추가/제거"에 등록
- 포터블 JRE + SikuliX JAR 포함

**설치 후 디렉토리 구조:**
```
C:\Users\{user}\AppData\Local\AIMS\AutoClicker\
├── AutoClicker.exe          ← PyInstaller 빌드 결과
├── updater.bat              ← 자동 업데이트 런처 (Phase 2에서 사용)
├── runtime/
│   ├── jre/                 ← 포터블 JRE (~50MB)
│   └── sikulix/             ← SikuliX JAR (~50MB)
├── resources/               ← 이미지, 설정 등
└── VERSION                  ← 버전 정보
```

**Phase 0에서 미리 포함하는 것들 (Phase 1~2에서 바로 사용):**
- `aims-ac://` URI Scheme 레지스트리 등록 — Phase 1에서 웹 버튼만 추가하면 바로 동작
- `updater.bat` — Phase 2에서 자동 업데이트 시 사용하는 런처 스크립트

**필요한 작업:**
- `pyinstaller --onedir gui_main.py` 빌드 설정
- SikuliX 실행 경로를 상대 경로로 변경 (현재 절대 경로)
- Inno Setup .iss 설정 파일 작성 (URI Scheme 등록 포함)
- `updater.bat` 작성 (AC 종료 대기 → 인스톨러 실행 → AC 재실행)
- 빌드 스크립트: PyInstaller 빌드 → Inno Setup 컴파일 → Setup.exe 생성

**예상 배포 크기:** ~100-150MB (Python ~15MB + JRE ~50MB + SikuliX ~50MB + 리소스)

### Phase 1: 사용자 식별 (AIMS 웹 연동)

AC에 별도 로그인 화면을 만들지 않는다. **AIMS 웹 로그인이 곧 AC 인증이다.**

```
AIMS 웹 로그인 상태에서 "자동수집" 버튼 클릭
  → aims-ac://start?token=xxx&userId=yyy 호출
  → 인스톨러가 등록한 URI Scheme → AutoClicker.exe 실행
  → AC가 토큰으로 사용자 식별 (별도 로그인 불필요)

AIMS 웹 미로그인
  → "자동수집" 버튼 자체를 못 누름
  → AC 실행 불가
```

**보안 고려사항:**
- **토큰은 1회용 or 단기 만료 (5분)**: `aims-ac://start?token=xxx`의 토큰이 프로세스 커맨드라인에 노출됨. 긴 수명 토큰은 위험
- **단일 인스턴스**: AC가 이미 실행 중일 때 유저가 버튼을 또 누르면? named mutex로 중복 실행 방지, 기존 인스턴스에 포커스
- **브라우저 확인 대화상자**: Chrome이 "이 사이트에서 aims-ac를 열려고 합니다" 팝업 표시 — 정상 동작이지만 유저에게 안내 필요

**필요한 작업:**

| 코드베이스 | 작업 |
|-----------|------|
| AIMS 웹 (React) | "자동수집" 버튼 추가, 1회용 토큰 발급 후 URI Scheme 호출 |
| AIMS 서버 (Node) | `POST /api/ac/verify-token` API 신규, 1회용 토큰 발급/검증 |
| AC (Python) | URI 파라미터 파싱, 토큰 서버 검증, named mutex 단일 인스턴스 |

**사용자 식별이 되면 가능해지는 것:**

| 기능 | userId 활용 |
|------|------------|
| OCR 크레딧 차감 | userId로 차감 |
| 자동 업데이트 | userId별 버전 관리, 업데이트 이력 |
| 사용 통계 | userId별 집계 |
| 시범 테스트 관리 | 누가 테스트 중인지 파악 |

### Phase 2: 자동 업데이트 (사일런트 인스톨러)

AC가 새 버전을 감지하면 **유저 모르게** 업데이트한다. Inno Setup의 `/VERYSILENT` 옵션 활용.

```
AC 실행 → 서버에 현재 버전 보고
  → 서버: { latest: "0.2.0", minRequired: "0.1.5", installerUrl: "..." }

  [선택적 업데이트] 현재 >= minRequired, 최신 아님
    → 백그라운드로 인스톨러 다운로드
    → AC 종료 시점에 /VERYSILENT 실행 → 자동 덮어쓰기 → 다음 실행 시 새 버전
    → 유저는 아무것도 모름

  [강제 업데이트] 현재 < minRequired
    → "필수 업데이트가 있습니다. 잠시만 기다려주세요..."
    → 인스톨러 다운로드 → AC 종료 → /VERYSILENT 실행 → AC 재실행
    → 유저는 잠깐 기다리기만 함

  [최신 버전] → 정상 실행
```

**Inno Setup 사일런트 모드:**
```bash
# 유저에게 아무것도 보이지 않음, 같은 경로에 덮어쓰기
AIMS_AutoClicker_Setup.exe /VERYSILENT /SUPPRESSMSGBOXES /DIR="기존설치경로"
```

**왜 이 방식이 좋은가:**
- 유저가 업데이트를 인지하지 못함 (최상의 UX)
- AC에 파일 교체 로직 불필요 (Inno Setup이 처리)
- 실행 중 exe 교체 문제 없음 (AC 종료 후 인스톨러 실행)
- 서버 API 1개만 필요 (`GET /api/ac/latest-version`)
- ActiveX처럼 매끄럽되, 최초 설치만 명시적 동의 필요

**자기 교체 메커니즘 (updater.bat 패턴):**

AC는 실행 중에 자기 자신(exe)을 교체할 수 없다. Windows가 파일을 잠그기 때문.
해결: AC가 외부 스크립트를 띄우고 자신은 종료 → 스크립트가 인스톨러 실행 → AC 재실행.

```
updater.bat:
  @echo off
  timeout /t 2 /nobreak >nul                    ← AC 프로세스 종료 대기
  "%~dp0temp\AIMS_Setup.exe" /VERYSILENT /SUPPRESSMSGBOXES /DIR="%~dp0"
  start "" "%~dp0AutoClicker.exe"                ← 업데이트 완료 후 AC 재실행
  del "%~dp0temp\AIMS_Setup.exe"                 ← 인스톨러 정리
```

```
AC 내부 로직:
  1. 서버에서 새 버전 감지 → 인스톨러를 temp/ 폴더에 다운로드
  2. updater.bat을 detached 프로세스로 실행 (subprocess.Popen, CREATE_NEW_PROCESS_GROUP)
  3. AC 자신은 종료
  4. updater.bat이 2초 대기 → 인스톨러 사일런트 실행 → AC 재실행
```

**필요한 작업:**
- AC에 버전 체크 + 인스톨러 다운로드 로직 (~50줄)
- `updater.bat` 작성 (Phase 0 인스톨러에 포함)
- AC에서 updater.bat detached 실행 + 자기 종료 로직
- 서버 API: `GET /api/ac/latest-version` → `{ latest, minRequired, installerUrl }`

### Phase 3: 크레딧 관리

**왜 어려운가 — 단순하지 않은 이유들:**

| 문제 | 설명 |
|------|------|
| 차감 단위 | OCR 1회당? 고객 1명당? 초성 1개당? 세션 1회당? |
| 차감 시점 | 실시간 매 호출마다? 세션 끝에 일괄? |
| 실패/재시도 | OCR 실패해도 차감? 재시도 시 이중 차감? |
| 중간 소진 | 처리 중 크레딧 바닥나면? 즉시 중단? 현재 페이지까지 완료? |
| 네트워크 끊김 | AC는 로컬인데 차감 API 호출 실패하면? |
| 크래시 복구 | AC가 죽었을 때 이미 차감한 건? 아직 안 한 건? |
| 기존 크레딧 통합 | 임베딩 크레딧과 같은 풀? 별도 풀? |
| 선차감 vs 후차감 | 시작 전 예상량 미리 차감? 끝나고 실제량 차감? |

**시범 테스트 기간 전략: 로깅 우선**

```
[시범 테스트 기간]
AC 실행 → 사용자 인증 → 자동화 수행 → 사용량 서버에 기록만 (차감 안 함)
```

- 실제 사용 패턴 데이터 축적 (고객당 OCR 횟수, 세션당 평균 등)
- 데이터 기반으로 크레딧 정책을 현실적으로 설계
- 시범 테스트 유저에게 부담 없음
- 충분한 데이터 확보 후 → 차감 로직 구현 + 정책 확정
