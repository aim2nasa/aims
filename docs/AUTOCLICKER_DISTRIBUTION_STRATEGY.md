# AutoClicker 배포 및 수익화 전략

> **상태**: 토의 중 (2026-02-11)
> **목적**: AC를 AIMS 고객에게 어떻게 제공하고, 정당한 대가를 받을 것인가

---

## 1. 현황

### AutoClicker란?
- AIMS 플랫폼의 **데스크톱 자동화 도구**
- 보험사 웹사이트에서 고객 정보/PDF를 자동 수집
- Python + SikuliX 기반, Windows 전용
- OCR 기능 내장 (크레딧 소모)

### 현재 구조
```
설계사 PC에서 직접 실행
├── Python 런타임 필요
├── SikuliX (Java) 필요
├── AIMS 서버와 연동 (문서 업로드, 크레딧 차감)
└── OCR 사용 시 크레딧 소모
```

### 비용 구조
| 항목 | 비용 발생 주체 |
|------|---------------|
| AC 개발/유지보수 | 우리 (개발 인력) |
| OCR API 호출 | 우리 (서버 비용) |
| 서버 인프라 | 우리 (AIMS 서버) |
| 문서 임베딩 | 우리 (AI API 비용) |

→ **단순 무료 다운로드 = 비용만 발생, 수익 없음**

## 2. 과거 기술과의 비교

### ActiveX 시대 (2000년대)
- 브라우저에서 `.cab` 다운로드 → 자동 설치/실행
- 보안 문제로 사라짐
- 하지만 **"웹에서 데스크톱 도구를 제공"**하는 개념은 유효

### AC가 ActiveX와 유사한 점
| 공통점 | 설명 |
|--------|------|
| 데스크톱 실행 | 브라우저 밖에서 동작하는 네이티브 도구 |
| 웹 플랫폼 연동 | AIMS 웹 서비스와 데이터 주고받음 |
| 자동 배포 필요 | 비개발자가 쉽게 설치/업데이트 |
| 라이선스 관리 | 정당한 사용자만 사용 가능해야 함 |

### AC가 ActiveX와 다른 점
| 차이점 | 설명 |
|--------|------|
| 독립 실행 | 브라우저 플러그인이 아닌 독립 앱 |
| 크레딧 연동 | OCR/임베딩 사용량에 따른 과금 필요 |
| 보안 | 사용자 인증 + 라이선스 검증 필요 |

## 3. 배포 방식 검토

### 3-1. 패키징

| 방식 | 장점 | 단점 |
|------|------|------|
| **PyInstaller (.exe)** | 단일 파일, Python 불필요 | 용량 큼, AV 오탐 가능 |
| **PyInstaller (폴더)** | 실행 빠름, SikuliX 번들 가능 | 배포 크기 큼 |
| **Nuitka** | 네이티브 컴파일, 성능 우수 | 빌드 복잡 |
| **MSI 인스톨러** | Windows 표준, 깔끔한 설치/제거 | 관리자 권한 필요할 수 있음 |

**권장**: PyInstaller (폴더) + SikuliX/JRE 번들 → ZIP 배포

### 3-2. 배포 채널

| 방식 | 장점 | 단점 |
|------|------|------|
| **AIMS 웹에서 다운로드** | 중앙 관리, 버전 통제 | 수동 업데이트 |
| **앱 내 자동 업데이트** | 항상 최신, 유저 부담 없음 | 구현 복잡 |
| **AIMS 웹 + 자동 업데이트** | 최초만 수동, 이후 자동 | 권장 조합 |

## 4. 수익화 모델

### 핵심 원칙
> **"AC는 AIMS의 부가가치 도구이며, 무료 배포 대상이 아니다."**

### 4-1. 라이선스 + 크레딧 하이브리드 모델

```
┌─────────────────────────────────────────────┐
│              AIMS 구독 플랜                    │
│                                              │
│  Basic     ─── AIMS 웹만 사용                 │
│  Pro       ─── AIMS 웹 + AC 사용권 + 월 N크레딧 │
│  Enterprise ── 전체 기능 + 대량 크레딧           │
└─────────────────────────────────────────────┘
```

| 과금 요소 | 방식 |
|-----------|------|
| AC 사용권 | 구독 플랜에 포함 (Pro 이상) |
| OCR 크레딧 | 사용량 기반 차감 |
| 문서 임베딩 크레딧 | 사용량 기반 차감 (기존 체계) |
| 초과 크레딧 | 추가 구매 가능 |

### 4-2. AC 실행 시 인증 흐름

```
AC 실행
  ├── 1. AIMS 서버에 라이선스 검증 요청
  │     POST /api/ac/verify-license
  │     { userId, machineId, acVersion }
  │
  ├── 2. 서버 응답
  │     { authorized: true, plan: "pro", credits: 150, minVersion: "0.2.0" }
  │
  ├── 3. 버전 체크
  │     현재 버전 < minVersion → 강제 업데이트 안내
  │
  ├── 4. 크레딧 체크
  │     credits <= 0 → "크레딧이 부족합니다" 안내
  │
  └── 5. 정상 실행
```

### 4-3. OCR 크레딧 차감

```
OCR 수행 시
  ├── 로컬에서 OCR 실행 (Tesseract 등)
  │     → 서버 비용 없음 → 크레딧 차감 소량 or 무료?
  │
  └── 서버 OCR API 호출
        → 서버 비용 발생 → 크레딧 차감
        → POST /api/ac/deduct-credit { userId, type: "ocr", count: 1 }
```

**토의 필요**: 로컬 OCR vs 서버 OCR에 따라 크레딧 정책이 달라짐

### 4-4. 크레딧 정책 (초안)

| 작업 | 크레딧 소모 | 비고 |
|------|------------|------|
| AC 실행 | 0 | 실행 자체는 무료 |
| 고객 1명 처리 | 0 | 자동화 자체는 무료 |
| OCR 1회 | ? | 로컬/서버 여부에 따라 결정 |
| PDF 다운로드 | 0 | 보험사 원본 다운로드 |
| 문서 임베딩 | 기존 정책 | AC와 무관, 업로드 시 차감 |

## 5. 기술 아키텍처 (배포 + 인증)

### 5-1. 전체 흐름

```
[AIMS 웹]                    [AC (설계사 PC)]              [AIMS 서버]
    │                              │                           │
    ├── AC 다운로드 ──────────────►│                           │
    │                              │                           │
    │                              ├── 라이선스 검증 ──────────►│
    │                              │◄── 인증 결과 ─────────────┤
    │                              │                           │
    │                              ├── 자동화 실행              │
    │                              │   (클릭, OCR, PDF 수집)    │
    │                              │                           │
    │                              ├── OCR 크레딧 차감 ────────►│
    │                              │                           │
    │                              ├── 문서 업로드 ─────────────►│
    │                              │                           │
    │                              ├── 버전 체크 ──────────────►│
    │                              │◄── 업데이트 URL ──────────┤
    │                              │                           │
```

### 5-2. 서버 API (신규 필요)

| 엔드포인트 | 용도 |
|-----------|------|
| `POST /api/ac/verify-license` | AC 실행 시 라이선스 + 크레딧 + 최소버전 검증 |
| `POST /api/ac/deduct-credit` | OCR 등 크레딧 차감 |
| `GET /api/ac/latest-version` | 최신 버전 정보 + 다운로드 URL |
| `POST /api/ac/usage-log` | 사용 통계 기록 (선택) |

### 5-3. AC 측 구현 필요 사항

| 기능 | 설명 |
|------|------|
| 로그인/인증 | AIMS 계정으로 로그인 (토큰 저장) |
| 라이선스 검증 | 실행 시 서버 호출, 오프라인 시 캐시된 토큰 사용 |
| 크레딧 표시 | GUI에 잔여 크레딧 표시 |
| 자동 업데이트 | 새 버전 감지 시 다운로드 + 교체 |
| 기기 바인딩 | machineId로 동시 사용 기기 수 제한 (선택) |

## 6. 유저 경험 원칙

### 최상위 철학
> **다운로드조차 하고 싶지 않다. 하지만 데스크톱 자동화라 어쩔 수 없이 하는 것이다.**
> **따라서 그 "어쩔 수 없는" 한 번을 최대한 매끄럽게 만들어야 한다.**

### 이상적 UX 계층 (위에서 아래로 갈수록 현실적)

| 등급 | 유저 경험 | 기술 | 가능 여부 |
|------|----------|------|----------|
| S | AIMS 웹에서 버튼 클릭 → 바로 동작 | 브라우저 자동화 | ❌ 데스크톱 GUI 제어 불가 |
| A | AIMS 웹에서 클릭 → AC 자동 실행 | Custom URI Scheme (`aims-ac://start`) | ✅ 가능 — 최초 1회 설치 후 |
| B | AIMS 웹에서 클릭 → 다운로드 → 더블클릭 → 끝 | 자동 설치 프로그램 | ✅ 가능 |
| C | 다운로드 → 압축 해제 → 더블클릭 | ZIP 배포 | ✅ 가능 (최소 구현) |

### 권장: A등급 — Custom URI Scheme

```
[AIMS 웹 — "자동수집 시작" 버튼 클릭]
         │
         ▼
  브라우저가 aims-ac://start?token=xxx&chosung=ㄱ 호출
         │
         ▼
  [AC 이미 설치됨?]──Yes──► AC 자동 실행 (설정 자동 적용)
         │
         No
         ▼
  "AutoClicker를 설치해주세요" 안내 + 원클릭 설치 버튼
         │
         ▼
  설치 프로그램 다운로드 → 실행 → 자동 설치 → URI Scheme 등록
         │
         ▼
  다음부터는 AIMS 웹에서 버튼만 클릭하면 AC 자동 실행
```

**현실 사례 (유저가 이미 익숙한 패턴):**
- Zoom: 웹에서 "회의 참가" → `zoommtg://` → Zoom 앱 자동 실행
- Slack: 웹에서 "Open in Slack" → `slack://` → Slack 앱 자동 실행
- VS Code: 웹에서 "Open in VS Code" → `vscode://` → VS Code 자동 실행

**AC도 동일:**
- AIMS 웹에서 "자동수집" → `aims-ac://` → AC 자동 실행 + 설정 자동 전달

### 유저가 하는 일
1. **최초 1회**: AIMS 웹에서 "AutoClicker 설치" 클릭 → 자동 설치
2. **이후**: AIMS 웹에서 "자동수집 시작" 클릭 → AC 자동 실행
3. **끝.**

### 유저가 절대 하지 않는 일
- ❌ Python, Java, SikuliX 등의 존재를 인식
- ❌ 환경변수, 명령 프롬프트, pip install
- ❌ 수동 업데이트 (자동 업데이트)
- ❌ AC 실행 파일을 직접 찾아서 실행

### 자동 업데이트 전략

```
AC 실행 시
  ├── AIMS 서버에 최신 버전 확인
  │     GET /api/ac/latest-version → { version: "0.2.0", url: "..." }
  │
  ├── 현재 버전 < 최신 버전?
  │     ├── Yes (선택적) → "업데이트 가능" 알림, 백그라운드 다운로드
  │     ├── Yes (강제)   → "필수 업데이트" → 자동 다운로드 + 재시작
  │     └── No           → 정상 실행
  │
  └── 업데이트 적용
        ├── 새 exe 다운로드 → 임시 폴더
        ├── AC 종료 → 교체 스크립트 실행 → 새 AC 실행
        └── 유저는 "잠시 업데이트 중..." 메시지만 봄
```

### 배포 패키지 구성 (유저에게 보이지 않는 내부 구조)

```
C:\Users\{user}\AppData\Local\AIMS\AutoClicker\
├── AutoClicker.exe          ← 실행 파일
├── runtime/
│   ├── python/              ← 임베디드 Python (~15MB, 시스템 설치 아님)
│   ├── jre/                 ← 임베디드 JRE (~50MB, 시스템 설치 아님)
│   └── sikulix/             ← SikuliX JAR
├── resources/               ← 이미지, 설정 등
└── version.json             ← 버전 정보
```

설치 경로가 `AppData\Local`이므로 **관리자 권한 불필요**.
