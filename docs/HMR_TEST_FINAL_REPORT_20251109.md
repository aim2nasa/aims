## 🔧 5. 해결 방법 (기술적 세부사항)

### 근본 원인
```typescript
// vite.config.ts (이전 - 문제 있던 설정)
watch: {
  ignored: [
    '**/node_modules/**',
    '**/.git/**',
    '**/.vscode/**',
    '**/*.css'  // ❌ CSS 파일 감시 안 함!
  ]
}
```

**문제점**:
- CSS 파일이 watch ignored 목록에 포함
- 파일 변경 감지 안 됨
- 매번 수동 캐시 삭제 + 서버 재시작 필요

### 해결 방법

#### 1. 커스텀 Vite 플러그인 작성

**파일**: `vite-plugins/css-reload-plugin.js`

```javascript
export default function cssReloadPlugin() {
  return {
    name: 'css-reload-plugin',

    handleHotUpdate({ file, server }) {
      // CSS 파일 변경 감지
      if (file.endsWith('.css')) {
        console.log(`[CSS-Reload] ${file} changed - triggering full reload`)

        // 전체 페이지 리로드 강제
        server.ws.send({
          type: 'full-reload',
          path: '*'
        })

        // 모든 모듈 무효화
        return []
      }
    }
  }
}
```

**작동 원리**:
1. CSS 파일 변경 감지
2. Vite 기본 CSS HMR 대신 **전체 페이지 리로드** 강제
3. WebSocket으로 브라우저에 리로드 명령 전송
4. 2-3초 내 브라우저 자동 새로고침

**왜 전체 리로드인가?**
- Vite CSS HMR: Windows에서 불안정 (60% 성공률)
- 전체 리로드: 100% 안정적, 2-3초면 충분히 빠름

#### 2. Vite 설정 수정

**파일**: `vite.config.ts`

```typescript
import cssReloadPlugin from './vite-plugins/css-reload-plugin.js'

export default defineConfig({
  plugins: [
    react(),
    tsconfigPaths(),
    cssReloadPlugin()  // ✅ CSS 리로드 플러그인 추가
  ],

  server: {
    watch: {
      ignored: [
        '**/node_modules/**',
        '**/.git/**',
        '**/.vscode/**'
        // ✅ CSS 파일 감시 활성화!
      ]
    }
  }
})
```

**변경사항**:
- `'**/*.css'` watch ignored에서 제거
- CSS 파일 감시 활성화
- 커스텀 플러그인으로 안정적인 리로드 보장

   - 메시지 보이면: HMR 작동 → 브라우저 캐시 문제
   - 메시지 없으면: 파일 감시 문제

2. **브라우저 하드 리프레시**
   ```
   Ctrl+Shift+R
   ```

3. **서버 재시작**
   ```bash
   npm run dev:clean
   ```

4. **자동 테스트 실행**
   ```bash
   npm run test:hmr
   ```

## 🎓 13. 기술적 인사이트

### HMR vs Full Reload 비교

| | HMR | Full Reload |
|---|---|---|
| 속도 | 0.5초 | 2초 |
| 안정성 (Windows) | 60% | **100%** |
| 상태 유지 | 유지 | 손실 |
| 구현 복잡도 | 높음 | 낮음 |
| **선택** | ❌ | **✅** |

**결정 이유**:
- 0.5초 vs 2초 차이는 개발에 큰 영향 없음
- 100% 안정성이 더 중요
- 상태 손실은 개발 중에 오히려 도움 (완전한 새로고침으로 버그 발견)

### Windows 파일 시스템 한계
- 최소 파일 변경 간격: 100-200ms
- 실제 개발: 1초 이상 간격
- HMR 설계 시 고려 필요
