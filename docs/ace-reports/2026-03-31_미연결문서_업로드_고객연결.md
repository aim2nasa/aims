# ACE 보고서: 미연결 문서 업로드 및 고객 연결

## 전체 흐름

사용자 기획 → AC 7건 작성·승인 → 테스트 35건 선행 생성 → Alex 구현 → Gini 검수(1차 FAIL → 수정) → 배포 → E2E → Jude 감사(1차 FAIL → 수정 → PASS)

## 결정의 맥락

### 미연결 필터: customerLink=unlinked vs customerIdFilter='null'

초기 구현은 `DocumentStatusProvider`의 `customerIdFilter` prop에 `'null'` 문자열을 전달하는 방식이었다.
Gini 검수에서 거부됨 — 백엔드 `documents-routes.js:1264`에서 `ObjectId.isValid('null')` = false로 무시되므로 **실제로 동작하지 않는 코드**였다.

백엔드에 이미 `customerLink=unlinked` 파라미터가 구현되어 있었고(documents-routes.js:1259), `DocumentStatusService.getRecentDocuments`도 5번째 인자로 `customerLink`를 받고 있었다. `DocumentStatusProvider`에 `customerLinkFilter` prop을 추가하여 올바른 API 경로를 사용하도록 수정.

### AC#5 테스트: 배열 vs 단건 API 호출

초기 테스트는 `{ documentIds: string[] }` 배열 payload를 검증했으나, 실제 구현은 `DocumentService.linkDocumentToCustomer`를 문서 수만큼 반복 호출하는 방식(`DocumentLinkModal.tsx`의 for loop). Jude 감사에서 발견하여 수정.

## 함정/주의점

1. **DocumentStatusProvider의 customerIdFilter는 ObjectId 유효성 검사를 거침** — 임의의 문자열을 넣으면 무시됨. 미연결 필터는 반드시 `customerLinkFilter`를 사용해야 함.

2. **미연결 필터와 고객 필터는 상호 배타** — 동시 활성화되면 백엔드에서 `customerLink=unlinked` + `customerId=특정ID`가 충돌. 프론트엔드에서 상호 배타 로직으로 방어.

3. **uploadService.ts 변경 불필요** — `if (customerId)` 조건이 이미 있어 customerId가 undefined/null이면 FormData에 추가하지 않음. 빈 문자열 전송 방지가 핵심.

## 당시 상황/제약

- SSH 키가 환경에 없어 배포 중단 → 사용자가 SSH 연결 후 재진행
- Node.js가 미설치 상태 → winget으로 Node.js 22 설치 후 진행
- 백엔드 변경 불필요 (customerId Optional, customerLink=unlinked, POST 연결 API 모두 기존 구현)
