# 기획안: 미연결 문서 업로드 및 고객 연결 기능

- **작성일**: 2026-03-31
- **상태**: 기획 확정, 구현 대기 (가벼운 방식 → 불만족 시 ACE 전환)
- **프로세스**: Alex 구현 → Gini 검수 → 배포 → 사용자 검수 (ACE 미적용)
- **폴백**: 결과 불만족 시 ACE 풀 프로세스로 전환

---

## 1. 배경 및 니즈

### 현재 상태
문서 업로드 시 고객을 **반드시** 먼저 선택해야 한다. 고객·계약·문서 등록 → 일반 문서 탭에서 "고객 선택" 버튼을 누르고 고객을 지정해야만 파일 업로드 UI가 활성화된다.

### 발견된 니즈
사용자가 문서를 **일단 올린 뒤**, 파이프라인이 별칭(displayName)을 부여하고 문서 유형을 파악한 후에 적절한 고객에게 할당하고 싶은 경우가 있다. 예를 들어:
- 여러 고객의 서류를 한꺼번에 스캔했지만, 어떤 고객 것인지 파일명만으로는 구분이 안 됨
- 먼저 올려서 OCR/분류를 거친 뒤, 별칭과 문서 유형을 보고 고객을 지정하는 게 더 효율적

### 적용 범위
- **고객·계약·문서 등록 → 일반 문서** 탭에만 해당
- Annual Report, Customer Review 탭은 해당 없음 (AR/CRS는 문서 자체에 고객명이 있어 자동 연결)

## 2. 기능 정의

### 2-1. 고객 미지정 업로드 허용

**현재 플로우:**
```
일반 문서 탭 진입
  → "고객 선택 (필수)" 표시
  → 고객 선택해야 파일 업로드 영역 활성화
  → 업로드된 문서는 선택한 고객에 자동 연결
```

**변경 후 플로우:**
```
일반 문서 탭 진입
  → 고객 미선택 상태에서도 파일 업로드 영역이 활성화됨
  → "고객 미선택" 안내 표시 (예: "고객 미지정으로 업로드됩니다. 나중에 전체 문서 보기에서 고객을 연결할 수 있습니다.")
  → 업로드된 문서는 customerId 없이 저장
  → 고객을 선택한 경우에는 기존과 동일하게 해당 고객에 자동 연결
```

### 2-2. 전체 문서 보기 — 미연결 문서 필터

**필터 성격:**
- 처리 상태 필터(전체/처리중/완료/에러)와 **완전히 독립적**인 별도 필터
- 처리 상태 = "문서가 어떻게 처리되었는가" (파이프라인 상태)
- 미연결 필터 = "이 문서가 고객과 연결되었는가" (비즈니스 관계)
- 두 필터는 AND 조합 가능 (예: "미연결 + 완료" = 처리는 끝났지만 고객 미지정인 문서)

**구현:**
- 기존 API가 이미 `customerId=null` 파라미터를 지원 (documents-routes.js:404-427)
- 프론트엔드에 "미연결 문서만" 토글/필터 UI 추가

### 2-3. 미연결 문서 → 고객 연결

**연결 방식:**
- 전체 문서 보기에서 미연결 문서를 필터링하여 조회
- 문서 1개 또는 여러 개를 선택 (체크박스)
- 기존 "고객 선택" 모달(CustomerSelectorModal)을 열어 고객 지정
- 기존 `POST /api/customers/:id/documents` API를 호출하여 연결
- 연결 시 Qdrant customer_id 자동 동기화 (기존 로직)

**기존 인프라 활용:**
- `CustomerSelectorModal` — 이미 구현된 고객 선택 모달 (스크린샷 참조)
- `POST /api/customers/:id/documents` — 고객-문서 연결 API (Qdrant 동기화 포함)
- `syncQdrantCustomerRelation()` — Qdrant payload 업데이트 함수

## 3. 기술 조사 결과

### 3-1. DB 스키마 — customerId는 Optional

```python
# document_pipeline/routers/doc_prep_main.py:218
customerId: Optional[str] = Form(None)

# doc_prep_main.py:454-456
if customerId:
    doc_data["customerId"] = ObjectId(customerId)
# → customerId 없으면 필드 자체 미포함
```

- MongoDB files 컬렉션에서 customerId는 **Optional** (필드 미포함 또는 null 가능)
- 백엔드/파이프라인 모두 customerId 없이 정상 동작 확인됨

### 3-2. 파이프라인 — customer_id 없어도 전체 정상 처리

| 단계 | customerId 없을 때 | 비고 |
|------|-------------------|------|
| 텍스트 추출 | 정상 | customerId 무관 |
| OCR | 정상 | 고객명 조회 실패해도 계속 진행 |
| 요약/분류 | 정상 | 22개 유형 분류에 customerId 불필요 |
| AR/CRS 감지 | 정상 | PDF 텍스트 기반 감지 (파일명 금지) |
| displayName 생성 | 정상 | AI 분류 결과 기반 |
| 임베딩 | 정상 | customer_id=null로 Qdrant 저장 |

### 3-3. 임베딩 전략 — 미리 임베딩 + 나중에 payload 업데이트 (확정)

**결정: 방안 A — 미리 임베딩, 고객 연결 시 payload만 업데이트**

```
[업로드 (고객 미지정)]
    ↓
[OCR → 분류 → 임베딩] — customer_id=null로 Qdrant에 저장
    ↓
(나중에 사용자가 고객 연결)
    ↓
[syncQdrantCustomerRelation(docId, customerId)]
    → setPayload({ customer_id: customerId }) — 모든 청크에 customer_id 설정
    → 즉시 AI 검색에 반영
```

**이 방안을 선택한 이유:**

1. **기존 인프라 완비**: `syncQdrantCustomerRelation()` 함수가 이미 존재하고 고객-문서 연결 API에서 호출 중. 추가 개발 불필요
2. **간단하고 직관적**: 임베딩은 무조건 수행, 고객 정보만 나중에 업데이트. 복잡한 상태 관리 불필요
3. **지연 없음**: 고객 연결 즉시 검색 가능. 임베딩 대기 시간 없음
4. **안전성**: customer_id=null인 벡터는 AI 검색 시 customer_id 필터에 걸리지 않으므로 다른 사용자/고객에게 노출되지 않음. 고객 연결 후 자동으로 검색에 포함

**기각된 방안:**

방안 B (고객 연결 전까지 임베딩 보류)는 기각:
- 새로운 상태(embed_waiting_customer 등) 관리 필요
- 고객 연결 시 임베딩을 트리거하는 별도 로직 필요
- 고객 연결 후 임베딩까지 1분+ 지연 (크론 주기)
- 복잡도 대비 이점 없음

### 3-4. AR/CRS 문서 — 미연결 불가

AR/CRS 문서는 PDF 텍스트에서 고객명을 감지하여 자동으로 고객에 연결된다.
따라서 AR/CRS는 **절대로 미연결 상태가 될 수 없다**.
이 기능은 AR/CRS 이외의 일반 문서에만 해당한다.

### 3-5. 미연결 문서 API 필터 — 이미 구현됨

```javascript
// documents-routes.js:411-417
if (customerIdFilter === 'null' || customerIdFilter === '') {
    query.$or = [
        { customerId: null },
        { customerId: { $exists: false } }
    ];
}
```

- `GET /api/documents/status?customerId=null` → 미연결 문서만 조회
- 프론트엔드에서 이 파라미터를 전달하는 UI만 추가하면 됨

### 3-6. 고객-문서 연결 API — 이미 구현됨

```
POST /api/customers/:id/documents
```

처리 순서:
1. 소유권 검증 (고객 + 문서 모두)
2. 중복 file_hash 체크
3. customers.documents[]에 push + files.customerId 설정
4. **Qdrant customer_id 동기화** (`syncQdrantCustomerRelation`)
5. AR 문서면 ar_parse_queue에 등록
6. SSE 알림 발송

## 5. 사용자 시나리오 (End-to-End)

### 시나리오 1: 기본 — 고객 미지정 업로드 후 나중에 연결

```
1. 설계사가 여러 고객의 서류를 한꺼번에 스캔함
2. 고객·계약·문서 등록 → 일반 문서 탭 진입
3. 고객을 선택하지 않고 파일 10개를 드래그 앤 드롭으로 업로드
4. 안내: "고객 미지정으로 업로드됩니다"
5. 파이프라인이 OCR/분류/별칭 부여/임베딩을 자동 수행 (customer_id=null)
6. 시간이 지나고, 설계사가 전체 문서 보기 진입
7. "미연결" 필터를 켬 → 고객 미지정 문서 10개가 표시됨
8. 별칭과 문서 유형을 보고 "이춘애" 고객 문서 3개를 체크박스로 선택
9. "고객 연결" 버튼 클릭 → 고객 선택 모달 → "이춘애" 선택 → 연결
10. 3개 문서가 이춘애에게 연결됨 (Qdrant customer_id도 자동 업데이트)
11. AI 검색에서 "이춘애"로 검색하면 3개 문서가 즉시 나옴
12. 나머지 7개는 여전히 미연결 상태 → 나중에 다른 고객에 연결
```

### 시나리오 2: 기존 방식 — 고객 선택 후 업로드 (변경 없음)

```
1. 설계사가 특정 고객의 서류만 스캔함
2. 고객·계약·문서 등록 → 일반 문서 탭 → "고객 선택" → "오광진" 선택
3. 파일 업로드 → 오광진에게 자동 연결
4. 기존과 완전히 동일한 동작
```

### 시나리오 3: 혼합 — 일부는 고객 지정, 일부는 미지정

```
1. "이현희" 고객 문서 5개를 먼저 고객 선택 후 업로드 (기존 방식)
2. 누구 것인지 모르는 문서 3개를 고객 미선택으로 업로드
3. 나중에 전체 문서 보기에서 미연결 3개를 확인 후 적절한 고객에 연결
```

## 6. 엣지 케이스 및 미묘한 부분

### 6-1. 파이프라인 관련

| # | 상황 | 처리 | 근거 |
|---|------|------|------|
| 1 | 고객 미지정 문서의 OCR 시 고객명 조회 | customerId 없으므로 고객명 조회 스킵. OCR 자체는 정상 수행 | ocr_worker.py에서 customerId가 없으면 고객명 없이 진행 |
| 2 | 고객 미지정 문서의 displayName 생성 | AI 분류 결과 기반으로 정상 생성. AR/CRS 패턴이 아닌 이상 고객명 불필요 | openai_service.py의 summarize_text() |
| 3 | 고객 미지정 문서가 AR/CRS로 감지됨 | PDF 텍스트에서 고객명 추출 → relatedCustomerId 설정. **customerId(소유권)는 여전히 null** — 별도 연결 필요 | doc_prep_main.py:743-800 |
| 4 | 임베딩 시 customer_id=null | Qdrant에 정상 저장. customer_id 필터 검색에서 제외됨 (의도된 동작) | full_pipeline.py:562 |

### 6-2. 고객 연결 관련

| # | 상황 | 처리 | 근거 |
|---|------|------|------|
| 5 | 미연결 문서를 연결할 때 중복 파일 | 기존 file_hash 중복 체크로 409 DUPLICATE_FILE 반환 | customers-routes.js 연결 API |
| 6 | 여러 문서를 한번에 1명의 고객에 연결 | 순차적으로 POST /api/customers/:id/documents 호출 (1건씩) | 프론트엔드에서 Promise.all 또는 순차 호출 |
| 7 | 연결 후 Qdrant 동기화 실패 | syncQdrantCustomerRelation이 실패해도 MongoDB 연결은 유지. Qdrant는 비동기 복구 가능 | 현재 에러 로깅만 수행, 트랜잭션 아님 |
| 8 | 고객 연결 후 해제 → 다시 미연결 | 기존 DELETE API로 처리. Qdrant에서 customer_id 제거. 다시 미연결 상태로 복원 | syncQdrantCustomerRelation(docId, null) |

### 6-3. 이중 저장 (고객-문서 관계) 주의

고객-문서 관계는 **이중 저장** 구조 (customer-skill 참조):
- `customers.documents[]` — 고객에 문서 ID 배열
- `files.customerId` — 문서에 고객 ID 참조
- Qdrant 임베딩에도 `customer_id` payload

미연결 문서의 경우:
- `files.customerId` = 없음 (필드 미존재 또는 null)
- 어떤 `customers.documents[]`에도 포함되지 않음
- Qdrant `customer_id` = null

고객 연결 시 **기존 POST API가 3곳을 모두 업데이트**하므로 추가 처리 불필요.

### 6-4. 소유자 격리

미연결 문서도 `ownerId`는 반드시 설정됨 (업로드한 설계사의 ID).
따라서 다른 설계사의 미연결 문서는 절대 보이지 않음.
전체 문서 보기에서 `ownerId` 필터가 항상 적용되므로 격리 보장.

### 6-5. 고객별 문서함과의 관계

미연결 문서는 고객별 문서함에 **표시되지 않는다** (올바른 동작).
고객별 문서함은 특정 고객에 연결된 문서만 보여주는 뷰이므로,
customerId가 없는 문서는 자연스럽게 제외된다.
미연결 문서를 확인하는 유일한 경로는 **전체 문서 보기**이다.

## 8. 범위 밖 (하지 않는 것)

- 배치 업로드(문서 일괄등록)에서 고객 미지정 허용 — 배치는 폴더-고객 매핑이 본질
- AR/CRS 탭에서 고객 미지정 허용 — AR/CRS는 문서 내 고객명으로 자동 감지/연결
- 미연결 문서 자동 고객 추천 (AI 기반) — 향후 별도 기능으로 검토 가능
- 고객별 문서함에서 미연결 문서 표시 — 고객별 문서함은 특정 고객의 문서만 표시하는 뷰
- 미연결 문서 일괄 삭제 — 기존 개별 삭제 기능으로 충분
- 문서 업로드 시 고객 자동 추천 — 범위 확대, 별도 기획 필요
