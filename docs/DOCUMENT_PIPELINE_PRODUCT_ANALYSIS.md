## 1. AIMS 파이프라인이 가진 것

현재 파이프라인의 핵심 역량을 객관적으로 정리:

| 기능 | 범용성 | 기술 |
|------|--------|------|
| 업로드 큐 + 재시도/자가복구 | 100% 범용 | MongoDB, asyncio |
| 메타데이터 추출 (PDF/이미지/Office) | 100% 범용 | PyMuPDF, openpyxl, python-docx |
| OCR (Upstage API) | 100% 범용 | REST API, 교체 가능 |
| AI 요약 + 태깅 | 100% 범용 | OpenAI GPT-4o-mini |
| 임베딩 + 벡터 검색 | 95% 범용 | OpenAI embedding, Qdrant |
| 크레딧 기반 과금 | 100% 범용 | 자체 구현 |
| 실시간 진행률 (SSE) | 100% 범용 | webhook |
| Office → PDF 변환 | 100% 범용 | LibreOffice headless |
| PDF 썸네일/메타 수정 | 100% 범용 | PyMuPDF, Pillow |
| AR/CRS 테이블 파싱 | 50% (도메인 특화) | pdfplumber |

**핵심 발견**: 파이프라인의 **약 95%가 이미 도메인 비종속적**. 보험 특화 부분은 AR/CRS 파서뿐이며, 이것도 pdfplumber 테이블 추출이라는 범용 기반 위에 구축되어 있음.

### 파이프라인 데이터 흐름

```
[1] Upload → document_pipeline (FastAPI) → MongoDB 큐 → Worker
     ├─ 파일 저장 (/data/files)
     ├─ 메타데이터 추출 (PyMuPDF, exifread, openpyxl, python-docx)
     └─ 크레딧 체크
          ├─ 충분 → OCR 큐 진입
          └─ 부족 → status: "credit_pending" (파일은 이미 저장됨)

[2] OCR → Upstage API → full_text 추출
     └─ Redis stream 큐 (비동기)

[3] 요약 → OpenAI GPT-4o-mini → summary + tags

[4] 임베딩 (Cron: 매 1분)
     └─ full_pipeline.py → 텍스트 청킹 (1500자, 150 overlap) → OpenAI embedding → Qdrant 저장

[5] 벡터 검색 → Qdrant query + GPT 리랭킹 → 하이브리드 검색 (시맨틱 + 키워드)

[6] 도메인 파싱 (AR/CRS) → pdfplumber 테이블 추출 + 자가복구 스캐너
```

### 수익 모델 (이미 구현된 크레딧 시스템)

```
OCR:       1 페이지 = 2 크레딧
임베딩:    1K 토큰 = 0.5 크레딧
요약:      ~0.5 크레딧/문서
RAG 검색:  ~1-2 크레딧/쿼리
```

**마진 분석** (가정: $49/월 스탠다드 플랜, 2000 크레딧):
- 고객 결제: $49/월
- COGS: ~$11.60 (OCR + 임베딩 + 요약 API 비용)
- **총 마진: ~76%** (건강한 SaaS 마진)

## 2. 시장 현황 (2025-2026)

### IDP(Intelligent Document Processing) 시장 규모

| 리서치 기관 | 2025년 추정 | 2030+ 전망 | CAGR |
|------------|------------|-----------|------|
| Research Nester | $3.0B | $66.68B (2032) | 33.4% |
| Fortune Business Insights | $10.57B | - | 30.1% |
| Precedence Research | $3.22B | $43.92B (2034) | 33.68% |
| MarketsandMarkets (Document AI) | $14.66B | $27.62B (2030) | - |

- Fortune 250 기업 중 **63%가 이미 IDP 솔루션 도입**
- 금융 부문은 **71% 도입률**
- 매우 빠른 성장 시장이나, 이미 레드오션화 진행 중

### 2025-2026 핵심 트렌드

1. **RAG → "Context Engine"**: 단순 검색 보강 → 지식 런타임 플랫폼으로 진화
2. **멀티모달 처리**: 텍스트 + 이미지 + 표 + 차트 통합 추출
3. **Agentic 자가교정**: 멀티패스 시스템 (OCR → VLM → 에이전트 검증) 표준화
4. **하이브리드 검색**: dense + sparse 검색 조합이 단일 방법 대비 일관적 우위
5. **VLM 선택적 사용**: 비용/속도 최적화 위해 복잡한 문서에만 Vision LM 적용

## 4. 강점과 약점 (솔직한 분석)

### 강점 (진짜 차별점이 될 수 있는 것)

**1. "Upload Now, Process Later" 크레딧 모델**
- 크레딧 부족해도 파일 즉시 저장, 충전 후 자동 처리
- Reducto 등 경쟁사에는 없는 UX 혁신
- SaaS 전환율 향상에 직접 기여하는 설계

**2. Self-healing 아키텍처**
- 고아 문서 자동 탐지 + 복구, stale job 정리
- 프로덕션 환경에서 검증된 안정성
- 대부분의 경쟁사가 수동 재처리에 의존

**3. End-to-End 파이프라인**
- 업로드 → OCR → 요약 → 임베딩 → 벡터 검색 → RAG까지 원스톱
- Unstructured.io는 파싱까지만, LlamaIndex는 인덱싱부터 — AIMS는 전 구간 커버

**4. 크레딧 과금 시스템 (이미 구현됨)**
- OCR 2크레딧/페이지, 임베딩 0.5크레딧/1K토큰 등 세분화된 과금
- SaaS 비즈니스 모델의 핵심이 이미 존재

### 약점 (솔직하게)

**1. 기술적 차별화 부족**
- OCR은 Upstage API 래핑, 임베딩은 OpenAI 래핑, 벡터DB는 Qdrant
- **핵심 AI 기술이 전부 외부 의존** — 경쟁사도 같은 API를 사용 가능
- Docling(97.9% 정확도), Reducto(멀티패스 VLM) 같은 자체 모델이 없음

**2. 스케일링 한계**
- 로컬 파일시스템, 크론 기반 배치 처리(1분 주기), MongoDB를 큐로 사용
- 엔터프라이즈 수준(월 수백만 페이지)으로 가려면 아키텍처 전면 개편 필요

**3. 단일 언어/지역**
- 현재 한국어 보험 문서에 최적화
- Rossum Aurora(276개 언어), Google Doc AI(100+개 언어)와 비교 불가

**4. 팀 규모와 자금**
- Reducto: Series B $75M (a16z), Unstructured.io: $40M+
- 문서 AI 시장은 자본 집약적 — 마케팅, 세일즈, 인프라 비용이 큼

## 5. 전략 옵션 평가

### Option A: 범용 Document AI SaaS로 독립 ❌ (비추천)

| 이유 | 상세 |
|------|------|
| 레드오션 | 이미 수십 개 플레이어가 수억 달러 투자로 경쟁 중 |
| 자체 기술 없음 | OpenAI/Upstage API 래핑은 moat(해자)가 아님 |
| GTM 비용 | B2B SaaS 마케팅/세일즈에 연 수억 원 필요 |
| 차별화 어려움 | "우리도 OCR+임베딩+RAG 됩니다"는 2026년에 commodity |

### Option B: 버티컬 SaaS (보험/금융 특화) ⚡ (검토 가치 있음)

| 강점 | 상세 |
|------|------|
| 도메인 전문성 | AR/CRS 파서, 보험 문서 구조 이해가 진짜 moat |
| 니치 마켓 | 보험 설계사/대리점 시장은 빅테크가 관심 없는 영역 |
| 검증된 프로덕트 | AIMS 자체가 PMF(Product-Market Fit) 검증 도구 |
| 작은 팀으로 가능 | 좁은 시장에서는 소규모 팀이 유리 |

**현실적 제약:**
- 한국 보험 설계사 시장 규모가 SaaS로 의미 있는 매출을 만들기 어려울 수 있음
- 보험사 자체 시스템과의 경쟁

### Option C: 오픈소스 + 매니지드 서비스 🔄 (흥미로운 전략)

Unstructured.io 모델 참고:
- 파이프라인 코어를 오픈소스로 공개
- 매니지드 클라우드 버전으로 수익화
- 커뮤니티가 도메인 파서를 기여 (법률, 의료, 부동산 등)

**장점**: 마케팅 비용 절감, 개발자 커뮤니티 확보
**단점**: 오픈소스 커뮤니티 구축 자체가 엄청난 노력

### Option D: 파이프라인을 AIMS의 경쟁력으로 유지 ✅ (가장 현실적)

| 근거 | 상세 |
|------|------|
| 파이프라인 ≠ 제품 | 좋은 엔진이 좋은 차를 만드는 것은 아님 |
| AIMS 자체가 제품 | 파이프라인은 AIMS라는 제품의 핵심 경쟁력 |
| 집중의 법칙 | 솔루션화에 투입할 리소스를 AIMS 고도화에 투입하는 게 ROI 높음 |

## 7. 최종 결론

**솔직한 결론**: 파이프라인의 기술적 완성도는 높지만, **독립 솔루션으로의 시장 진입은 타이밍과 자본 면에서 불리하다.**

- 2023~2024년이었다면 선점 효과가 있었을 것
- 2026년 현재, Reducto($75M), Unstructured.io 등이 이미 시장을 점유
- "API 래핑 + 좋은 아키텍처"만으로는 defensible한 비즈니스가 되기 어려움

**만약 정말 솔루션화를 한다면:**
1. **보험/금융 버티컬 SaaS** (Option B)가 유일하게 승산 있는 방향
2. AR/CRS 파서 같은 **도메인 전문성**이 진짜 moat
3. 범용 Document AI로 가면 자본력 싸움에서 패배

**가장 현명한 선택**: AIMS를 잘 키우면서, 파이프라인 아키텍처를 **내부 기술 자산**으로 활용하고, 추후 기회(인수, 파트너십, 화이트라벨)가 왔을 때 대응하는 것.

> *"좋은 기술을 가졌다고 반드시 제품화해야 하는 것은 아닙니다. 때로는 기술이 다른 제품의 unfair advantage로 작동할 때 가장 큰 가치를 만듭니다."*

## 8. 소규모 업체 대상 니치 전략 분석

> **2026-02-07 추가** | "빅테크가 커버 안 하는 영역, 소규모 업체/서비스 대상" 방향 탐색

### 빅테크 솔루션의 구조적 빈틈

빅테크(AWS Textract, Google Doc AI, Azure)는 **API 제공자**이지 **솔루션 제공자**가 아니다. 이것이 핵심 빈틈:

| 빅테크가 하는 것 | 빅테크가 안 하는 것 |
|------------------|---------------------|
| OCR API 제공 | 업종별 워크플로우 |
| 페이지당 과금 ($1.50/1K) | 월정액 예측 가능한 가격 |
| 개발자가 통합 | 비개발자가 바로 사용 |
| 범용 텍스트 추출 | 업종 특화 데이터 구조화 |
| 스케일 인프라 | 클라이언트 포털/문서 수집 |
| 영어 최적화 | 한국어/일본어 특화 |

**핵심 인사이트**: 소규모 업체가 필요한 건 "OCR API"가 아니라 **"내 업종에서 10분 안에 쓸 수 있는 문서 솔루션"**

### 소규모 업체의 공통 문서 고통

- 기업들이 문서 비효율로 **연간 $1조** 손실 (SenseTask 2025)
- 미국 중소기업 **45%가 아직 종이** 기반
- 소규모 업체 **62%가 AI 도구를 "너무 복잡하다"**고 인식
- 평균 **연 15일을 미수금 인보이스 추적**에 소비

### 유망 니치 매핑: "빅테크가 무시하는 서류 지옥"

#### Tier 1: 가장 유망 (높은 지불의향 + 반복 수요 + 낮은 경쟁)

**1. 소형 법률사무소 / 1인 변호사**

| 항목 | 내용 |
|------|------|
| 시장 규모 | 미국 40만+ 1인 변호사, 한국 3만+ 개인 사무소 |
| 월 지불의향 | $50~200 |
| 핵심 고통 | 사건별 문서 버전관리, 기한 추적, 청구서-문서 연동 |
| 빅테크 빈틈 | 법원 서식 템플릿, 사건 기한 자동 추적, 의뢰인 포털 없음 |

## 9. 전략 제안: "나만의 영역"

### 제안 A: 한국형 "클라이언트 문서 포털" ⭐ (가장 유력)

**컨셉**: 전문직(변호사/세무사/설계사/관세사 등)이 고객에게서 서류를 수집·관리하는 도구

**왜 이것인가?**

1. **업종 횡단** — 보험에 국한되지 않음
2. **빅테크 빈틈** — AWS/Google은 "서류 수집" 워크플로우 제공 안 함
3. **한국 미진출** — FileInvite, Collect 등 모두 영어권
4. **AIMS 기술 100% 재활용** — 업로드, OCR, 분류, 검색, 크레딧 시스템
5. **카톡 대체** — 명확한 페인포인트
6. **반복 수익** — 매월 고객에게서 서류를 받는 업무는 없어지지 않음

**MVP 스코프**:
```
1. 링크 생성 → 고객이 모바일에서 서류 업로드
2. 자동 OCR + 분류
3. 누락 서류 알림 (카카오 알림톡)
4. 건별/의뢰인별 자동 정리
5. AI 검색
```

**확장 경로**: 전문직 포털 → 전자서명 연동 → 업종별 템플릿 → 임베디드 결제

### 제안 B: 특정 버티컬 올인 (예: 관세사/무역)

**왜 관세사/무역?**
- 서류 종류 수십 가지, 실수 시 통관 지연 = 금전 손실
- 한국은 수출 중심 경제, 소형 무역업체 다수
- 빅테크가 절대 안 하는 영역
- ARPU 높음 ($100~300/월)

### 제안 C: AIMS 자체를 "전문직 문서 플랫폼"으로 확장

보험설계사 → 금융 어드바이저 → 세무사 → 변호사로 점진적 확장
- AIMS의 고객 관리 + 문서 관리 + AI 검색이 그대로 적용됨
- 각 업종별 "파서"만 추가하면 됨 (AR/CRS 파서처럼)

## 10. PDF 테이블 파서 고도화 — 독립 제품 가능성

> **2026-02-07 추가** | "텍스트 기반 PDF에서 OCR 없이 표를 정확히 추출하는 범용 파서"

### 문제 정의

현재 대부분의 도구가 PDF 표를 추출할 때:
- 텍스트 기반 PDF인데도 **OCR(이미지 변환)을 사용** → 느리고 비쌈
- 또는 순수 구조 파싱(pdfplumber)을 사용 → 복잡한 표에서 실패

**아이디어**: 텍스트 기반 PDF에서 텍스트 레이어를 직접 활용하여, OCR 없이 표 구조를 정확히 추출하는 고도화된 파서

### 시장 규모

**테이블 추출 소프트웨어 시장**:
- 2024년: **$14.7억**
- 2033년: **$41.6억** (CAGR 13.2%)
- 아시아태평양: CAGR **16.4%** (가장 빠른 성장)

**페인포인트**:
- 수동 데이터 입력 비용: **직원당 연 $28,500**
- 직원 60%가 반복적 데이터 작업에 번아웃
- 단순 문서: 10~15분/건, 복잡 문서: 최대 60분/건

### 기존 도구들의 현주소

| 도구 | 방식 | 정확도 | 한계 |
|------|------|--------|------|
| **pdfplumber** | 구조 파싱 (좌표 기반) | ~96% (단순) / ~73% (복잡) | 병합 셀, 테두리 없는 표 실패 |
| **Camelot** | OpenCV 라인 감지 + 텍스트 그룹핑 | ~73% | 모드 수동 선택 필요, 하이브리드 표 불가 |
| **Tabula** | 규칙 기반 | ~68% | 낮은 정확도, 오래된 기술 |
| **PyMuPDF** | find_tables() | 단순 표만 | 복잡한 표에 부적합 |
| **Docling** (IBM) | 비전 모델 (TableFormer) | **97.9%** | 이미지 렌더링 필요, GPU 권장 |
| **AWS Textract** | 비전 AI | 높음 | $1.50~10/1K 페이지, API만 |
| **Azure Doc AI** | 비전 AI | 높음 (불규칙 문서에 강함) | 비싼 엔터프라이즈 가격 |

### 핵심 기술적 발견

**1. "No OCR"의 진짜 의미**

순수하게 OCR을 안 쓰는 건 아님. Docling조차 페이지를 이미지로 렌더링함. 진짜 차별점은:

```
기존: PDF → 이미지 렌더링 → OCR(텍스트 추출) → 패턴 매칭 (느리고 비쌈)
제안: PDF → 텍스트 레이어 직접 추출(좌표 포함) → 구조 인식 (빠르고 저렴)
```

텍스트 기반 PDF는 이미 텍스트와 좌표 정보를 가지고 있다. **이미지로 변환할 이유가 없다.**

**2. PDF 표 추출이 여전히 어려운 이유**

| 문제 | 원인 | 현재 해결 상태 |
|------|------|---------------|
| 테두리 없는 표 | 구조 힌트 없음, 공백만으로 판단 | 미해결 (모든 도구 취약) |
| 병합 셀 | 열 정렬 깨짐 | 부분 해결 (비전 모델만) |
| 다중 페이지 표 | 페이지 경계에서 표 절단 | 거의 미해결 |
| 복합 헤더 | 다단 헤더, 중첩 | 부분 해결 |
| 셀 내 줄바꿈 | 한 셀의 텍스트가 여러 줄 | 도구마다 다름 |

**3. AIMS의 pdfplumber 경험이 증명한 것**

AIMS AR/CRS 파서는 이미:
- ✅ `find_tables()`로 텍스트 레이어에서 직접 표 추출 (OCR 안 씀)
- ✅ 동적 헤더 매핑 (하드코딩 아님)
- ✅ 셀 내 줄바꿈 처리
- ✅ 보험사 연차보고서에서 **100% 정확도** (215/215 테스트 통과)

하지만:
- ❌ 테두리 없는 표 → 실패
- ❌ 병합 셀 → 열 정렬 깨짐
- ❌ 다중 페이지 → 미구현
- ❌ 파라미터 튜닝이 MetLife 형식에 최적화됨

### 제안 아키텍처: 하이브리드 파서

```
                    ┌──────────────────────────┐
                    │    PDF 업로드 (텍스트 기반)  │
```

## 11. Self-hosted Document AI — 빅테크의 구조적 사각지대

> **2026-02-07 추가** | "자체 호스팅으로 문서 처리를 유지하려는 소규모 업체" 방향

### 왜 빅테크가 구조적으로 못 하나

빅테크의 수익 구조 자체가 **클라우드 소비**:

```
AWS/Google/Azure의 수익:
  고객 문서 → 우리 클라우드에서 처리 → 페이지당 과금 → $$$

Self-hosted를 만들면:
  고객 문서 → 고객 서버에서 처리 → 우리에게 돈 안 냄 → ???
```

**"자체 보유/자체 처리"는 빅테크의 이해관계와 정면 충돌.** "관심 없음"이 아니라 **"하면 안 됨"**.

### 문서를 외부로 보내기 싫은 사람들

| 대상 | 이유 | 민감도 |
|------|------|--------|
| 법률사무소 | 의뢰인 비밀유지 의무 | 매우 높음 |
| 세무/회계사무소 | 재무 정보, 세금 데이터 | 높음 |
| 의원/병원 | 환자 정보 (HIPAA, 의료법) | 매우 높음 |
| 관세사/무역 | 거래 기밀, 가격 정보 | 높음 |
| 보험설계사 | 고객 개인정보 | 높음 |
| 소규모 기업 | "우리 서류를 남의 서버에?" 심리적 저항 | 중간 |
| 공공기관/지자체 | 데이터 주권, 규정 | 매우 높음 |

### 현재 선택지와 빈틈

| | 클라우드 SaaS | 오픈소스 | 엔터프라이즈 온프레미스 | **빈 영역** |
|---|-------------|---------|---------------------|-----------|
| 설치 | 없음 | 개발자 필수 | SI 업체 필요 | **10분 설치** |
| 비용 | $20~200/월 | 무료 | $50K+ | **$30~100/월** |
| 데이터 | 클라우드 전송 | 로컬 유지 | 로컬 유지 | **로컬 유지** |
| 대상 | 일반 | 개발자 | 대기업 | **소규모 비개발자** |
| 예시 | Reducto | Docling | ABBYY | **???** |

### 제품 컨셉: "Self-hosted Document AI in a Box"

```
Docker 한 줄로 설치:
  docker run -p 8080:8080 docbox/server

→ 브라우저에서 localhost:8080 접속
→ PDF 업로드 → OCR / 표 추출 / 검색 → 결과 다운로드
→ 모든 데이터가 내 서버에만 존재
```

**AIMS 파이프라인에서 교체가 필요한 부분**:

| 현재 (클라우드 의존) | 자체 호스팅 대안 |
|---------------------|-----------------|
| Upstage OCR API | PaddleOCR / Tesseract (오픈소스) |
| OpenAI Embedding | sentence-transformers (로컬) |
| OpenAI GPT 요약 | Ollama + llama3 (로컬) |
| Qdrant Cloud | Qdrant self-hosted (Docker) |
| MongoDB Atlas | MongoDB self-hosted |

### 이 방향이 매력적인 이유

1. **빅테크와 경쟁이 아니라 보완** — 클라우드 싫은 고객 = AWS/Google과 고객 안 겹침
2. **규제가 아군** — 개인정보보호법, GDPR, HIPAA 모두 강화 추세
3. **인프라 비용 없음** — 고객이 자기 서버 사용, 우리는 소프트웨어만 제공
4. **1~3인 팀으로 가능** — Docker 패키징 + 웹 UI, AI는 오픈소스 조합

### 수익화 모델 분석

**핵심 질문**: 고객 서버에서 돌아가는데, 어떻게 돈을 받나?

#### Model A: 오픈코어 (Open-Core) ⭐ 가장 현실적

```
무료 (Community Edition)          유료 (Pro/Enterprise)
├─ 기본 OCR (Tesseract)           ├─ 고성능 OCR (PaddleOCR+)
├─ PDF 텍스트 추출                 ├─ AI 요약 (로컬 LLM)
├─ 단일 사용자                     ├─ 멀티 유저 + 권한 관리
├─ CSV 내보내기                    ├─ Excel/JSON/API 내보내기
├─ 수동 분류                       ├─ AI 자동 분류
└─ 커뮤니티 지원                   ├─ 업종별 템플릿 (법률/세무/관세)
                                  ├─ 클라이언트 포털 (서류 수집 링크)
                                  ├─ 감사 로그 (컴플라이언스)
                                  └─ 우선 기술 지원
```

**성공 사례**:

| 회사 | 모델 | 무료 | 유료 | 매출 |
|------|------|------|------|------|
| **GitLab** | 오픈코어 | CE (기본) | $29/user/월 (Premium) | $580M+ ARR |

**왜 이 모델이 맞나**:
- 무료 버전으로 **신뢰 확보** + 커뮤니티 형성 (→ 마케팅 비용 절감)
- 유료 전환은 **"업무에 쓰기 시작하면"** 자연스럽게 발생
- 멀티유저, 감사 로그, 업종 템플릿은 **무료로 줄 이유가 없는** 비즈니스 기능
- 개발 노력의 대부분은 코어에 집중, 유료 기능은 **래핑/설정** 수준

#### Model B: 라이선스 구독

```
설치는 자유, 라이선스 키로 활성화:

Starter:  $30/월 — 1 사용자, 월 500건

```

**장점**: 단순하고 예측 가능한 수익
**단점**: 무료 체험 없이는 진입장벽, 크랙/우회 가능성
**참고**: JetBrains 모델 ($15~25/월/사용자, 연간 결제 할인)

#### Model C: 하이브리드 (Self-hosted + Cloud AI)

```
코어: 고객 서버에서 실행 (무료 또는 저렴)
  ├─ 파일 저장, 기본 OCR, 검색 — 모두 로컬

AI 기능: 우리 클라우드 API 호출 (종량제)
  ├─ 고급 요약 (GPT-4 수준)
  ├─ 스마트 분류
  ├─ 자연어 검색
  └─ 표 추출 고도화 (비전 모델)
```

**가격**: 코어 무료 + AI 크레딧 $20~100/월

**장점**:
- 민감 데이터(원본 파일)는 로컬 유지 → 보안 우려 해소
- AI 처리만 클라우드 → "문서 전체를 보내는 것 아님" 포지셔닝
- AIMS의 크레딧 시스템 **그대로 재활용**

**단점**:
- "완전 자체 호스팅"을 원하는 고객에겐 타협
- 클라우드 AI 의존 = 빅테크 API 비용 발생

**이 모델의 진짜 매력**: 고객이 성장하면 자연스럽게 AI 사용량 증가 → **수익이 고객 성장에 비례**

### 2026-02-07 #3 | PDF 테이블 파서 고도화 아이디어

**질문**: 텍스트 기반 PDF에서 OCR 없이 표를 정확히 추출하는 범용 파서를 만들 수 있을까? 독립 제품으로서의 가능성은?

**핵심 발견**:
- 테이블 추출 소프트웨어 시장: $14.7B → $41.6B (CAGR 13.2%)
- 중간가 영역($10~50/월, 고정확도) 비어있음 — 무료(68%)와 비싼 엔터프라이즈 사이
- AIMS pdfplumber 경험으로 기술 검증됨, 하이브리드(규칙+경량ML) 접근 가능
- 하지만 Docling(IBM, 무료, 97.9%)과 빅테크 경쟁은 불가피
- **범용 파서보다 특정 문서 유형 특화가 현실적 승리 전략**

### 2026-02-07 #4 | "범용 테이블 파서" 기술적 실현 가능성 검증

**질문**: 범용 테이블 파서를 만드는 게 기술적으로 가능한가?

**결론**: "범용"의 범위에 따라 다르다.

| 목표 | 가능성 | 필요한 것 |
|------|--------|----------|
| 테두리 있는 표 90%+ | ✅ 지금도 가능 | pdfplumber 튜닝 |
| 다양한 표 80%+ | ⚠️ 가능하나 노력 큼 | 경량 ML + 휴리스틱 |
| 모든 표 95%+ | ❌ OCR 없이는 불가 | 비전 모델 필수 (=Docling과 같은 길) |

**근본적 한계**: PDF 내부에 "표"라는 개념이 없음. 좌표와 선으로부터 표를 "추론"해야 하며, 테두리 없는 표/병합 셀은 텍스트 좌표만으로는 사람에게도 맥락이 필요한 판단. Docling(97.9%)조차 이미지 렌더링 + 비전 모델을 사용 — "OCR 없이 범용 97%+"는 현재 기술로 불가능.

**방향 재확인**: 범용 파서가 아닌, **"이 업종의 이 문서를 완벽하게"** 전략이 유일하게 현실적. AIMS AR 파서가 100%인 이유도 MetLife 연차보고서라는 좁은 범위에서 완벽하기 때문.

### 2026-02-07 #5 | Self-hosted = 빅테크의 구조적 사각지대

**질문**: 자체적으로 문서 처리를 유지하려는 소규모 업체라면 빅테크가 관심을 안 갖는 영역 아닌가? 수익화는 어떻게?

**핵심 발견**:
- 빅테크의 수익 구조(클라우드 소비)와 self-hosted는 **이해관계가 정면 충돌** → 단순 무관심이 아니라 구조적으로 못 함
- 법률/세무/의료/관세 등 민감 문서를 다루는 소규모 업체가 타겟
- 기존 선택지: 클라우드(보안 우려) / 오픈소스(개발자 필요) / 엔터프라이즈($50K+) → 중간이 비어있음

**수익화 추천**: 오픈코어 + 하이브리드 Cloud AI 결합
- 무료 Community Edition으로 커뮤니티 형성
- Pro License $50~80/월 (멀티유저, 업종 템플릿, 감사로그)
- Cloud AI 크레딧 $20~50/월 (원본은 로컬, AI 처리만 클라우드)
- 보수적 예상: 24개월 차 ARR $300K, 36개월 차 ARR $960K

**수렴하는 전략**: 한국형 문서 포털 + 자체 호스팅 + 업종 특화 = 빅테크가 "할 수 없는" 영역
