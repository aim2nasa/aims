# AIMS 표준 문서 분류 체계

> **버전**: 3.1 | **최종 수정**: 2026-03-05 | **상태**: 설계 확정 (Alex/Gini 검증 완료)

## 핵심 요약

```
문서 텍스트 (최대 10,000자)
    ↓
기존 OpenAI 호출 (요약+태그+제목)에 분류 통합 — 추가 API 호출 0회
    ↓
JSON 응답: { type: "diagnosis", confidence: 0.85, title: "...", summary: "...", tags: [...] }
```

**문서 1건이 입력되면, 기존 요약 생성과 동시에 41개 유형 중 1개로 분류된다.**
현재 이미 문서마다 OpenAI를 호출하여 요약+태그+제목을 생성하고 있으므로, 같은 호출에 분류(type, confidence)를 추가한 것이다. API 호출 횟수는 동일하며, 프롬프트 503 토큰 추가로 문서당 약 $0.00013의 비용만 증가한다.

## 개요

보험 설계사가 관리하는 고객 문서를 표준화된 트리 구조로 분류합니다.

### 분석 기반
- **TestData** (505개 파일): AR/CRS, 법인자료, 개인고객 문서
- **고객폴더 백업** (21,596개 파일): 개인 고객 ~300명, 법인 ~20개사, 보험사별 자료
- **MongoDB 실데이터**: 2,080건 (AR 1,225, CRS 704, 미분류 134, 기타 17)
- **파일명 키워드 빈도 분석**: 22,101개 파일 전수 조사

### 설계 원칙
- **2단계 트리**: 대분류(category) → 소분류(document_type)
- **빈도 기반**: 실제 파일 키워드 출현 빈도로 분류 정당성 검증
- **보험 도메인 특화**: 설계사 실무 흐름(계약→유지→청구→변경)에 맞춤
- **MECE**: 상호 배타적, 전체 포괄적
- **확장 가능**: 새로운 소분류 추가 시 대분류 구조 유지

### 검증 결론

- **누락 유형 없음**: 실제 폴더에서 발견된 모든 문서 유형이 분류 체계에 포함됨
- **폴더 구조 ≠ 문서 유형**: `블루고객`, `해지고객` 등은 고객 상태 분류이며 문서 유형이 아님
- **파일 포맷 ≠ 문서 유형**: `.hwp`, `.pptx` 등은 포맷이며, 내용 기반으로 분류

## 태그 표준 체계

태그는 문서 유형과 **별개 축**으로, 문서의 속성/맥락을 나타냅니다.

### 태그 규칙
- `#` 접두사 사용하지 않음
- 띄어쓰기 없이 붙여쓰기 (예: `메트라이프`, NOT `메트라이프 생명`)
- 보험사명은 약칭으로 정규화 (예: `메트라이프생명보험` → `메트라이프`)

## 설계 결정 사항 (v3.0)

### 결정 1: AI 직접 분류 (태그 기반 분류 기각)

```
❌ 기각: 텍스트 → AI → 태그 → 규칙매칭 → 분류 (4단계, 오류 누적)
✅ 채택: 텍스트 → AI → 분류 + 태그 동시 반환      (2단계, 단순)
```

- 이미 OpenAI를 호출하고 있으므로 같은 프롬프트에서 document_type을 직접 선택하게 함
- 추가 API 비용 0원, documentTypeClassifier.js 규칙 엔진 불필요
- 태그는 분류 결정에 관여하지 않고, 검색/필터 용도로만 사용

### 결정 2: 분류는 document_type 1개만

- 한 문서에 document_type은 **1개**만 부여 (주된 용도 기준)
- 복수 유형 해당 시 AI가 "이 문서의 주된 목적"을 판단하여 1개 선택
- 보조 정보는 태그로 보존 (예: 진단서+영수증 합본 → `diagnosis` + 태그 `영수증`)

### 결정 3: category는 document_types 컬렉션에서 파생

- `files` 컬렉션에 `document_category` 중복 저장하지 않음 (SSOT)
- `document_types` 컬렉션의 `category` 필드에서 룩업

### 결정 4: Gini Critical 이슈 대응

| 이슈 | 대응 |
|------|------|
| `claim` → `claim_form` rename 시 DB 파괴 | 마이그레이션 스크립트 필수. rename 포함 변경은 DB 마이그레이션과 동시 배포 |
| `income_employment` 분리 | 기존 데이터는 `income_proof`로 일괄 변환 |
| `is_annual_report` + `document_type` 이중 저장 | 점진적 통합 (영향 범위 크므로 별도 과제) |

## AI 분류 기준 (프롬프트 가이드)

AI가 document_type을 결정할 때 사용하는 판단 기준입니다.
이 섹션은 OpenAI/Claude 프롬프트에 그대로 주입됩니다.

### 분류 원칙

1. **주된 목적 기준**: 문서가 무엇을 위해 만들어졌는가? (첨부물이 아닌 본 문서의 목적)
2. **1문서 1유형**: 반드시 하나만 선택. 확신 없으면 `general`
3. **시스템 유형 직접 선택 금지**: `annual_report`, `customer_review`, `unspecified`는 시스템이 자동 판단. AI가 선택하지 않음

### 모호한 경우 우선순위 규칙

여러 유형에 해당할 수 있을 때, 아래 우선순위를 따릅니다.

1. **문서의 주된 목적**이 무엇인가? (첨부물이 아닌 본 문서)
2. **구체적 유형 우선**: `general` < `memo` < 구체적 유형
3. **보험금청구 맥락 우선**: 청구용으로 제출된 진단서/영수증은 `claim` 카테고리
4. **법인 vs 개인**: 법인 기본 서류(정관, 주주명부)는 `corporate`, 개인 증빙(신분증, 통장)은 `identity`
5. **가입 전 vs 가입 후**: 가입 전 설계 자료는 `plan_design`, 가입 후 분석은 `coverage_analysis`

#### 설계 원칙

| 항목 | 설정 | 이유 |

#### 토큰 최적화 전략

1. **유형 목록 압축**: `코드=설명` 한 줄 형식 (테이블 대비 -60% 토큰)
2. **판단 기준은 프롬프트에 넣지 않음**: 유형명+설명만으로 gpt-4o-mini가 충분히 분류 가능
3. **모호 규칙은 3줄로 압축**: 핵심 원칙만 포함
4. **태그 사전 제거**: 자유 추출 후 후처리로 정규화 (토큰 절약 + 새 보험사/상품 대응)

#### AR/CRS 우선순위 규칙

```
우선순위: doc_prep_main.py 시스템 감지(AR/CRS) > AI 분류(document_type)
```

- `doc_prep_main.py`가 PDF 텍스트 패턴으로 AR/CRS를 감지하면 `annual_report`/`customer_review` 부여
- AI 분류 결과와 무관하게 시스템 감지가 우선
- AI가 실수로 `annual_report`/`customer_review`/`unspecified`를 반환해도 → `general`로 교체

#### 토큰 사용량 (tiktoken 실측)

| 구성 | 토큰 수 | 비고 |
|------|---------|------|
| 시스템 프롬프트 | 26 | 고정 (실측) |

#### 일관성 보장 메커니즘

1. **temperature=0**: 동일 입력 시 동일 출력 보장 (greedy decoding)
2. **열거형 제약**: 유형 목록을 명시적으로 나열하여 AI가 임의 유형 생성 방지
3. **JSON 응답 강제**: 자유 텍스트 응답 방지, 파싱 실패 최소화
4. **confidence 활용**: 초기 배포 시 로깅만 수행. 실데이터 100~500건 분포 확인 후 임계값 결정
5. **response_format**: `{"type": "json_object"}` API 파라미터로 JSON 구조 강제 (프롬프트만으로 불충분)
6. **AR/CRS 우선순위**: 시스템 감지(doc_prep_main.py) > AI 분류. 충돌 시 시스템 감지 우선
7. **후처리 정규화**:
   - AI 응답의 `type`이 목록에 없으면 → `general`
   - `tags`에서 보험사명/상품명 정규화 (예: "메트라이프생명" → "메트라이프")
   - `annual_report`/`customer_review`/`unspecified` 반환 시 → `general`로 교체
