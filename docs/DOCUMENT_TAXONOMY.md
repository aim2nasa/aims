## 핵심 요약

```
문서 텍스트 (최대 10,000자)
    ↓
기존 OpenAI 호출 (요약+태그+제목)에 분류 통합 — 추가 API 호출 0회
    ↓
JSON 응답: { type: "diagnosis", confidence: 0.85, title: "...", summary: "...", tags: [...] }
```

**문서 1건이 입력되면, 기존 요약 생성과 동시에 45개 유형 중 1개로 분류된다.**
현재 이미 문서마다 OpenAI를 호출하여 요약+태그+제목을 생성하고 있으므로, 같은 호출에 분류(type, confidence)를 추가한 것이다. API 호출 횟수는 동일하며, 프롬프트 503 토큰 추가로 문서당 약 $0.00013의 비용만 증가한다.

**엔티티 모델** *(experimental)*: 법인과 개인은 독립 엔티티이며, 연관(association) 관계로 연결된다. 문서는 엔티티에 귀속되고, 관계 정보는 별도 관리된다 (SSOT). → [상세](#문서-고객-관계-모델-v32-experimental)

**관계인 문서 뷰** *(experimental)*: 분류 체계(9개)와 별도로, 문서 탐색기에 탭을 추가하여 관계인별 문서를 조회한다. 법인은 "관계자 문서"(대표/임원/직원), 개인은 "가족 문서"(배우자/부모/자녀). → [상세](#관계인-문서-뷰-ui-탭-v32-experimental)

## 개요

보험 설계사가 관리하는 고객 문서를 표준화된 트리 구조로 분류합니다.

### 분석 기반
- **TestData** (505개 파일): AR/CRS, 법인자료, 개인고객 문서
- **고객폴더 백업** (21,596개 파일): 개인 고객 ~300명, 법인 ~20개사, 보험사별 자료
- **MongoDB 실데이터**: 2,080건 (AR 1,225, CRS 704, 미분류 134, 기타 17)
- **파일명 키워드 빈도 분석**: 22,101개 파일 전수 조사

### 설계 원칙
- **2단계 트리**: 대분류(category) → 소분류(document_type)
- **빈도 기반**: 실제 파일 키워드 출현 빈도로 분류 정당성 검증
- **보험 도메인 특화**: 설계사 실무 흐름(계약→유지→청구→변경)에 맞춤
- **MECE**: 상호 배타적, 전체 포괄적
- **확장 가능**: 새로운 소분류 추가 시 대분류 구조 유지

## 문서-고객-관계 모델 [v3.2-experimental]

> **이 섹션은 실험적 설계입니다.** 구현 여부 미정이며, 사용 패턴 관찰 후 정식 버전으로 승격할지 결정합니다.

문서 분류는 **개별 문서의 유형**을 결정하지만, 실무에서는 문서가 어떤 **고객(엔티티)**에 속하고, 그 고객이 다른 고객과 어떤 **관계**를 맺는지가 중요합니다.

### 엔티티 구조

```
설계사 (Agent)
  │
  ├──── 고객A: 캐치업코리아 [법인]          ← 독립 엔티티
  ├──── 고객B: 김보성 [개인]                ← 독립 엔티티
  ├──── 고객C: 안영미 [개인]                ← 독립 엔티티
  └──── 고객D: 라이콘코리아 [법인]          ← 독립 엔티티
```

**핵심**: 법인과 개인은 모두 독립적인 고객 엔티티입니다. 포함(containment) 관계가 아닌 **연관(association)** 관계로 연결됩니다.

### 연관 관계 (Association)

```
캐치업코리아 [법인]                        김보성 [개인]
       │                                    │
       │  "관계자 추가"                      │  "가족 추가"
       │  (corporate)                        │  (family)
       │                                    │
       ├── ceo ──→ 김보성 [개인]             ├── spouse ──→ 안영미 [개인]
       ├── executive ──→ 유아영 [개인]       ├── child ──→ 김하늘 [개인]
       └── employee ──→ 박지우 [개인]        └── parent ──→ 김철수 [개인]
```

관계는 **양방향 탐색** 가능하되, 각 엔티티는 자신만의 문서를 독립적으로 소유합니다.

### AIMS 관계 유형 (customer_relationships)

#### 법인 관계 — "관계자 추가" (법인 ↔ 개인)

> Source: `CorporateRelationshipModal.tsx` CORPORATE_RELATIONSHIP_TYPES

| 코드 | 설명 | UI 표시 |
|------|------|---------|
| `ceo` | 대표이사 | 👔 대표 |
| `executive` | 임원 | 🎯 임원 |
| `employee` | 직원 | 👤 직원 |

- 법인 고객 상세 → "관계자 추가" 버튼으로 개인 고객을 연결
- 사용자 정의 관계 타입 허용 (`allowCustomRelation: true`)
- 법인 고객은 "가족 추가" 불가
- 백엔드에는 `shareholder`(주주), `director`(이사)도 정의되어 있으며, `company`/`employer`는 역방향 조회용 내부 타입

#### 개인 관계 — "가족 추가" (개인 ↔ 개인)

> Source: `FamilyRelationshipModal.tsx` FAMILY_RELATIONSHIP_TYPES

| 코드 | 설명 | UI 표시 |
|------|------|---------|
| `spouse` | 배우자 | ❤️ 배우자 |
| `parent` | 부모 | 👨‍👩 부모 |
| `child` | 자녀 | 👶 자녀 |

- 개인 고객 상세 → "가족 추가" 버튼으로 다른 개인 고객을 연결
- 가족관계등록부 범위 내 관계 유형만 허용

> 백엔드 `RELATIONSHIP_TYPES`에는 relative, social, professional 등 추가 카테고리가 정의되어 있으나,
> 현재 UI에서 사용하는 관계 추가 경로는 위 두 가지(법인 관계자 / 개인 가족)뿐입니다.

#### 미지정 관계 [미구현 — 설계]

> 아래는 v3.2-experimental에서 설계된 신규 관계 유형입니다. 현재 백엔드/프론트엔드에 구현되어 있지 않으며,
> 폴더 일괄 업로드 기능 구현 시 함께 추가됩니다.

| 카테고리 | 코드 | 설명 | 방향 |
|---------|------|------|------|
| **unidentified** | `unidentified` | 관계 있음, 유형 미지정 | 양방향 |

> `unidentified`: 폴더 분석 등으로 관계가 감지되었으나 구체적 유형을 아직 모르는 상태.
> 법인↔개인, 개인↔개인 어느 쪽이든 사용 가능. 설계사가 나중에 정확한 관계로 변경합니다.
> **구현 필요 항목**: RELATIONSHIP_TYPES에 `unidentified` 카테고리 추가, Zod enum 확장, 프론트엔드 UI 지원.

### 문서 소유와 관계의 분리

```
문서 소유 (1:N)                    엔티티 관계 (M:N)
─────────────                    ──────────────
캐치업코리아 ─→ 정관.hwp           캐치업코리아 ←──ceo──→ 김보성
캐치업코리아 ─→ 급여대장.xlsx      캐치업코리아 ←──employee──→ 박지우
김보성 ─→ 신분증.pdf              김보성 ←──spouse──→ 안영미
김보성 ─→ 건강검진결과.pdf         김보성 ←──child──→ 김하늘
안영미 ─→ 진단서.pdf
```

- **문서 소유**: 각 문서는 정확히 1명의 고객에게 귀속 (`documents.customer_id`)
- **엔티티 관계**: 고객 간 연관 관계는 별도 컬렉션 (`customer_relationships`)에 저장
- **SSOT**: 문서에 관계 정보를 중복 저장하지 않음. 관계는 항상 `customer_relationships`에서 조회

### 분류와 관계의 교차점

문서 분류(document_type)와 엔티티 관계가 교차하는 실무 시나리오:

| 시나리오 | 문서 유형 | 소유 엔티티 | 관련 관계 |
|---------|----------|-----------|----------|
| 법인 대표의 신분증 | `id_card` | 김보성(개인) | 캐치업코리아의 ceo |
| 법인 근로계약서 | `hr_document` | 캐치업코리아(법인) | 직원들과 employee 관계 |
| 배우자 보험증권 | `policy` | 안영미(개인) | 김보성의 spouse |
| 직원 급여대장 | `hr_document` | 캐치업코리아(법인) | 박지우와 employee 관계 |
| 가족 건강검진 | `health_checkup` | 김하늘(개인) | 김보성의 child |

> **설계 결정**: AI 분류는 문서 유형만 판단합니다. 단건 업로드 시 문서 소유는 업로드 컨텍스트(어떤 고객 화면에서 업로드했는지)로 결정되며, 고객 간 관계는 설계사가 직접 설정합니다.

### 폴더 일괄 업로드 시 엔티티 분리 파이프라인 [미구현 — 설계]

법인 폴더를 통째로 업로드하면 법인 문서와 관련 개인의 문서가 뒤섞여 있습니다.
이를 올바른 엔티티에 귀속시키기 위한 3-Phase 파이프라인:

```
캐치업코리아 폴더 업로드 (440건)
    │
    ▼ Phase 1: 엔티티 분리 (문서별 소유자 판단)
    │
    │  AI가 문서 내용을 분석하여 owner_hint 추출
    │  ┌─────────────────────────────────────────────────────┐
    │  │ 정관_캐치업코리아.hwp    → owner: "캐치업코리아"      │
    │  │ 김보성신분증.pdf         → owner: "김보성"            │
    │  │ 급여대장.xlsx            → owner: "캐치업코리아"      │
    │  │ 안영미진단서.pdf         → owner: "안영미"            │
    │  │ 자동차등록증.pdf         → owner: ?  (판단 불가)      │
    │  │ img001.jpg               → owner: ?  (판단 불가)      │
    │  └─────────────────────────────────────────────────────┘
    │
    ▼ Phase 2: 소유자 매칭 및 관계 식별
    │
    │  owner_hint를 기존 고객/관계 목록과 대조
    │  ┌──────────────────────────────────────────────────────┐
    │  │ "캐치업코리아" → 업로드 대상 고객 (그대로 귀속)       │
    │  │ "김보성"       → 관계 고객에 있음 (ceo) → 김보성 귀속 │
    │  │ "안영미"       → 관계 고객에 없음 → ⚠ 설계사에게 제안 │
    │  │ owner 불명     → 업로드 대상(캐치업코리아)에 귀속      │
    │  └──────────────────────────────────────────────────────┘
    │
    ▼ Phase 3: 엔티티별 문서 분류 (기존 taxonomy 적용)
    │
    │  각 엔티티 내에서 동일한 45개 분류 체계 적용
    │  캐치업코리아: 정관.hwp → corp_registry, 급여대장.xlsx → hr_document
    │  김보성:       신분증.pdf → id_card, 건강검진결과.pdf → health_checkup
    │  안영미:       진단서.pdf → diagnosis
```

#### 설계 결정 사항 [미구현 — 설계]

**DD-1. 소유자 미식별 문서의 귀속**

소유자를 판단할 수 없는 문서(내용 불명, 이미지만 있는 파일 등)는
**업로드 대상 엔티티에 귀속**시킵니다.

- 캐치업코리아 폴더를 업로드했으면 → 캐치업코리아 소유
- 김보성 폴더를 업로드했으면 → 김보성 소유
- 설계사가 나중에 수동으로 재배정 가능

근거: 설계사가 "캐치업코리아 폴더"를 올렸다는 것은 해당 법인의 맥락에서 작업 중이라는 의미이며,
소유자 불명 문서를 미소속으로 남겨두는 것보다 업로드 맥락에 따르는 것이 실무적으로 합리적입니다.

**DD-2. 미등록 고객 발견 시 처리**

AI가 문서에서 기존 고객/관계 목록에 없는 이름을 발견한 경우,
**기본적으로 설계사에게 제안**하며, 향후 자동 등록 옵션도 열어둡니다.

주의 사항:
- 문서 내 언급된 이름이 반드시 소유자는 아님 (보험 수익자, 계약 상대방 등)
- 동명이인, 표기 차이("김보성" vs "김 보성") 오매칭 위험

**기본 동작 — 설계사에게 제안**:

```
┌─────────────────────────────────────────────────┐
│ ⚠ "안영미"의 문서로 추정되는 파일 3건 발견       │
│                                                  │
│ • 안영미진단서20230913.pdf                       │
│ • 안영미병원비영수증.pdf                          │
│ • 안영미진료비영수증.jpg                          │
│                                                  │
│ [ 기존 고객에 연결 ] [ 새 고객 생성 ] [ 무시 ]    │
│                                                  │
│ "무시" 선택 시 캐치업코리아에 귀속됩니다.         │
└─────────────────────────────────────────────────┘
```

- **기존 고객에 연결**: 설계사의 고객 목록에서 선택 → 해당 고객에 문서 귀속 + 관계 설정 제안
- **새 고객 생성**: 고객 등록 후 문서 귀속 + `unidentified` 관계로 연결
- **무시**: 업로드 대상 엔티티(캐치업코리아)에 그대로 귀속

**향후 확장 — 자동 고객 등록 (옵션)**:

설계사가 원할 경우, 미등록 고객을 자동으로 생성하는 모드를 활성화할 수 있습니다.
이 경우에도 `unidentified` 관계로 연결되며, 설계사가 나중에 정확한 관계를 지정합니다.

```
자동 등록 흐름:
"안영미" 감지 → 기존 고객 매칭 실패 → 개인 고객 자동 생성
→ 캐치업코리아 ←── unidentified ──→ 안영미
→ 안영미에 문서 3건 귀속
→ 설계사에게 사후 알림: "안영미(개인) 자동 등록됨, 관계를 지정해주세요"
```

> **구현 우선순위**: 기본 동작(제안 방식)을 먼저 구현하고,
> 자동 등록은 사용 패턴을 관찰한 후 도입 여부를 결정합니다.

**DD-3. 비용 최적화 — 엔티티 분리와 분류의 통합 호출**

엔티티 분리(Phase 1)와 문서 분류(Phase 3)를 **별도 AI 호출로 분리하지 않습니다.**
기존 문서별 분류 호출에 `owner_hint` 필드를 추가하여 단일 호출로 처리합니다.

```
현재 (v3.2):      문서 → AI 1회 → { type, confidence, title, summary, tags }
experimental:  문서 → AI 1회 → { type, confidence, title, summary, tags, owner_hint }
```

- 추가 API 호출: **0회** (기존과 동일)
- 추가 토큰: `owner_hint` 필드 설명 ~30토큰, 응답에 이름 ~5토큰
- Phase 2(소유자 매칭)는 **AI 호출 없이** 기존 고객/관계 DB 조회로 처리

```

## 태그 표준 체계

태그는 문서 유형과 **별개 축**으로, 문서의 속성/맥락을 나타냅니다.

### 태그 규칙
- `#` 접두사 사용하지 않음
- 띄어쓰기 없이 붙여쓰기 (예: `메트라이프`, NOT `메트라이프 생명`)
- 보험사명은 약칭으로 정규화 (예: `메트라이프생명보험` → `메트라이프`)

## 설계 결정 사항 (v3.0)

Alex(아키텍트) + Gini(품질검증) 리뷰 결과를 반영한 설계 결정입니다.

### 결정 1: AI 직접 분류 (태그 기반 분류 기각)

```
❌ 기각: 텍스트 → AI → 태그 → 규칙매칭 → 분류 (4단계, 오류 누적)
✅ 채택: 텍스트 → AI → 분류 + 태그 동시 반환      (2단계, 단순)
```

- 이미 OpenAI를 호출하고 있으므로 같은 프롬프트에서 document_type을 직접 선택하게 함
- 추가 API 비용 0원, documentTypeClassifier.js 규칙 엔진 불필요
- 태그는 분류 결정에 관여하지 않고, 검색/필터 용도로만 사용

### 결정 2: 분류는 document_type 1개만

- 한 문서에 document_type은 **1개**만 부여 (주된 용도 기준)
- 복수 유형 해당 시 AI가 "이 문서의 주된 목적"을 판단하여 1개 선택
- 보조 정보는 태그로 보존 (예: 진단서+영수증 합본 → `diagnosis` + 태그 `영수증`)

### 결정 3: category는 document_types 컬렉션에서 파생

- `files` 컬렉션에 `document_category` 중복 저장하지 않음 (SSOT)
- `document_types` 컬렉션의 `category` 필드에서 룩업

### 결정 4: Gini Critical 이슈 대응

| 이슈 | 대응 |
|------|------|
| `claim` → `claim_form` rename 시 DB 파괴 | 마이그레이션 스크립트 필수. rename 포함 변경은 DB 마이그레이션과 동시 배포 |
| `income_employment` 분리 | 기존 데이터는 `income_proof`로 일괄 변환 |
| `is_annual_report` + `document_type` 이중 저장 | 점진적 통합 (영향 범위 크므로 별도 과제) |

## AI 분류 기준 (프롬프트 가이드)

AI가 document_type을 결정할 때 사용하는 판단 기준입니다.
이 섹션은 OpenAI/Claude 프롬프트에 그대로 주입됩니다.

### 분류 원칙

1. **주된 목적 기준**: 문서가 무엇을 위해 만들어졌는가? (첨부물이 아닌 본 문서의 목적)
2. **1문서 1유형**: 반드시 하나만 선택. 확신 없으면 `general`
3. **시스템 유형 직접 선택 금지**: `annual_report`, `customer_review`, `unspecified`는 시스템이 자동 판단. AI가 선택하지 않음

| `business_plan` 사업계획서 | 사업 기획/인증. "사업계획서", "벤처기업확인", "연구전담부서인증" | 컨설팅 제안은 `proposal` |
| `inheritance_gift` 상속/증여 | 자산 승계. "상속", "증여", "가업승계", "경정청구" | 세무 신고는 `tax_document` |

#### 법률 (legal)

| 유형 | 이 문서인가? | 아닌 경우 |
|------|------------|----------|
| `contract` 계약서 | 보험 외 일반 계약. "계약서", "합의서", "각서", "양수도계약" | 보험 청약서는 `application` |
| `legal_document` 법률서류 | 소송/공증/법적 서류. "판결문", "내용증명", "공증", "가압류" | 일반 합의서는 `contract` |

#### 기타 (general)

| 유형 | 이 문서인가? | 아닌 경우 |
|------|------------|----------|
| `memo` 메모 | 상담 기록, 고객 메모. "메모", 자유 텍스트 기록 | 보장 분석 내용이면 `coverage_analysis` |
| `general` 일반문서 | 문서이지만 위 어디에도 해당하지 않는 것 | 최대한 구체적 유형을 선택할 것 |
| `unclassifiable` 분류불가 | 문서가 아니거나 내용 파악이 불가능. 로고, 디자인, 번호만 있는 이미지 등 | 문서 내용이 있으면 `general` |

### 모호한 경우 우선순위 규칙

여러 유형에 해당할 수 있을 때, 아래 우선순위를 따릅니다.

1. **문서의 주된 목적**이 무엇인가? (첨부물이 아닌 본 문서)
2. **구체적 유형 우선**: `general` < `memo` < 구체적 유형
3. **보험금청구 맥락 우선**: 청구용으로 제출된 진단서/영수증은 `claim` 카테고리
4. **법인 vs 개인**: 법인 기본 서류(정관, 주주명부)는 `corporate`, 개인 증빙(신분증, 통장)은 `identity`
5. **가입 전 vs 가입 후**: 가입 전 설계 자료는 `plan_design`, 가입 후 분석은 `coverage_analysis`

#### AR/CRS 우선순위 규칙

```
우선순위: doc_prep_main.py 시스템 감지(AR/CRS) > AI 분류(document_type)

#### 일관성 보장 메커니즘

1. **temperature=0**: 동일 입력 시 동일 출력 보장 (greedy decoding)
2. **열거형 제약**: 유형 목록을 명시적으로 나열하여 AI가 임의 유형 생성 방지
3. **JSON 응답 강제**: 자유 텍스트 응답 방지, 파싱 실패 최소화
4. **confidence 활용**: 초기 배포 시 로깅만 수행. 실데이터 100~500건 분포 확인 후 임계값 결정
5. **response_format**: `{"type": "json_object"}` API 파라미터로 JSON 구조 강제 (프롬프트만으로 불충분)
6. **AR/CRS 우선순위**: 시스템 감지(doc_prep_main.py) > AI 분류. 충돌 시 시스템 감지 우선
7. **후처리 정규화**:
   - AI 응답의 `type`이 목록에 없으면 → `general`
   - `tags`에서 보험사명/상품명 정규화 (예: "메트라이프생명" → "메트라이프")
   - `annual_report`/`customer_review`/`unspecified` 반환 시 → `general`로 교체
