# MetLife 고객목록 자동화 GUI 모니터 구현 계획

## Context

현재 MetLife SikuliX 자동화(`MetlifeCustomerList.py`)의 진행 상황은 PowerShell 콘솔 로그로만 확인 가능하다. 이를 Python GUI로 시각화하여:
- **개발 모드**: 기존 로그 파일을 리플레이하여 MetDO 접속 없이 GUI UX 개발/테스트
- **운영 모드**: SikuliX 프로세스의 실시간 stdout을 GUI에 표시

## 로그 구조 분석

로그 파일(`run_*.log`)에서 파싱할 이벤트 유형:

```
1. [헤더]       설정 정보 (초성, 옵션, OCR 모드)
2. [1단계]      고객목록조회 진입 (1-1 ~ 1-4)
3. [2단계]      초성별 처리
   ├─ 초성 버튼 클릭
   ├─ 네비 페이지 (N1, N2, ...)
   │   ├─ 스크롤 페이지 (S1, S2, ...)
   │   │   ├─ [OCR] 캡처 + API 호출 + 인식 결과
   │   │   ├─ [OCR] 고객 테이블 (이름, 구분, 휴대폰 ...)
   │   │   ├─ [고객처리] N명 처리 시작
   │   │   │   ├─ 고객 클릭 → 통합뷰 진입
   │   │   │   ├─ 변액리포트 (PDF 저장/중복/실패)
   │   │   │   ├─ Annual Report (존재/미존재)
   │   │   │   └─ 통합뷰 종료
   │   │   └─ [COMPARE] 스크롤 비교
   │   └─ 다음 버튼 확인
   └─ Summary (총 행수, 오류, 소요시간, PDF 결과)
4. [완료]       전체 결과
```

## 기술 스택

|------|------|
| GUI | CustomTkinter |

### LogParser 이벤트 유형

```python
@dataclass
class LogEvent:
    type: str          # "header", "phase", "ocr_start", "ocr_result",
                       # "customer_table", "customer_click", "customer_done",
                       # "pdf_save", "pdf_duplicate", "ar_not_found",
                       # "scroll", "compare", "summary", "complete"
    timestamp: float   # 이벤트 시점 (리플레이용)
    data: dict         # 이벤트별 데이터
```

### DataSource 인터페이스

```python
class DataSource(ABC):
    @abstractmethod
    def start(self) -> None: ...

    @abstractmethod
    def next_event(self) -> Optional[LogEvent]: ...

class FileReplaySource(DataSource):
    """기존 로그 파일 리플레이 (속도 조절 가능)"""

class LiveProcessSource(DataSource):
    """SikuliX 프로세스 stdout 실시간 읽기"""
```

### SikuliX 전체 작업 영역 (코드 분석 기반)

스크린샷 클릭 좌표 외에 SikuliX **코드** 분석으로 파악한 전체 영역:

| 항목 | Y 범위 | 근거 |
|------|--------|------|
| 클릭 좌표 (실측) | 113~809 | 스크린샷 866건 분석 |
| 테이블 리전 (OCR) | 362~952 | `TABLE_REGION_Y=362, HEIGHT=590` |
| OCR 인식 범위 | 420~900 | `TABLE_Y_START=420, TABLE_Y_END=900` |
| 마지막 행 클릭 | ~881 | `header_y + 32 + 37*12` (Row 12) |
| `scroll_page_down()` | ~785 | `header_y + 370` (테이블 하단 포커스) |
| **"다음" 버튼** | **~955~985** | **Nexacro 페이지네이션 (테이블 리전 하단)** |
| Nexacro 상태바 | ~990~1020 | "정상 처리 되었습니다" |
| 브라우저 하단 | ~1020~1040 | 브라우저 상태바/스크롤바 |
| Windows 작업 표시줄 | 1040~1080 | OS taskbar |

#### "다음" 버튼 특별 분석

```python
# MetlifeCustomerList.py:1693
next_btn = exists(IMG_NEXT_BTN, 5)  # 전체 화면 스캔!
```

- `exists()`는 **전체 화면**을 스캔하여 이미지 패턴 매칭
- "다음" 버튼 이미지: 녹색 배경 + 흰색 "다음" 텍스트
- "이전" 버튼: **코드에 미존재** (전진 네비게이션만 사용)

### 컴팩트 모드 설계 (극소형)

SikuliX 작업 영역과 **완전히 분리**된 극소형 단일 행:

```
                                                           700px
┌──────────────────────────────────────────────────────────────────┐
│ [일반] ████ [ㅋ] 3/8 37% | 코우머스 | P:2/0/0 A:0/3 | N1 S1/2  │  32px
└──────────────────────────────────────────────────────────────────┘
```

- **위치**: x=0, **y=1044** (Windows 작업 표시줄 위에 배치)
- **크기**: 700x32px (극소형 단일 행)
- **타이틀바 제거**: `overrideredirect(True)` (borderless)
- **항상 위 (Always on Top)**: SikuliX 작업 중에도 보임
- **드래그 이동**: 마우스 드래그로 위치 이동 가능
- **[일반] 버튼**: 클릭 시 4패널 모드로 복귀

#### 왜 Y=1044인가?

1. SikuliX의 모든 클릭/이미지매칭은 Y < 960에서 발생
2. `exists(IMG_NEXT_BTN)` 전체 화면 스캔 → 녹색 "다음" 버튼 vs 어두운 GUI = 혼동 불가
3. Y=1044는 브라우저 영역(~1040) 아래 = **물리적 완전 분리**
4. 작업 표시줄 일부만 가리므로 사용에 지장 없음
