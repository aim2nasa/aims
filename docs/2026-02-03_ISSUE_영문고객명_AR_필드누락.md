## Bug 목록

### Bug 1: displayName 누락 (1차 발견) - Resolved

**증상**: 7건 모두 `displayName` 없음, 전체 문서 보기에서 파일명 미표시

**원인**: `doc_prep_main.py`의 고객명 추출 regex가 한글만 매칭:

```python
customer_pattern = r'([가-힣]{2,10})\s*고객님을?\s*위한'
```

영문 고객명 → regex 매칭 실패 → `customer_name = None` → `displayName` 생성 조건 실패

**1차 수정**: PDF 텍스트 추출 실패 시 파일명 기반 fallback 추가

```python
if not customer_name or not issue_date:
    base = os.path.splitext(original_name)[0]
    fn_match = re.match(r'^(.+?)_AR_(\d{4}-\d{2}-\d{2})$', base)
```

**1차 수정 결과**: displayName 7/7 복구

### Bug 2: JUNGCLAIREBOKYUNG → "자료는" 잘못된 고객 매핑 (2차 발견) - Resolved

**증상**: 5번 파일(JUNGCLAIREBOKYUNG)이 "자료는"이라는 잘못된 고객명에 매핑됨

**원인**: PDF 본문에 "이 자료는 고객님을 위한" 텍스트 존재

```
1. 한글 regex [가-힣]{2,10}\s*고객님을?\s*위한
2. "자료는 고객님을 위한" → "자료는" 매칭! (한글 3자)
3. customer_name = "자료는" (틀렸지만 None이 아님)
4. 파일명 fallback 조건: if not customer_name → False (발동 안 됨!)
5. 결과: "자료는"이라는 유령 고객 생성
```

**핵심**: 1차 수정의 fallback은 `customer_name`이 None일 때만 발동 → 잘못된 값이 추출되면 무방비

**2차 수정**: 추출 순서 역전 (파일명 우선, PDF 텍스트는 fallback)

```python
# 파일명 우선 (Source of Truth)
base = os.path.splitext(original_name)[0]
fn_match = re.match(r'^(.+?)_AR_(\d{4}-\d{2}-\d{2})$', base)
if fn_match:
    customer_name = fn_match.group(1).strip()
else:
    # Fallback: PDF 텍스트
    customer_pattern = r'([가-힣]{2,10})\s*고객님을?\s*위한'
    ...
```

**데이터 복구**: `db.customers.updateOne({...}, {$set: {"personal_info.name": "JUNGCLAIREBOKYUNG"}})`

## 근본 원인

### 핵심 설계 결함

```
filename_parser.py = Source of Truth (한글/영문 모두 지원)
    ↕ 이 파서가 사용되지 않고 있었음!
doc_prep_main.py = PDF 텍스트에서 한글 regex로 추출 (영문 실패)
```

`doc_prep_main.py`가 고객명을 PDF 텍스트의 한글 regex로 추출하고, 파일명은 마지막 fallback으로만 사용.
이 순서가 잘못됨: PDF 텍스트에서 "자료는"처럼 오매칭되면 fallback 발동 불가.

### 올바른 원칙: "파일명 = Source of Truth"

파일명은 업로드 전에 이미 정제된 데이터. PDF 텍스트 추출은 OCR 오류, 인코딩 문제 등으로 신뢰성이 낮음.

```
수정 전: PDF 텍스트 → (None일 때만) → 파일명 fallback  ← 오매칭 무방비
수정 후: 파일명 → (실패 시) → PDF 텍스트 fallback      ← 안전
```
