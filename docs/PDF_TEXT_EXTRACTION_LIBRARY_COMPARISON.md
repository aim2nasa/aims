## 배경

- R6 프롬프트 기준, pdfplumber 로컬 파싱 시 91.5% → DB 텍스트(PyMuPDF) 기반 76.5%로 하락

- **원인**: PyMuPDF가 테이블/양식 PDF에서 텍스트를 파편화하여 추출 → LLM 분류 정확도 저하

## 1. 속도 비교

| 라이브러리 | 평균 | 최대 | 총 시간 | PyMuPDF 대비 |
|-----------|------|------|---------|-------------|
| **PyMuPDF** | 16ms | 530ms | 5.4초 | 1.0x |
| **pypdfium2** | 21ms | 469ms | 7.3초 | **1.4x** |
| pypdf | 254ms | 12,031ms | 87.7초 | 16.4x |
| pymupdf4llm | 1,546ms | 37,092ms | 533초 | **96.4x** |

### 분석
- **pypdfium2**: PyMuPDF와 거의 동급 속도. 실용적 차이 없음
- **pypdf**: 16배 느리고, 일부 파일에서 12초 소요. 한글 폰트 경고(Illegal character) 다수 발생
- **pymupdf4llm**: 96배 느림. 마크다운 변환 오버헤드가 치명적

### 사례 1: 보험증권 테이블 (이방-유병자실손증권.pdf)

## 6. 결론 및 권장사항

### 최종 선택: **pypdfium2**

**선정 이유:**
1. **속도**: PyMuPDF 대비 1.4배로 실용적 차이 없음 (21ms vs 16ms)
2. **텍스트 품질**: 테이블/양식에서 행 단위 텍스트 정리가 월등히 우수
3. **띄어쓰기**: 한글 문서에서 자연스러운 단어 구분 (PyMuPDF는 붙여쓰기 경향)
4. **커버리지**: PyMuPDF와 100% 동일 (243건 텍스트, 102건 이미지)
5. **안정성**: 에러 0건, 타임아웃 0건, 경고 0건
6. **엔진**: Google PDFium (Chrome PDF 엔진) 기반으로 검증된 안정성

### 탈락 사유

| 라이브러리 | 탈락 사유 |
|-----------|----------|
| pymupdf4llm | 96배 느림, 커버리지 16건 손실, 마크다운 오버헤드 과도 |
| pypdf | 16배 느림, 한글 폰트 경고 다수, 띄어쓰기 불규칙 |

### 적용 계획

- **변경 파일**: `backend/api/document_pipeline/services/meta_service.py` (`_extract_pdf_info()`)
- **범위**: 텍스트 추출 부분만 pypdfium2로 교체. 페이지 수/메타데이터는 PyMuPDF 유지
- **AR/CRS 파싱**: 별도 서비스이므로 영향 없음
- **기존 DB 문서**: 향후 업로드분부터 적용
