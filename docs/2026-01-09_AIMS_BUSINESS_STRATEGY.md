## 1. AIMS 핵심 가치

### 핵심 컨셉

```
문서 (PDF, 이미지, 스캔, 텍스트) → 검색 가능한 지식
```

**이게 끝이고, 이게 전부.**

### 왜 가치 있나

1. **수요는 있는데** — 누구나 "그 문서 어디 있지?" 하고 찾음
2. **제대로 된 솔루션이 없음** — Paperless-ngx 정도, 나머지는 반쪽짜리
3. **AIMS는 이미 동작함** — 코드가 있고, 검증됨

## 2. 문서 처리 아키텍처

### 문서 분류

| 구분 | 파일 유형 | 처리 방식 |
|------|----------|----------|
| **OCR 불필요** | HWP, DOCX, XLSX, PPTX, TXT, HTML, 텍스트 PDF | 파서로 텍스트 추출 |
| **OCR 필수** | 스캔 PDF, JPG, PNG, TIFF 등 이미지 | OCR로 텍스트 추출 |

### 처리 흐름

```
[문서 입력]
     │
     ├─ 텍스트 기반 ─→ AIMS 파서 (직접 처리) ─→ PDF 프리뷰 제공
     │   (HWP, DOCX 등)
     │
     └─ 이미지 기반 ─→ OCR 어댑터 ─→ 고객이 선택
                            │
                            ├─ 업스테이지
                            ├─ 로민
                            ├─ 네이버 CLOVA
                            ├─ 사이냅소프트
                            ├─ Tesseract (오픈소스)
                            └─ 기타...

     ↓
[AIMS 핵심]
구조화 → 벡터화 → 검색 → AI 질의응답
```

### OCR 전략: 벤더 종속 방지

| 항목 | 전략 |
|------|------|
| **텍스트 파싱** | 직접 잘 처리 |
| **OCR** | 고객이 선택하게 함 |
| **효과** | OCR 공급사 갑질 방지 |

**"OCR은 골라 쓰세요, 그 이후는 제가 합니다"**


## 4. 데이터 주권 모델

### 기존 vs AIMS

```
[기존 플레이어]
고객 문서 ──→ 외부 클라우드 ──→ 결과 반환
         (문서가 나감)

[AIMS]
┌─────────────────────────────────┐
│        고객사 서버 내부          │
│                                 │
│  문서 → 파싱/OCR → DB → 검색    │
│         (문서가 안 나감)         │
└─────────────────────────────────┘
```

### 운영 모델

```
┌─────────────────────────────────┐
│        고객사 서버               │
│                                 │
│   [DB] + [파일]                 │
│      ↑                          │
│      │ 연결                     │
└──────┼──────────────────────────┘
       │
       ▼
┌─────────────────────────────────┐
│     AIMS 서비스 (운영 제공)      │
│                                 │
│   파싱, 검색, AI, 업데이트 등    │
└─────────────────────────────────┘
```

| 구분 | 소유/관리 |
|------|----------|
| **데이터 (DB, 파일)** | 고객사 |
| **서비스 운영** | AIMS |
| **업데이트/유지보수** | AIMS |

**"데이터는 당신 것, 서비스는 내가 합니다"**

## 5. 제품 로드맵

### 두 개의 제품

```
[현재]
AIMS (보험 도메인 = 모든 것 포함)

        ↓ 핵심 추출

[미래]
┌─────────────────────────────────────┐
│            AIMS Core (범용 엔진)     │
│                                     │
│   - 텍스트 파싱                      │
│   - OCR 어댑터 (고객 선택)           │
│   - 구조화 → 벡터화 → 검색 → AI     │
│   - 고객 서버 DB + 서비스 운영       │
│                                     │
└─────────────────────────────────────┘
        │
        ├─────────────┬─────────────┐
        ▼             ▼             ▼
     [AIMS]      [고객 A]     [고객 B]
      보험        법률         의료
```

### 두 개의 게임

| 구분 | AIMS | AIMS Core |
|------|------|-----------|
| 성격 | 생계형 프로젝트 | 패러다임 시프트 |
| 시장 | 보험 설계사 (니치) | 모든 문서 사용자 (범용) |
| 스케일 | 작다 | **크다** |
| VC 관심 | ✗ | **✓** |

## 6. 투자 심사관 Q&A

### Q1. 첫 10명의 돈 내는 고객을 어디서 찾을 건가?

> 와이프가 메트라이프 보험 설계사다. 설계사로서 고객들 문서 관리에 어려움을 겪는 것을 보고 만든 게 AIMS다. 와이프의 평가가 나쁘면 실패, 그렇지 않다면 같은 지점의 설계사들에게 확산. 그 지점에서만 최소 10명은 가능.

### Q2. 10명 → 100명 → 1000명 가는 길은?

> 지인 마케팅이 우선. 일차 목표는 메트라이프 현 지점이고, 전국은 그 다음. 현재 메트라이프 지점은 **전국 1위 지점**이라, 여기서 선택받으면 파급력이 클 것으로 본다.

### Q3. 수익 모델은?

> 개인별 월 구독 모델. 부담이 안 되는 가격 (1~5만원 수준). AI/OCR 사용량에 따라 조정. 당분간은 적자 감수.

### Q4. 어떻게 버틸 건가?

> 적어도 1년은 버틸 돈 있다.

### Q5. 1년 후 목표는?

> 최소 100명 / 월 300만원. 원가도 안 되지만, 이 정도가 최소 기준선.

### Q6. 경쟁자가 따라 만들면?

> 끊임없는 진화만이 답. 가격은 안 내린다. 1인 개발자로 더 이상 내려갈 가격이 없다. 이 조그만 시장에 들어오는 큰 회사 담당자는 아마 짤릴 거다.

### Q7. 당신이 쓰러지면?

> 고객 서버에 데이터가 있다 — 이게 안전장치. 서비스 죽어도 고객 데이터는 고객 것. 오히려 영업 포인트: "저 망해도 당신 데이터는 안전합니다"

### Q8. 왜 당신이 이걸 해야 하나?

> 소프트웨어 개발 **30년 이상 종사**해온 탑클래스 전문가다. AI와 치열하게 토론하며 객관화해서 찾은, 1인 개발자가 할 수 있는 영역이다. 단순 IT기술만이 아닌 시장, 비즈니스에 대한 경험이 없으면 그릴 수 없는 그림이 있는데, 난 그걸 할 수 있는 사람이다.

## 7. 전략 요약

### 왜 당신인가

| 요소 | 내용 |
|------|------|
| **30년 경력** | 소프트웨어 개발 탑클래스 전문가 |
| **통합적 시야** | 기술 + 비즈니스 + 시장을 동시에 보는 눈 |
| **AI 활용 능력** | AI와 토론하며 객관화, 방향 도출 |
| **도메인 접근성** | 와이프 = 현직 설계사, 실제 페인 포인트 |
| **비용 구조** | 1인 개발자, 오버헤드 없음 |
| **절박함** | 먹고 살아야 함 |

### 핵심 전략

| 항목 | 내용 |
|------|------|
| **텍스트 파싱** | 직접 잘 함 |
| **OCR** | 고객이 선택 (종속 방지) |
| **데이터** | 고객 서버 (유출 방지) |
| **운영** | AIMS가 제공 (구독 수익) |
| **시장** | 빅테크가 안 보는 틈새 |

### 1인 개발자의 해자(Moat)

> "이 조그만 시장에 들어오는 큰 회사 담당자는 짤린다"

- 작은 시장 + 낮은 비용 구조 = 방어막
- 30년 경험 기반 설계 = 쉽게 못 따라옴

## 8. 심사 총평

| 항목 | 평가 |
|------|------|
| 시장 | 작지만 존재, 빅테크가 안 보는 틈새 |
| 창업자 | 30년 전문가, 기술+비즈니스 통합 역량 |
| 진입장벽 | 작은 시장 + 경험 기반 설계 = 쉽게 못 따라옴 |
| 첫 고객 | 확보 경로 명확 (전국 1위 지점) |
| 수익 모델 | 현실적 |
| 리스크 | 1인 리스크 있음, 완화책 존재 |
| 스케일 | AIMS = 생존, AIMS Core = 스타트업 |

### 결론

**AIMS** = 생존 + 검증
**AIMS Core** = 스케일업 가능한 스타트업 아이템

VC 투자 대상으로서:
- AIMS 단독: 스케일 작음
- AIMS Core: **패러다임 시프트, 투자 가치 있음**

1인 사업으로서:
- **성공 가능성 높음**
- 30년 경력자가 틈새 찾아서 절박하게 뛰어드는 것
