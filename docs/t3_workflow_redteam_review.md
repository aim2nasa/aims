# DocUpload (T3) 워크플로우 레드팀 검토

---

## ✅ 강점

1. **에러 플로우 완전 커버**
   - Case-001~007까지 사용자에게 명확한 JSON 응답 제공
   - `exitCode` 기반 감지 + stderr 반환으로 디버깅 용이
   - Temp Save, mkdir, mv, DB 삽입까지 전 구간 방어

2. **경로 생성 무결성 보장**
   - Generate Final Path는 재시도 + fallback으로 절대 실패 없음
   - Case-003 제거 가능하게 설계

3. **불필요한 케이스 제거**
   - Case-006/008/009는 실패 가능성 0% 또는 시스템 레벨 문제로 제거 → 복잡성 최소화

4. **사용자 응답 일관성**
   - 실패 시 항상 `500/400 + code + message` JSON 반환
   - 클라이언트 UX/디버깅 친화적

---

## ⚠️ 레드팀 관점 위험요소

1. **n8n 내부 의존성**
   - `exitCode`, `stderr` 등 n8n 엔진 동작에 강하게 의존
   - n8n 버전 업/환경 변경 시 동작 보장 불명확

2. **권한/디스크 문제 대응 한계**
   - mkdir/mv/Temp Save 실패 시 즉시 500 반환 → 자동 복구 없음
   - Temp 파일 일부가 orphan으로 남을 수 있음

3. **DB 장애 시 복구 어려움**
   - DB 삽입 실패 시 파일은 남지만 메타데이터 없음 → orphan 발생
   - 이후 자동 재시도/큐잉 로직 부재 → 수동 복구 필요

4. **로그/모니터링 부족**
   - 사용자에게는 메시지 제공되지만 관리자에게 별도 알림 없음
   - 장애 발생 로그가 n8n 내부에만 남아 운영 모니터링 부재

5. **성능/부하 테스트 미비**
   - 동시 업로드 대량 요청 시 race condition, 디스크 IO 병목 가능성
   - n8n Executor 단일 처리 한계 고려 필요

## 🟥 개선 제안

1. **orphan 파일 관리**
   - Temp 파일 → 실패 시 즉시 삭제(현재 일부 남을 가능성 있음)
   - DB 실패 시 파일을 격리(`orphan` 폴더)해 추적 가능하게 관리

2. **자동 재시도/큐잉**
   - DB 삽입 실패 시 일정 횟수 재시도 or 별도 큐에 넣어 후속 처리
   - mkdir/mv 실패도 1~2회 재시도 후 최종 실패 처리

3. **관리자 알림 연동**
   - Case-002/004/005/007 발생 시 Slack/Email/Webhook 알림 추가
   - 운영자 즉각 대응 가능하게

4. **모니터링 강화**
   - 디스크 용량, DB 연결 상태 사전 체크
   - Prometheus/Grafana 연계 모니터링 고려

5. **부하 및 동시성 테스트**
   - n8n Executor가 race condition 없이 안정적으로 처리 가능한지 부하 테스트 필요

## ✅ 결론

- 현재 워크플로우는 **단일 요청 기준 안정성 90% 이상**
- **운영 환경(부하, 시스템 자원 문제, orphan 관리)**까지 고려하면 **운영 보강 필요**
