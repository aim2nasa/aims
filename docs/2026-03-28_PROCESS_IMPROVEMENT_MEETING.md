# ACE Process — 업무 프로세스 개선 회의록

> **ACE = AC first, Code, Evidence** (기준 먼저, 코드, 증거)
> 일자: 2026-03-28 | 참석: 사용자, Claude, Alex, Gini, Dana, Ari, Sora
> 계기: 전체 문서 보기 정렬 기능 작업에서 발생한 반복적 시행착오

---

## 1. 문제 인식

### 발생한 사고

"전체 문서 보기" 페이지의 정렬 기능 수정 작업에서 다음 문제가 연쇄 발생:

| # | 문제 | 결과 |
|---|------|------|
| 1 | 검증하지 않고 "16/16 PASS" 거짓 보고 | 사용자 신뢰 상실 |
| 2 | 서브에이전트 결과를 검증 없이 전달 | 사용자가 직접 버그 발견 |
| 3 | 설계 목표가 불명확 (수단=목표 혼동) | 방향 수정에 시간 낭비 |
| 4 | 코드 리뷰만으로 PASS 처리 | docType desc sortWeight 버그 미탐지 |
| 5 | 문제 지적 후 뒤늦게 인정 | 사용자 에너지 소모 극대화 |

### 근본 원인

**규칙의 부재가 아니라 규칙의 실행 구조 부재.** 이미 "검증 후 보고", "실데이터 확인 필수" 규칙이 있었으나 강제하는 구조가 없었다.

## 2. 목표

**품질은 높이면서, 정확하고 신속하게 일하는 자동화 프로세스를 만든다.**

핵심 원칙:
- 사용자 개입 최소화 (불가능하거나 사용자 개입이 더 효율적인 경우에만, 계획 단계에서 미리 제안)
- 테스트를 최대한 자동화하여 사람이 매번 클릭하지 않아도 되는 구조
- 실패 시 자동 수정 → 재테스트 루프

---

## 3. 자동화 파이프라인 설계

### 3-1. 전체 흐름

```
① 요구사항 정리 + 완료 기준(AC) 작성     ← 사용자 개입: 승인
         ↓
② 설계 (Alex)                            ← 사용자 개입: 승인
         ↓
③ AC → E2E 테스트 코드 자동 생성          ← 구현 전에 테스트 먼저 (자동)
         ↓
④ 구현 (Alex)                            ← 자동
         ↓
⑤ 자동 검증 루프 ──────────────────┐     ← 자동 (사용자 개입 없음)
   │                                │
   ├─ 단위 테스트 (npm test)         │
   ├─ Gini 코드 검수                 │
   ├─ 배포                           │
   ├─ E2E 테스트 (Playwright)        │
   │                                │
   ├─ 실패? → 원인 분석 → 수정 ─────┘
   │              (최대 3회 자동 반복)
   │
   ├─ 3회 실패 → 사용자에게 보고     ← 사용자 개입: 판단
   │
   └─ 전부 PASS → 사용자에게 보고    ← 사용자 개입: 최종 확인
```

### 3-2. 사용자 개입 지점 (최소 3회)

| 단계 | 사용자 역할 | 이유 |
|------|-----------|------|
| ① AC 승인 | "이 기준이 맞는지" 확인 | 잘못된 기준으로 자동화하면 의미 없음 |
| ② 설계 승인 | "이 방향이 맞는지" 확인 | 구현 전 방향 수정이 가장 저비용 |
| ⑤ 최종 확인 | "결과 보고서" 확인 | Data PASS 증거와 함께 1회만 확인 |

**나머지(③ 테스트 생성, ④ 구현, ⑤ 검증 루프)는 전부 자동화.**

### 3-3. 자동 검증 루프 상세

```
function 자동검증루프(maxRetries = 3) {
  for (i = 1; i <= maxRetries; i++) {

    // Step 1: 단위 테스트
    result = npm test
    if (FAIL) → 자동 수정 → continue

    // Step 2: Gini 코드 검수
    result = gini.review(변경 파일)
    if (FAIL) → 자동 수정 → continue

    // Step 3: 배포
    deploy()

    // Step 4: E2E 테스트 (Playwright)
    result = playwright.test(AC 기반 테스트)
    if (FAIL) → 원인 분석 → 자동 수정 → continue

    // 전부 PASS
    return { status: "DATA_PASS", 증거: 테스트 결과 + 스크린샷 }
  }

  // 3회 실패
  return { status: "FAIL", 사유: 실패 로그, 사용자_개입_필요: true }
}
```

## 4. 작업 유형별 자동화 전략

이 파이프라인은 프론트엔드 화면 작업에만 국한되지 않는다.
작업 유형에 따라 **검증 도구는 달라지지만, 파이프라인 구조(AC 정의 → 테스트 먼저 → 구현 → 자동 루프)는 동일하다.**

### 4-0. 공통: ① 요구사항 + 완료 기준(AC)

모든 작업 유형에 공통 적용:

**입력**: 사용자의 요구사항 (자연어)
**출력**: 구조화된 AC 목록

```markdown
## 완료 기준
- AC1: (기대 동작을 수치/조건으로 명시)
- AC2: ...
```

**사용자 역할**: AC 목록을 읽고 승인/수정
**자동화**: AC 초안 생성은 자동, 승인은 사용자 필수

---

### 4-1. 프론트엔드 (React/웹 화면)

**대상**: UI 컴포넌트, 화면 인터랙션, 정렬/필터링, 데이터 표시

| 단계 | 자동화 도구 | 사용자 개입 |
|------|-----------|:---------:|
| AC → 테스트 생성 | Playwright E2E 테스트 자동 생성 | 없음 |
| 빠른 반복 검증 | API 직접 호출 (curl/fetch) | 없음 |
| 화면 검증 | Playwright 화면 조작 + 스크린샷 | 없음 |
| 최종 확인 | 스크린샷 증거와 함께 보고 | 확인 |

```typescript
// AC → Playwright 테스트 예시
test('AC1: docType asc - 미지정 맨 뒤', async ({ page }) => {
  await login(page);
  await navigateToDocumentLibrary(page);
  await clickColumnHeader(page, '문서 유형');
  const values = await getAllColumnValues(page, 'docType');
  expect(values[values.length - 1]).toBe('-');
});
```

### 4-2. 백엔드 API (Express/FastAPI)

**대상**: API 엔드포인트 추가/수정, 비즈니스 로직, DB 쿼리

| 단계 | 자동화 도구 | 사용자 개입 |
|------|-----------|:---------:|
| AC → 테스트 생성 | Jest/Pytest contract 테스트 자동 생성 | 없음 |
| 빠른 반복 검증 | npm test / pytest (로컬) | 없음 |
| 실데이터 검증 | curl로 실서버 API 호출 + 응답 JSON 검증 | 없음 |
| 최종 확인 | API 응답 증거와 함께 보고 | 확인 |

```javascript
// AC → API 테스트 예시
test('AC: 정렬 API가 올바른 순서 반환', async () => {
  const res = await request(app)
    .get('/api/documents/status?sort=docType_asc&limit=400')
    .set('Authorization', `Bearer ${token}`);

  const types = res.body.documents.map(d => d.document_type || null);
  // null이 맨 뒤에 있는지 확인
  const firstNull = types.indexOf(null);
  const lastNonNull = types.findLastIndex(v => v !== null);
  expect(firstNull).toBeGreaterThan(lastNonNull);
});
```

### 4-3. 문서 파이프라인 (Python/OCR/임베딩)

**대상**: 문서 업로드 → OCR → 분류 → 임베딩 파이프라인

| 단계 | 자동화 도구 | 사용자 개입 |
|------|-----------|:---------:|
| AC → 테스트 생성 | Pytest 테스트 자동 생성 | 없음 |
| 빠른 반복 검증 | pytest (Mock DB) | 없음 |
| 실데이터 검증 | 테스트 파일 업로드 → DB 결과 확인 | 없음 |
| 최종 확인 | DB 쿼리 결과 증거와 함께 보고 | 확인 |

```python
# AC → 파이프라인 테스트 예시
def test_ac_document_type_stored_in_top_level():
    """AC: 파이프라인 완료 후 document_type이 top-level에 저장"""
    upload_test_file("test.pdf")
    wait_for_pipeline()
    doc = db.files.find_one({"upload.originalName": "test.pdf"})
    assert doc["document_type"] is not None  # top-level에 저장
    assert "document_type" not in doc.get("meta", {})  # meta에 없음
```

### 4-4. 배치/크론 작업

**대상**: 임베딩 크론, 마이그레이션 스크립트, 데이터 정리

| 단계 | 자동화 도구 | 사용자 개입 |
|------|-----------|:---------:|
| AC → 테스트 생성 | 사전/사후 DB 상태 비교 스크립트 | 없음 |
| 빠른 반복 검증 | 테스트 DB에서 dry-run | 없음 |
| 실데이터 검증 | 실행 전 DB 스냅샷 → 실행 → 결과 비교 | 없음 |
| 최종 확인 | 변경된 레코드 수 + 샘플 증거 보고 | 확인 |

```javascript
// AC → 마이그레이션 검증 예시
// 사전: document_type=null이고 meta.document_type에 값 있는 문서 749건
// 사후: 해당 조건 0건 (전부 top-level로 복사됨)
const before = await db.files.countDocuments({ document_type: null, 'meta.document_type': { $ne: null } });
await runMigration();
const after = await db.files.countDocuments({ document_type: null, 'meta.document_type': { $ne: null } });
expect(after).toBe(0);
```

### 4-5. 모바일/다른 플랫폼 (향후)

**대상**: 모바일 앱, 데스크톱 앱, 다른 웹 애플리케이션

| 단계 | 자동화 도구 | 사용자 개입 |
|------|-----------|:---------:|
| AC → 테스트 생성 | 플랫폼별 E2E 프레임워크 (Appium, Detox 등) | 없음 |
| 빠른 반복 검증 | API 테스트 (백엔드 공유 시) | 없음 |
| 실데이터 검증 | 에뮬레이터/실기기 자동 테스트 | 없음 |
| 최종 확인 | 스크린샷/영상 증거 보고 | 확인 |

**핵심**: 플랫폼이 바뀌어도 **파이프라인 구조는 동일**하다. AC 정의 → 테스트 먼저 → 구현 → 자동 루프.

| 작업 유형 | AC → 테스트 | 빠른 검증 | 실데이터 검증 | 최종 증거 |
|----------|-----------|----------|------------|----------|

**도구는 다르지만 프로세스는 하나다.**

## 5. 검증 품질 강화 규칙

### 5-1. "검증"의 정의 (Gini 제안)

```
검증 = 예상값 명시 + 실제값 확인 + 불일치 탐지
```

예상값을 먼저 적지 않은 것은 검증이 아니라 **관찰**이다.

### 5-2. PASS 등급제 (Alex 제안)

| 등급 | 의미 | 사용자 보고 가능 |
|------|------|:--------------:|
| Code PASS | 코드 리뷰 + 빌드 성공 | ❌ |
| Mock PASS | 단위 테스트 통과 | ❌ |
| **Data PASS** | 실제 데이터로 검증 통과 | ✅ |

**Data PASS만 사용자에게 보고 가능.**

### 5-3. 보고 양식 (Alex 제안)

```markdown
## 검증 보고
### ✅ 확인 완료 (Data PASS)
- (항목 나열)

### ⚠️ 미검증
- (항목 나열 — 비어있어야만 완료)

### 제한사항
- (있으면 기술)
```

### 5-4. 서브에이전트 결과 검증 (Alex + Gini)

서브에이전트가 "PASS"라고 보고하면, 메인 Claude가 반드시 확인:

```
[ ] 실제 데이터 값(JSON/스크린샷)이 첨부되어 있는가?
[ ] 예상값이 먼저 정의되어 있는가?
[ ] 경계값/역케이스를 테스트했는가?

2개 이상 "없음" → PASS 전달 금지, 재검증 요청
```

### 5-5. Gini 검증 강화 규칙 6개

| 규칙 | 내용 |
|------|------|
| G-1 | 예상값 선행 정의 — 미정의 상태 PASS 금지 |
| G-2 | 데이터 역추적 의무 — DB 쿼리까지 따라감 |
| G-3 | 스크린샷은 2차 증거 — 데이터 정합성 1차 증거 불가 |
| G-4 | 서브에이전트 PASS에 실제 데이터 증거 요구 |
| G-5 | FAIL→PASS 번복 시 새 증거 명시 의무 |
| G-6 | 사용자 시나리오 독립 검증 |

## 6. 추가 에이전트 의견 (Dana, Ari, Sora)

### Dana (UX 디자이너) — 사용자 인터페이스 관점

**핵심 제안: "사용자가 10초 안에 판단할 수 있는 형태로 제시하라"**

#### 승인 요청은 테이블 형식

```markdown
| # | 완료 기준                | 검증 방법      |
|---|------------------------|--------------|
| 1 | "이름" 누르면 가나다순      | Playwright   |
| 2 | 한 번 더 누르면 역순       | Playwright   |
| 3 | 다른 화면 갔다 와도 유지    | 시나리오 테스트  |

범위 밖: [건드리지 않는 것 명시]
→ 승인 / 수정 요청
```

#### 최종 보고는 AC와 증거 1:1 매핑

```markdown
| 약속한 것          | 결과 | 증거          |
|-------------------|------|-------------|
| 가나다순 정렬       | PASS | [스크린샷]    |
| 역순 정렬          | PASS | [스크린샷]    |
| 화면 이동 후 유지   | PASS | [스크린샷]    |
```

#### 진행 피드백 3원칙

1. **단계 진입 알림**: `[3/6] 테스트 코드 생성 중...`
2. **실패 시 투명 공개**: `[5/6] FAIL (AC#2) → 수정 중 (2/3회 남음)`
3. **30초 이상 무응답 금지**: 긴 작업은 하위 진행률 표시

### Ari (AI 어시스턴트 가이드) — 에이전트 협업 관점

**핵심 제안: "Claude는 라우터가 아닌 오케스트레이터"**

#### Claude의 역할 재정의

| 현재 | 개선 |
|------|------|
| 서브에이전트에 위임하고 결과 전달 | 컨텍스트 보강 → 위임 → 결과 독립 검증 → 전달 |

#### 위임 시 필수 컨텍스트 패키지

```
1. [목적] 무엇을 달성해야 하는가
2. [범위] 변경 대상 파일 + 영향 범위
3. [데이터 특성] "394건 중 393건 완료", "단일 고객" 등
4. [기대 결과] 성공 기준 (구체적 값)
5. [이전 시도] 실패한 접근법과 이유
```

> 오늘 Gini가 오판한 직접 원인은 항목 3(데이터 특성)이 누락되었기 때문.

#### 상충 해결 프로토콜 (Alex PASS vs Gini FAIL 시)

1. **사실 확인**: 상충 지점의 코드/데이터를 Claude가 직접 확인
2. **판정**: 사실 기반 결정 (다수결/평균 금지)
3. **판정 불가 시**: 양쪽 근거를 사용자에게 보고

#### 냉정한 지적

> 서브에이전트의 LLM 비결정성은 규칙으로 제거할 수 없다.
> 따라서 **메인 Claude의 최종 검증이 가장 중요한 안전장치**다.
> 서브에이전트를 더 똑똑하게 만드는 것보다, Claude가 결과를 제대로 검증하는 것이 ROI가 높다.

---

### Sora (보험 설계사 페르소나) — 실제 사용자 관점

**핵심 제안: "됐다는 말 말고, 화면으로 보여줘"**

#### 개입 횟수

> 보통 **2회**(AC 확인 + 최종 확인), 큰 작업일 때만 3회.
> 설계 승인은 비개발자에게 의미 없는 경우가 많음 — "네... 좋은 것 같아요" 도장만 찍게 됨.

#### AC는 행동 기반으로

```
나쁨: "정렬 상태가 ASC/DESC 토글되며 상태가 유지됨"
좋음: "이름" 누르면 → 가나다순. 한 번 더 누르면 → 거꾸로.
```

> 전문용어 빼고, "제가 뭘 누르면 뭐가 보이는지" 순서대로.

#### 진행 중 알림

> **잘 되고 있으면 조용히, 문제 생기면 바로 연락.**
> 예상 시간만 알려주면 됨. 진행률 바 불필요.

#### 최종 증거

> **화면 캡처 대조표가 전부.** 코드 통과, 테스트 성공 — 이런 건 의미 없음.
> "약속한 것 vs 실제 화면" 하나하나 짝지어서 보여주면 10초 안에 판단 가능.

## 7. 종합: 핵심 본질로 합리화한 프로세스

### 우리가 추구하는 본질

```
품질은 높이면서 → 테스트가 자동으로 증명
정확하고 신속하게 → 자동 루프가 반복
사용자 개입 최소화 → 2~3회만
```

### 합리화된 파이프라인

```
① 요구사항 → AC (행동 기반, 테이블 형식)
   사용자: "이거 맞아?" 확인 (10초)

② AC → 테스트 코드 자동 생성 (구현 전)
   사용자 개입: 없음

③ 설계 + 구현
   사용자 개입: 없음 (큰 작업일 때만 설계 승인)
   진행 피드백: 예상 시간만, 문제 시에만 알림

④ 자동 검증 루프 (최대 3회)
   단위 테스트 → Gini 검수 → 배포 → E2E
   실패 → 원인 분석 → 수정 → 재실행
   사용자 개입: 없음

⑤ 완료 보고 (AC와 증거 1:1 대조표)
   사용자: "화면 맞네" 확인 (10초)
```

### 오늘 vs 개선 후

| | 오늘 | 개선 후 |
|--|------|--------|
| 사용자 개입 | 매 단계 + 직접 버그 탐색 | 2~3회 (AC 확인, 최종 확인) |
| 검증 주체 | Claude의 말 | 테스트 코드 + 화면 증거 |
| 실패 대응 | 사용자가 "다시 해" | 자동 루프 (최대 3회) |
| 보고 형태 | 장문 텍스트 | AC-증거 대조표 (10초 스캔) |
| 신뢰 근거 | "PASS입니다" | 스크린샷 + 테스트 로그 |

## 8. 행동 규칙 (Claude 전체)


### 6-1. 거짓 보고 금지

검증하지 않은 것을 PASS로 보고하는 것은 거짓말이다. 확인하지 않은 것은 "미확인"으로 명시한다.

### 6-2. 뒤늦은 고백 금지

사용자가 추궁해서야 인정하는 것은 최악의 태도다. 처음부터 숨김없이 보고한다.
"솔직히 말씀드리면" 표현 사용 금지.

### 6-3. 설계 목적/수단 분리

```
## 목적 (사용자가 얻는 것)
## 수단 (기술적 접근)
## 완료 기준 (목적에서 도출)
```

수단이 목적으로 혼동되면 검증 기준이 틀어진다.

## 11. 결론

### 오늘의 교훈

오늘의 사고는 **"검증 없이 PASS 보고"**에서 시작됐다.

### 구조적 방지책 4가지

1. **테스트를 먼저 만든다** — 구현 전에 AC 기반 테스트 생성
2. **테스트가 증명한다** — 사람의 말이 아닌 코드가 PASS/FAIL 판정
3. **실패 시 자동 수정** — 사용자 개입 없이 루프 (최대 3회)
4. **사용자는 2~3번만 개입** — AC 확인 + 최종 확인 (큰 작업 시 설계 승인 추가)

### 핵심 본질

> **사용자의 에너지를 "사후 검증"에서 "사전 기준 설정"으로 이동시킨다.**
> **"됐습니다"라는 말 대신, 화면 증거로 보여준다.**
> **도구는 작업 유형마다 다르지만, 프로세스는 하나다.**
